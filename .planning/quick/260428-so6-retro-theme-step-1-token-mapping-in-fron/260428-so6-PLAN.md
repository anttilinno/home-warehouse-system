Notes on intentional choices (per `<constraints>` from /gsd-quick discuss):

1. **Source vars scoped, not global.** Defining `--bg-base`, `--amber`, etc. at
   `:root` would leak into the light + `.dark` themes (their tokens reference
   them via `var(...)` and would override). Keeping them inside
   `[data-theme="retro-terminal"]` ensures they only resolve when the retro
   theme is active.
2. **Hex source + var(...) shadcn map.** Existing `:root` and `.dark` use
   `oklch(...)`; the premium-terminal palette is locked as hex in
   `sources/themes/default.css`. Mixing is acceptable here because the new
   block is fully self-contained — no token in this block is consumed by light
   or dark mode.
3. **`--primary-foreground: #1a0e00`** — near-black on amber. AAA contrast on
   `--amber: #ffd07a`. Premium-terminal sources don't expose this hue
   explicitly so we declare the literal directly.
4. **`--radius: 0` is enough for Step 1.** shadcn's `@theme inline` derives
   `--radius-sm/md/lg/xl/2xl` from `--radius` via `calc(var(--radius) ± Npx)`,
   so all derived radii compute to small negatives or zero — visually sharp.
   Step 3 of the rollout will explicitly set each derived radius to `0` for
   safety; Step 1 leaves them computed.
5. **`--font-sans: var(--font-mono)`.** The existing `@theme inline` defines
   `--font-mono: var(--font-quicksand)`, so this swap is enough to flip body
   text to mono without touching the font import (Quicksand is the project's
   shipped mono pairing per existing config).
6. **No next-themes / tsx changes in Step 1.** The block activates only when
   something sets `data-theme="retro-terminal"` on `<html>`. next-themes does
   not yet emit this string — that's Step 2 of the rollout.
7. **No new files, no component edits.** Pure CSS append. Rollback is `git
   restore frontend/app/globals.css`.
