## Open Data-Shape Decisions (planner must resolve)

The loan API in `lib/api/loans.ts` shipped in Phase 56 but has gaps the UI contract must acknowledge. The contract below is written so each decision resolves cleanly without visual rework.

| # | Gap | Options | Planner guidance |
|---|-----|---------|------------------|
| O-01 | LOAN-04 — "edit a non-returned loan's due date and notes". `loansApi.extend()` accepts only `new_due_date`; there is no endpoint for editing notes. | (A) Add `loansApi.update(id, { due_date, notes })` backed by `PATCH /loans/{id}`. (B) Split: use `extend` for the date field and add `loansApi.updateNotes(id, { notes })` for notes. (C) Keep only `extend` in scope and defer notes editing. | **Option A strongly preferred.** The edit panel is designed around a single PATCH that takes both fields. Option C breaks LOAN-04 acceptance (notes are part of the requirement). |
| O-02 | Loan list decoration — active/overdue/history rows MUST show item name + borrower name inline without N extra round-trips. | (A) Backend embeds `item: { id, name, primary_photo_thumbnail_url }` and `borrower: { id, name }` in the loan list response. (B) Frontend issues parallel item/borrower GETs per row (N×2 requests). (C) Frontend batches via existing list endpoints + client-side join keyed by IDs. | **Option A strongly preferred.** Matches Phase 61's `primary_photo_thumbnail_url` decoration pattern on items list. If unavailable, Option C is acceptable — load full item + borrower lists once (cached aggressively, `staleTime: 60_000`), build an ID→name map, render synchronously. Option B is rejected. |
| O-03 | Tab counts — the Active / Overdue / History tabs MUST show counts (e.g. `ACTIVE · 3`). | (A) Backend returns counts alongside list responses (e.g. list responses include `active_count`, `overdue_count`, `history_count` or a separate `/loans/stats` endpoint). (B) Frontend fires all three list endpoints on mount and derives counts from `.length`. (C) Show counts only for the currently active tab (no cross-tab counts). | **Option B acceptable for this phase.** The three endpoints are cheap and the tabs are the landing UI — triggering all three on page load is fine. Planner may upgrade to Option A if backend gains a stats endpoint. Option C is a fallback if any list call fails. |

The UI contract mandates: (1) tab counts MUST be visible on all three tabs simultaneously when the page first loads; (2) rows MUST render item name + borrower name + (when available) item primary thumbnail on first render, not after a cascading per-row fetch; (3) LOAN-04 edit flow MUST accept both due date and notes in a single submit.

## Focal Point — `/loans` Route

1. **Primary focal point:** `+ NEW LOAN` button, amber-filled, rendered in the top-right of the list header. It is the only amber-filled element above the fold in the empty/small-list case and draws the eye first.
2. **Secondary focus:** the **tabs row** (`ACTIVE` / `OVERDUE` / `HISTORY`) with its amber top border on the active tab — the one amber accent inside the data region. The tab counters (mono 14px / 600) let the user scan volume across states in a single glance.
3. **Tertiary focus:** the loans table — rows use charcoal text on cream; overdue-date cells are the only red accents; the active-row amber rail (color rule #5) appears when a row's edit panel is open.
4. When a slide-over panel is open, focus shifts to the panel: the panel's submit button (`CREATE LOAN` or `SAVE LOAN`) becomes the only amber-filled element on screen; the backdrop overlays the page at 40% charcoal; the table's amber rail on the edited row persists to anchor context.

## Focal Point — `/items/:id` (ACTIVE LOAN + LOAN HISTORY sections)

1. **Primary focal point** (when an active loan exists): the `MARK RETURNED` row-action button inside the `ACTIVE LOAN` section — charcoal outline, not amber, but the only interactive control in that section draws the eye.
2. **Primary focal point** (when no active loan exists): the `ACTIVE LOAN` empty-state body with its neutral "no active loan" message; a `+ NEW LOAN` button is NOT rendered here (that CTA lives on `/loans` — inline creation from item detail is deferred per ROADMAP scope).
3. **Secondary focus:** the `LOAN HISTORY` block — mono timestamps set a strong vertical rhythm; one row per past loan, returned-row gray styling.

## Focal Point — `/borrowers/:id` (ACTIVE LOANS + LOAN HISTORY sections)

1. **Primary focal point** (when active loans exist): the leftmost active-loan row's `MARK RETURNED` button, charcoal outline, inside the `ACTIVE LOANS` section. Multiple rows may be present; each row owns its own MARK RETURNED control.
2. **Primary focal point** (when no active loans): the `ACTIVE LOANS` section renders a `RetroEmptyState` with body copy directing the user to `/loans` for loan creation.
3. **Secondary focus:** the `LOAN HISTORY` block, same rhythm as on the item detail page.

- **Page size:** 25 rows / page (inherits Phase 60 list convention); pagination control hidden when total ≤ 25
- **Tabs row:** uses the existing `RetroTabs` primitive. Three fixed tabs: `ACTIVE`, `OVERDUE`, `HISTORY`. Each tab label reads `{LABEL} · {count}` where `{count}` is mono. The active tab pill gets a 3px amber top border + cream fill (per Color rule #4); inactive tabs use the existing `bg-retro-gray` treatment.
- **Tab state persistence:** active tab stored in URL hash (`/loans#active`, `/loans#overdue`, `/loans#history`) — mirrors Phase 58's `useHashTab` pattern so deep-links into a specific tab work and back-button navigates between tabs.
- **Default tab on mount:** `ACTIVE` if no hash present.
- **Table columns (ordered left-to-right):**

  | Column | Header | Content | Alignment | Mobile collapse |
  |--------|--------|---------|-----------|-----------------|
  | 1 | (sr-only `Thumbnail`) | 40×40 item primary thumb or `<ImageOff>` placeholder | left | persists (56px fixed) |
  | 2 | `ITEM` | item name — `<Link to="/items/{id}">` | left | persists; wraps |
  | 3 | `BORROWER` | borrower name — `<Link to="/borrowers/{id}">` | left | persists; wraps |
  | 4 | `QTY` | `×{quantity}` in mono | center | hidden <640px (moves inline below item name in compact-row layout) |
  | 5 | `LOANED` | loaned-at date in mono `YYYY-MM-DD` | left | hidden <640px |
  | 6 | `DUE` | due date in mono `YYYY-MM-DD`, or `—` if null; overdue rows add a red `<AlertTriangle>` icon 16px and red text | left | persists |
  | 7 | `ACTIONS` | row action cluster (see below) | right | persists |

- **History tab additional/replaced columns:** the `DUE` column is replaced by `RETURNED` (mono `YYYY-MM-DD HH:MM`, charcoal). The `ACTIONS` column collapses to just `[EDIT notes only]`? **No** — per LOAN-04 only non-returned loans are editable, so History rows show NO action buttons (mirrors the Phase 59 archived-row-read-only convention).
- **Row-level action buttons by tab:**
  - `ACTIVE` and `OVERDUE` tabs: `[MARK RETURNED]` (undo icon + label) + `[EDIT]` (pencil icon + label). Desktop shows label beside icon; mobile shows icon-only with `aria-label`.
  - `HISTORY` tab: no action buttons. Returned loans are read-only.
- **Row click target:** row itself is passive. `ITEM` column and `BORROWER` column are `<Link>`s to `/items/:id` and `/borrowers/:id` respectively. Clicking outside those links and outside the action cluster does nothing.
- **Overdue visual signal:** in the `DUE` cell, when `is_overdue === true`, render the date text in `--color-retro-red` with an `AlertTriangle` icon 16px prefix, 4px gap between icon and text. No other part of the row changes color (keeps 10% rule — red is destructive/warning reserved for overdue + errors).
- **Returned-row styling (History tab):** entire row text color becomes `--color-retro-gray`; mono fields retain mono; item and borrower links remain underlined on hover but link color defaults to gray as well (returned loans are completed records, not-hidden-but-not-current — gray conveys "done" without the `line-through` used for archived).
- **Pagination:** `RetroPagination` at the bottom of each tab's table; URL query `?page=N` per tab (decoupled — moving between tabs resets `page` to 1 unless the URL already encodes a page for that tab).
- **Empty states per tab:**
  - `ACTIVE` tab with zero active loans: `RetroEmptyState` — heading `NO ACTIVE LOANS`, body `Nothing is currently out on loan.`, primary action `+ NEW LOAN` (amber). (Even if this is the landing tab on an empty account, the CTA drives them into the create flow.)
  - `OVERDUE` tab with zero overdue loans: `RetroEmptyState` — heading `NO OVERDUE LOANS`, body `Nothing is past its due date. Nice.`, no primary action.
  - `HISTORY` tab with zero returned loans: `RetroEmptyState` — heading `NO LOAN HISTORY`, body `Returned loans will appear here.`, no primary action.
  - All-empty account (zero across every tab): the ACTIVE tab empty state applies as the default landing; tabs still render (OVERDUE and HISTORY also show their own empty states if the user switches).

### Slide-over panel (create loan / edit loan)

Same shell as Phase 58/59/60. Phase 62 specifics:

**CREATE flow body** (header: `NEW LOAN`, 20px / 600 / UPPERCASE):

1. **ITEM** (required) — `RetroFormField` + `RetroCombobox` over `itemsApi.list({ archived: false })`. Async option loading (300ms debounce). Trigger shows `Pick an item…` placeholder; selected item shows `{primary_photo_thumb if any} {item.name}` in the trigger. Dropdown rows include a 24×24 primary thumb on the left + item name + SKU (mono, small). Empty state inside the dropdown: `No items match that search.`
2. **BORROWER** (required) — `RetroFormField` + `RetroCombobox` over `borrowersApi.list({ archived: false })`. Async option loading. Trigger shows `Pick a borrower…` placeholder; selected shows `{borrower.name}`. Dropdown rows show borrower name + email in mono (small) as secondary text.
3. **QUANTITY** (required) — `RetroFormField` + `RetroInput` `type="number"`, `min="1"`, `inputMode="numeric"`, mono font, default `1`. Helper: `Whole units only`.
4. **LOANED ON** (optional override — defaults to "today") — `RetroFormField` + `RetroInput` `type="date"`, mono font. Helper: `Today by default`. If left blank, the backend receives `loaned_at = now()`.
5. **DUE DATE** (optional) — `RetroFormField` + `RetroInput` `type="date"`, mono font. Helper: `Optional — leave blank for open-ended loans`.
6. **NOTES** (optional) — `RetroFormField` + `RetroTextarea`, 4 rows visible. Helper: `Optional — up to 1000 characters`.

**EDIT flow body** (header: `EDIT LOAN`, 20px / 600 / UPPERCASE):

The edit flow intentionally shows LESS than create. Per LOAN-04, ONLY due date and notes are editable on a non-returned loan. Item + borrower + quantity + loaned-at are locked at creation time.

1. **ITEM** (read-only display) — rendered as a `RetroPanel` row: small thumb + item name + SKU (mono). Not a form field; not submittable.
2. **BORROWER** (read-only display) — rendered as a `RetroPanel` row: borrower name + email (mono).
3. **DUE DATE** (editable) — `RetroFormField` + `RetroInput` `type="date"`, mono.
4. **NOTES** (editable) — `RetroFormField` + `RetroTextarea`, 4 rows visible, max 1000 chars.

Footer buttons:
- Create flow: `← BACK` (charcoal outline) left, `CREATE LOAN` (amber fill) right. In-flight label `WORKING…` (mono).
- Edit flow: `← BACK` left, `SAVE LOAN` (amber fill) right. In-flight label `WORKING…` (mono).

Backdrop: charcoal at 40% opacity; click-outside cancels (with unsaved-changes guard identical to Phase 60).

**Edit flow does NOT appear for History-tab rows** — returned loans have no edit affordance (their action column is empty per the table contract above).

## Interaction Contracts

- **Tab switch:** clicking a tab updates `location.hash` to `#active` / `#overdue` / `#history` and swaps the query result below. React Query cache keys are scoped per-tab (`loanKeys.list({ active: true })`, `loanKeys.list({ overdue: true })`, `loanKeys.list({})` for history with a client-side filter). URL hash persists across reload and supports back/forward navigation.
- **Tab counts refresh:** whenever any mutation succeeds (create / edit / mark-returned), all three tab counts refresh by invalidating `loanKeys.lists()` — ensures a returned loan decrements the active count and increments the history count simultaneously.
- **`+ NEW LOAN` click:** opens the create slide-over. Form `defaultValues`: `{ quantity: 1, loaned_at: todayISO(), due_date: "", notes: "" }`, item and borrower both empty. The Item combobox receives focus on open (first focusable field).
- **Slide-over create submit:** calls `loansApi.create(wsId, body)`. On success: close panel, fire `Loan created.` toast, invalidate `loanKeys.lists()` + `itemKeys.detail(inventory_id)` + `borrowerKeys.detail(borrower_id)` (so item detail and borrower detail reflect the new active loan immediately). On failure: keep panel open, show inline error under the submit button, fire the red error toast.
- **`EDIT` row action click:** opens the edit slide-over pre-filled with the selected loan's `due_date` and `notes`. Item + borrower + quantity + loaned-at shown in the read-only block at the top. First focusable field: DUE DATE input.
- **Slide-over edit submit:** calls `loansApi.update(wsId, id, { due_date, notes })` (per Open Data-Shape Decision O-01 option A). On success: close panel, fire `Loan updated.` toast, invalidate `loanKeys.all` + `loanKeys.detail(id)`. On failure: same error handling as create.
- **`MARK RETURNED` row action click:** opens the `CONFIRM RETURN` dialog (not an inline toggle — the confirmation prevents accidents). On confirm, calls `loansApi.return(wsId, id)`. On success: close dialog, fire `Loan returned.` toast, invalidate `loanKeys.lists()` + `loanKeys.detail(id)` + `itemKeys.detail(inventory_id)` + `borrowerKeys.detail(borrower_id)`. Row leaves the current tab and appears in the `HISTORY` tab on next navigation or invalidation. The row's position in the current tab should smoothly unmount (no layout jank — React Query invalidation causes a re-render, and the row disappears because the filter `is_active: false` now excludes it).
- **Slide-over unsaved-changes guard:** identical to Phase 60. If `formState.isDirty`, clicking `← BACK`, the close `×`, the backdrop, or pressing Esc opens a nested mini-confirm dialog: title `DISCARD CHANGES?`, body `Your edits to this loan will be lost.` (use `this loan` in both create and edit modes — no entity-name interpolation needed because create mode has no identifiable name yet), escape `← BACK`, destructive `DISCARD CHANGES` (red). No guard when pristine.
- **Combobox async loading (Item and Borrower pickers):** RetroCombobox calls `itemsApi.list({ archived: false, search: query })` and `borrowersApi.list({ archived: false })` respectively. Item picker uses 300ms debounce. Borrower picker loads the full list once (borrowers are typically small-N) and filters client-side. Both pickers show `Loading…` in the dropdown during fetch, `No matches.` when the query returns nothing.
- **Overdue computation:** the UI trusts the `is_overdue` boolean on each loan (backend-computed). No client-side date math for this signal.
- **Due-date input calendar icon:** `<input type="date">` on supported browsers renders a native calendar picker. On Safari/iOS, the browser date picker opens; mono font is preserved in the input chrome. No custom calendar component in Phase 62.
- **Focus ring:** every focusable control gets `outline: 2px solid var(--color-retro-amber); outline-offset: 2px` on `:focus-visible`. Never remove outline without replacement.
- **Slide-over ARIA:** `role="dialog"` + `aria-modal="true"` + `aria-labelledby` pointing at panel title. Focus trap active. First focusable element receives focus on open.
- **Tab ARIA:** the tab row uses `role="tablist"`; each tab button `role="tab"`, `aria-selected={isActive}`, `aria-controls={panelId}`. The panel below (table + empty state) gets `role="tabpanel"` + matching `id`. Keyboard: ← / → arrow keys navigate between tabs; Home / End jump to first / last. Tab / Shift-Tab move into / out of the tab panel content.
- **Table keyboard navigation:** row focus with Tab cycles through ITEM link → BORROWER link → MARK RETURNED → EDIT. Enter on a link navigates; Enter on an action button invokes it.
- **Icon-only buttons:** every icon-only row action on mobile carries `aria-label` (see Row-level action buttons table).
- **Loading:**
  - Initial tab load: `RetroPanel` skeleton — three placeholder rows with mono dashes in each cell, no shimmer (matches Phase 58/59/60/61 plain-loading approach)
  - Save-in-flight: disables the submit button and swaps its label to `WORKING…` in mono. Row-level action buttons disable while their mutation is pending (gray text, `cursor: not-allowed`).
  - Mark-returned dialog in flight: `RETURN LOAN` button disabled, label `WORKING…`; `← BACK` remains enabled (user can abort).
  - Tab-counter loading: when the three count queries haven't all resolved, show `{LABEL} · …` (mono ellipsis) in the tab label until the count arrives; tab remains clickable during this state.
- **Cache invalidation summary:**
  - Create success → invalidate `loanKeys.lists()` + `itemKeys.detail(inventory_id)` + `borrowerKeys.detail(borrower_id)` + `itemKeys.lists()` (list may show "× has loans" indicator in future; keep in sync) + `borrowerKeys.lists()`
  - Edit success → invalidate `loanKeys.all` (simpler — edit touches a specific loan but lists may show due-date changes)
  - Mark-returned success → invalidate `loanKeys.lists()` + `loanKeys.detail(id)` + `itemKeys.detail(inventory_id)` + `borrowerKeys.detail(borrower_id)`
  - No SSE listeners in this phase
  - No optimistic updates in Phase 62 (match Phase 58/59/60 baseline — `useMutation` `onSuccess` invalidation is enough for the short mutation windows involved)
- **Motion:** honor `prefers-reduced-motion` — slide-over transform, dialog fade/scale, toast slide, tab-indicator transition all replaced with instant state swaps. Overdue AlertTriangle icon has no animation regardless of motion preference.
- **Phase 63 seam:** Phase 62 wires the `LOANS` sidebar link target. Phase 63 finalises the sidebar navigation entry; this contract assumes the existing Phase 54 sidebar LOANS NavLink already points at `/loans` and requires no change here.
- **Item-detail + borrower-detail seam completion:** Phase 62 replaces the Phase 60 `PHOTOS LOANS` placeholder on `/items/:id` and the Phase 59 `ACTIVE LOANS` / `LOAN HISTORY` placeholders on `/borrowers/:id`. The section heading structure and outer `RetroPanel` markup remain; only the body content changes.
