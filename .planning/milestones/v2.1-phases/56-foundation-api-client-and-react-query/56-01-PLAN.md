<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Backend `/api/*` | All entity and auth requests cross this boundary; JWT cookie + in-memory refresh token authenticate. |
| Build-time -> Runtime | `import.meta.env.DEV` decides whether Devtools ship; static-replaced at build time. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-56-01 | Information Disclosure | React Query Devtools in production bundle | mitigate | Lazy-load via `lazy(() => import(...))` inside an `import.meta.env.DEV` gate so Vite tree-shakes it out of prod. Verified by build-step bundle inspection (plan 04 prebuild + production `bun run build`). |
| T-56-02 | Tampering | postMultipart Content-Type header | mitigate | `request()` must OMIT the JSON Content-Type when body is FormData; explicitly tested in api.multipart.test.ts (Tests 1+2). A manual `Content-Type` would break boundary and could enable smuggling. |
| T-56-03 | Spoofing | 401-refresh retry on multipart | mitigate | postMultipart delegates to the single `request()` implementation (no duplicate refresh logic) — verified by api.multipart.test.ts Test 4 asserting exactly one refresh call between retries. |
| T-56-04 | Denial of Service | QueryClient retry storm on auth failures | accept | `retry: 1` caps retries at one, and 401s go through the refresh path (single-flight via `refreshPromise`). Low exposure; no user-facing amplification vector. |
</threat_model>
