### Integration Gaps (Non-Critical)

**1. BATCH-02: Location pill is cosmetic during quick capture** (design decision)
- `settings.locationId` is stored in `BatchCaptureProvider` (Phase 44) and shown in `BatchSettingsBar` (Phase 45), but the Phase 45 `handleSave` explicitly omits `location_id` from the POST payload with a code comment: *"location belongs to inventory, not items."*
- **Impact:** The location selection has no downstream effect on created items during quick capture.
- **Severity:** Design decision, documented in code. BATCH-02 requirement ("user can set a default location") is satisfied by the UI affordance; applying location to items would require a post-sync inventory creation step.

**2. COMP-02: Archive guard applied to server-filtered needs_review results** (edge case)
- When `showNeedsReview=true`, server returns only `needs_review=true` items. Client-side `filteredItems` memo still applies `if (!showArchived && item.is_archived) return false`.
- **Impact:** An item with both `needs_review=true` AND `is_archived=true` would be silently excluded. Extremely unlikely — quick-captured items are new and not archived.
- **Severity:** Edge case, non-blocking.
