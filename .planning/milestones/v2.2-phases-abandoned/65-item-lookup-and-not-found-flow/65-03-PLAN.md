<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → public /barcode/{code} endpoint | Unauthenticated; decoded barcode crosses network to a third-party pass-through (OpenFoodFacts / OpenProductsDB) |
| external API response → React render tree | Third-party returns `name`, `brand`, `category` strings the user will see in UI |
| URL path segment | The barcode is interpolated into the URL path (`/barcode/${code}`) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-65-03-01 | Injection (V5) | barcodeApi.lookup URL construction | mitigate | Use `encodeURIComponent(code)` in the URL template literal: `get(\`/barcode/${encodeURIComponent(code)}\`)`. Combined with the Plan 65-02 schema's `/^[A-Za-z0-9\-_]+$/` and the hook's `/^\d{8,14}$/` gate (this plan) the attack surface is: `enabled` blocks non-UPC characters → characters reaching the URL are `/\d{8,14}/` — but defense-in-depth with `encodeURIComponent` costs nothing and protects if the gate is ever loosened |
| T-65-03-02 | Denial of service (upstream) | useBarcodeEnrichment fetch cadence | mitigate | Regex gate `/^\d{8,14}$/` prevents spurious calls for QR URLs / short codes; `staleTime: Infinity` keeps session-scoped cache so rescans don't re-fetch; `retry: false` prevents amplification on transient errors |
| T-65-03-03 | Information disclosure | user's scanned barcode → third-party endpoint | accept | The decoded numeric code IS the query — it has no PII by construction (EAN/UPC numbers are public product identifiers). Response is never persisted server-side by the frontend. Document in action text |
| T-65-03-04 | Tampering (XSS via response fields) | banner rendering of name/brand/category strings | mitigate | React auto-escapes JSX `{data.name}` / `{data.brand}`; NO `dangerouslyInnerHTML` anywhere in Phase 65. Response fields typed as strings only — the hook does not evaluate or parse them. (Render-site enforcement lives in Plan 65-05's UpcSuggestionBanner) |
| T-65-03-05 | Repudiation | silent-failure observability | mitigate | D-16 structured log `console.error({ kind: "upc-enrichment-fail", code, error, timestamp })` — dev-tools surfaces the failure without user-facing UI disruption |
</threat_model>

    ```ts
    // Phase 65 LOOK-03 — external UPC enrichment hook. Calls
    // barcodeApi.lookup(code) via TanStack Query. Gated by /^\d{8,14}$/ so
    // QR URLs and short alphanumeric codes never hit the public endpoint.
    // Silent-failure: staleTime: Infinity + retry: false + structured
    // console.error on queryFn throw (kind: "upc-enrichment-fail").
    //
    // D-15: the consumer (ItemFormPage / UpcSuggestionBanner) is responsible
    // for rendering the banner only when data?.found === true — this hook
    // does not filter found:false responses.
    import { useQuery } from "@tanstack/react-query";
    import { barcodeApi, barcodeKeys, type BarcodeProduct } from "@/lib/api/barcode";

    const NUMERIC_8_TO_14 = /^\d{8,14}$/;

    export function useBarcodeEnrichment(code: string | null) {
      return useQuery<BarcodeProduct>({
        queryKey: barcodeKeys.lookup(code ?? ""),
        queryFn: async () => {
          try {
            return await barcodeApi.lookup(code!);
          } catch (err) {
            console.error({
              kind: "upc-enrichment-fail",
              code,
              error: err instanceof Error ? err.message : String(err),
              timestamp: Date.now(),
            });
            throw err;
          }
        },
        enabled: !!code && NUMERIC_8_TO_14.test(code),
        staleTime: Infinity,
        gcTime: Infinity,
        retry: false,
      });
    }
    ```

    Note: `NUMERIC_8_TO_14.test(code)` not `NUMERIC_8_TO_14.test(code ?? "")` because `!!code` short-circuits when code is null; writing it without the fallback keeps the type assertion `code!` in `queryFn` safe-by-construction.
