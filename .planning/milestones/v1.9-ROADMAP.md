# Milestone v1.9: Quick Capture

**Status:** ✅ SHIPPED 2026-03-14
**Phases:** 43-47
**Total Plans:** 9

## Overview

Quick Capture adds a camera-first rapid item entry mode for mobile bulk onboarding. Users snap photos, type a name, and move to the next item — capturing dozens of items in minutes instead of filling out full forms for each. Items are flagged "needs details" for later desktop completion. The entire flow works offline with queued sync, including photo upload chaining after item sync resolves temporary IDs to real server IDs.

## Milestone Summary

**Key Decisions:**

- Single-route quick capture design — iOS PWA camera permission is volatile; navigating away resets it
- Photos separate from mutation queue — new `quickCapturePhotos` IndexedDB store; chained upload after item sync resolves temp IDs
- Zero new npm dependencies — all features built on existing stack
- `needs_review` as simple boolean with `DEFAULT false` — threaded through entity/service/repo/handler/sync
- `BatchCaptureProvider` sessionStorage-backed context — persists within browser session, clears on tab close
- Pre-write `resolvedItemId` pattern — ensures photo retry is always stateless even if process interrupted mid-loop
- Location pill is cosmetic during quick capture — location belongs to inventory, not items (documented in code comment)

**Issues Resolved:**

- iOS camera permission persistence achieved via single-route design
- Photo upload chaining solved via `MUTATION_SYNCED` event from OfflineContext
- Offline item pending indicator achieved via mount-time IndexedDB scan in items page

**Technical Debt Incurred:**

- Stale closure in unmount useEffect in QuickCapturePage — photos captured at mount may not be revoked on navigate-away; mitigated by per-action revocation
- 2 pre-existing test failures in `use-offline-mutation.test.ts` (workspaceId assertions) — not introduced by v1.9
- 4 pre-existing test failures in `frontend/lib/api/__tests__/client.test.ts` — unrelated to v1.9
- Nyquist compliance PARTIAL — Phases 43-45 have no VALIDATION.md; Phases 46-47 have VALIDATION.md but unsigned
