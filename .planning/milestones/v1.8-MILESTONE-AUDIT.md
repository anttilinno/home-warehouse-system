## Critical Gap: ACCT-05

**Requirement:** OAuth-only user can set a password from Security settings (no current password required)

**Root Cause:** The `UpdatePasswordInput` struct in `handler.go:1083-1088` marks `current_password` with `required:"true"`:

```go
type UpdatePasswordInput struct {
    Body struct {
        CurrentPassword string `json:"current_password" required:"true"`
        NewPassword     string `json:"new_password" required:"true" minLength:"8"`
    }
}
```

The frontend `setPassword()` in `auth.ts:137-139` sends only `{ new_password }`:

```typescript
setPassword: async (newPassword: string): Promise<void> => {
    await apiClient.patch("/users/me/password", { new_password: newPassword });
}
```

Huma v2 validates the JSON request against the generated OpenAPI schema **before** the Go handler runs. Since `current_password` is in the JSON Schema `required` array but absent from the request body, Huma returns 422 Unprocessable Entity. The request never reaches the service layer which correctly handles OAuth-only users (service.go:178-184).

**Fix:** Remove `required:"true"` from `CurrentPassword` in `UpdatePasswordInput`. The service layer already handles the empty-password case correctly.

**Route analysis correction:** The integration checker incorrectly flagged a "dual callback" issue, assuming `/auth/callback` served the OLD callback at `app/[locale]/(auth)/callback/page.tsx`. In fact, `(auth)` is a Next.js route group (no URL segment), so:
- `app/[locale]/(auth)/callback/page.tsx` → URL: `/{locale}/callback` (dead code)
- `app/[locale]/(auth)/auth/callback/page.tsx` → URL: `/{locale}/auth/callback` (correctly served)

The backend redirects to `{APP_URL}/auth/callback`, which with `localePrefix: "as-needed"` correctly serves the NEW callback page. All error params (`?oauth_error=`), sessionStorage keys (`oauth_return_to`), and linking flow logic work as designed.
