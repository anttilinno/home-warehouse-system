Verified backend surface (orchestrator + planner re-verified pendingchange/handler.go this session):
- `GET /pending-changes?status=pending|approved|rejected` → `{ changes: PendingChangeResponse[], total }`.
  Owner/admin ONLY — the handler calls canReviewChanges and returns `huma.Error403Forbidden` for
  everyone else. Envelope key is `changes` (handler.go:381), NOT `items`.
- `POST /pending-changes/{id}/approve` → applies the change, returns the updated PendingChangeResponse
  (owner/admin; 403 else; 400 if already reviewed).
- `POST /pending-changes/{id}/reject` → body `{ "reason": string }` (minLength 1) → rejects (same gating;
  400 if already reviewed).
- There is NO `defer` endpoint. ROADMAP success-criteria-1 says "A/R/D" but Defer has no backend —
  ship Approve/Reject ONLY and document the omission (OQ1 resolved). Do NOT add a client-only Defer
  no-op chip that implies server state.
- PendingChangeResponse fields used by the table: `id, requester_name, requester_email, entity_type,
  entity_id?, action (create|update|delete), status (pending|approved|rejected), created_at`.

Bulk = iterate the selected ids (NO bulk endpoint). Fire the per-id mutations, collect results, and
on ANY failure surface a partial-failure toast ("Approved N, M failed") — do NOT assume all-or-nothing
(VALIDATION landmine). Invalidate the list once after the batch settles.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| frontend → /api/workspaces/{ws}/pending-changes* | owner/admin-gated review writes cross here; the server re-checks canReviewChanges on every approve/reject (the UI guard is cosmetic, the server is authoritative) |
| pending-change payload → rendered table | server-supplied requester/entity strings render into the TUI table; React escapes them — no raw HTML |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Elevation of Privilege | a non-admin reaching approve/reject | mitigate | the server gates every list/approve/reject on canReviewChanges (403); the UI hides the table on isForbidden but never relies on hiding for security — the server is authoritative |
| T-14-02 | Information Disclosure | 403 list error storm leaking existence via retries | mitigate | retry:false (one settle) + isForbidden renders a calm static guard; no count/existence leak, no retry-spam |
| T-14-03 | DoS | shortcut/effect render-loop | mitigate | bindings useMemo'd, actions useCallback over stable refs, t via ref; the page test guards against the loop (Landmine hit 4×) |
| T-14-04 | Tampering | bulk approve assuming all-or-nothing on partial failure | mitigate | Promise.allSettled per-id, a partial-failure toast, and a single post-batch invalidate so the table reflects exactly what the server applied |
| T-14-SC | Tampering | npm installs | mitigate | none — this plan installs NO packages (composes existing @tanstack/react-query + @/lib/api + retro atoms) |
</threat_model>
