SYS-04 — the /imports page: a CSV import action + an import-history jobs table + a workspace export,
all in the activity-table pattern. Reuses the existing settingsApi.exportWorkspace for the workspace

Verified backend surface (planner re-verified importjob/handler.go + importexport/handler.go this session):
- `GET /imports/jobs?page&limit` → `{ jobs: ImportJobResponse[], total, page, total_pages }`. Envelope key
  is `jobs` (importjob/handler.go:101) — NOT items/changes. limit default 20, max 100.
- `GET /imports/jobs/{id}` → one ImportJobResponse; `GET /imports/jobs/{id}/errors` → `{ errors:
  ImportErrorResponse[], total }`.
- `POST /imports/upload` — **multipart/form-data** (Chi handler, importjob/upload_handler.go): form fields
  `entity_type` (string) + `file` (the upload). 10 MB limit. CREATES an ImportJob + enqueues the async
  worker → returns 201 + the job JSON. THIS is the endpoint that populates `GET /imports/jobs`.
  entity_type ∈ **items, inventory, locations, containers, categories, borrowers** (importjob EntityType —
  PLURAL, includes `inventory`, excludes label/company). Verified upload_handler.go:70-81.
  (NOTE: the OTHER endpoint `POST /import/{entity_type}` in the importexport domain is SYNCHRONOUS — returns
  ImportResult inline, creates NO job — so it does NOT feed the jobs table. Do NOT use it here; the coherent
  upload→job→history flow is /imports/upload.)
- ImportJobResponse fields used by the table: `id, entity_type, status (pending|processing|completed|failed|
  cancelled), file_name, file_size_bytes, total_rows?, processed_rows, success_count, error_count, progress
  (int 0–100), created_at, completed_at?, error_message?`.
- Workspace export already exists in settingsApi: `settingsApi.exportWorkspace(wsId, "xlsx"|"json")` →
  downloadBlob of GET /export/workspace (admin-gated). REUSE it.

The import worker is async (the POST enqueues a job; the jobs table reflects progress). After a successful
import POST, invalidate the jobs list so the new job appears. Admin gating: the export button + the import
form should both surface a calm "requires admin" state when the current workspace role is not owner/admin
(mirror DataStoragePage's ADMIN_ROLES gate), but the SERVER is authoritative (403 handled defensively).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| frontend → POST /api/workspaces/{ws}/imports/upload | admin-gated bulk write crosses here (bypasses the member approval pipeline); the server is authoritative — the UI gate is cosmetic |
| chosen file → multipart → server | a user-supplied file is uploaded as multipart FormData; the server caps payload size (10 MB) + validates entity_type |
| job payload → rendered table | server-supplied file_name/error_message render into the TUI table; React escapes them — no raw HTML |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-14 | Elevation of Privilege | a non-admin reaching the import write | mitigate | the server requireAdminRole returns 403; the UI gates the import form on ADMIN_ROLES but never relies on hiding for security — the server is authoritative; a 403 surfaces a calm toast |
| T-14-15 | DoS | a huge file uploaded | mitigate | the server caps the multipart payload (10 MB → 400); the UI accepts only .csv and can pre-warn on large files; one-shot upload, not looped |
| T-14-16 | Tampering | offline sync engine import (FOUND-02 CI guard) | mitigate | the page imports NO `sync*` module — only @/lib/api + react-query + retro atoms; lint:imports stays green |
| T-14-SC | Tampering | npm installs | mitigate | none — installs NO packages (composes existing react-query + @/lib/api + settingsApi + retro atoms) |
