Verified backend surface (planner re-verified wishlist/handler.go + the input/output structs this session):
- `GET /wishlist?status=wanted|ordered|acquired&page&limit` → `{ items: WishlistItemResponse[], total }`.
  Envelope key is `items` (handler.go:253) — NOT `changes`. Omit `status` for all. limit default 50, max 100.
- `POST /wishlist` body: `{ name (required, 1–200), notes?, url? (≤2000), price_estimate? (CENTS, ≥0),
  currency_code? (^[A-Z]{3}$), priority? (1–5, default 3), desired_category_id? }` → WishlistItemResponse.
- `PATCH /wishlist/{id}` body: all create fields OPTIONAL plus `status? (wanted|ordered|acquired)` and
  `acquired_item_id?`. Setting `status:"acquired"` (optionally with acquired_item_id) is the "mark acquired"
  path — there is NO separate acquire endpoint. An illegal transition returns 409 Conflict
  (ErrInvalidStatusTransition); a bad field returns 400.
- `DELETE /wishlist/{id}` → 204.
- WishlistItemResponse fields used by the table/form: `id, name, notes?, url?, price_estimate? (CENTS),
  currency_code?, priority (int), desired_category_id?, status (string), acquired_item_id?, created_at`.

price_estimate is CENTS → format via the existing null-safe `formatCents` (@/lib/utils/money);
NEVER pass a null currency through money formatting (white-screen landmine — guard the optional currency).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| frontend → /api/workspaces/{ws}/wishlist* | workspace-scoped CRUD crosses here; the server scopes every read/write to {ws} and runs member writes through the approval pipeline |
| wishlist payload (name/notes/url) → rendered table + CSV-free UI | server strings render into the TUI table; React escapes them; a url is rendered as an anchor with rel/noopener |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-07 | Tampering | client trusting a status transition the server rejects | mitigate | the server validates transitions (409 ErrInvalidStatusTransition); the form surfaces the 409 as a calm error and re-reads the list — the client never assumes a transition succeeded |
| T-14-08 | DoS | null currency through money formatting (white-screen) | mitigate | formatCents is called with a guarded currency (?? fallback); a null currency renders "—" or the major unit, never crashes (VALIDATION landmine) |
| T-14-09 | Spoofing/XSS | a wishlist `url` rendered as a link | mitigate | render the url as an anchor with `rel="noopener noreferrer"` and target="_blank"; React escapes the href text; no javascript: scheme passthrough (validate http(s) prefix) |
| T-14-SC | Tampering | npm installs | mitigate | none — installs NO packages (composes existing react-hook-form + zod + @/lib/api + retro atoms, all already in the project) |
</threat_model>
