## Open Data-Shape Decision (planner must resolve)

The items-list thumbnail column (D-08) needs a primary-photo URL per item row. The current `Item` type carries no such field. Three planner-level options, this contract is data-source-agnostic and works with all three:

| Option | Frontend impact | Backend impact |
|--------|-----------------|----------------|
| A. Add `primary_photo_thumbnail_url` (and optionally `primary_photo_url`) to the `Item` list response | Single batched render; trivial — append a column reading `item.primary_photo_thumbnail_url` | Backend join in items list query |
| B. Frontend issues `itemPhotosApi.listForItem()` per visible row | No backend change | N requests per page (25 rows = 25 GETs); rejected unless paired with caching |
| C. New endpoint `itemPhotosApi.listPrimaries(itemIds[])` returning a map | Single batched fetch keyed by item IDs after list loads | One new endpoint |

The UI contract **mandates** that whatever option is chosen, the items list MUST render a thumbnail cell for every visible row in a single render pass — no per-row spinners cascading after the table mounts. If Option B is taken, the planner MUST add aggressive React Query caching (`staleTime: Infinity` until invalidation) and a placeholder tile during the per-item fetch. Option A is strongly preferred for clarity and performance.

## Color

Inherited 60/30/10 from v2.0 / Phase 57. Phase 61 mirrors Phase 60's reservation model and adds **one new accent site** (the primary-photo badge fill).

| Role | Value | Phase 61 usage |
|------|-------|----------------|

3. **`SET AS PRIMARY` button** — amber fill (primary CTA inside the lightbox control bar). When the currently-displayed photo is already primary, the button collapses to a disabled `★ PRIMARY` state with charcoal outline and no amber fill (no orphan amber on a no-op control).

## Focal Point

### `/items/:id` PHOTOS section (when photos exist)

1. **Primary focal point:** the **primary photo's gallery tile** — it is the only tile carrying the amber `★ PRIMARY` badge. The badge alone draws the eye to it among an otherwise uniform charcoal-on-cream grid.
2. **Secondary focus:** the rest of the gallery grid (3–4 cols of square cream tiles with charcoal borders), reading left-to-right, top-to-bottom.
3. **Tertiary focus:** the `ADD PHOTOS` button above the grid (charcoal outline, not amber — see Color reservation).

### Lightbox overlay (when open)

1. **Primary focal point:** the **photo itself**, centered on the dark backdrop, framed with the cream border + amber drop-shadow. The amber shadow is the only accent on the dark backdrop and pulls the eye to the photo.
2. **Secondary focus:** the bottom control bar (cream-on-charcoal-border), with `SET AS PRIMARY` (amber fill) as the secondary focal accent. When the displayed photo is already primary, the button collapses to a non-amber `★ PRIMARY (disabled)` state and `DELETE PHOTO` becomes the only highlighted control.
3. **Tertiary focus:** prev/next chevrons and the photo counter (`2 / 5`) flanking the bar.
4. **Escape affordances:** top-right `×` close, ESC key, backdrop click — all equally weighted, none accent-colored.

### Items list (`/items`) thumbnail column (D-08)

The new `[thumb]` column is **decorative**, NOT focal. Tiles are 40px squares with charcoal border, no amber treatment, no badge. The amber `+ NEW ITEM` button (Phase 60 focal) remains the primary focal point of the list page.

## Interaction Contracts

- **ADD PHOTOS click:** opens the native file picker via the existing `RetroFileInput` component (D-04). The button itself is the trigger; we do not render `RetroFileInput`'s default chip-list UI inside the gallery (the gallery IS the chip list). Implementation may use `RetroFileInput` with `value={[]}` and intercept `onChange`, or it may inline a thin wrapper around `<input type="file" hidden />` + `<RetroButton>`. Either approach satisfies the contract.
- **File-picker accept attribute (D-10):** `image/jpeg,image/png,image/webp`. **Action item for the planner:** the existing `RetroFileInput` constant `DEFAULT_ACCEPT` in `frontend2/src/components/retro/RetroFileInput.tsx:22` MUST be updated from `image/jpeg,image/png,image/heic` to `image/jpeg,image/png,image/webp`. Helper text in the same file (line 121) must update from `JPEG, PNG, or HEIC up to 10 MB each.` to `JPEG, PNG, or WebP up to 10 MB each.`
- **HEIC client-side rejection (D-10):** even with the updated accept attribute, drag-and-drop or some browsers may surface HEIC files. After the file picker returns, iterate the selected files and split into `accepted` (mime starts with `image/jpeg`, `image/png`, or `image/webp`) and `rejected` (anything else, including `image/heic`, `image/heif`). For each rejected file, surface the inline error `HEIC not supported — convert to JPEG or PNG first.` Other rejected types use a generic `{filename} is not a supported image format.` message. Accepted files proceed to upload.
- **Sequential auto-upload (D-05):** after acceptance, files upload one-by-one via `itemPhotosApi.upload(wsId, itemId, file)`. The ADD PHOTOS button shows the disabled `WORKING…` state for the entire batch (D-06). Each successful upload invalidates `itemPhotoKeys.list(itemId)` immediately so the new tile appears in the grid. If any upload fails, the toast surfaces, the batch continues with the next file (no batch abort), and at the end of the batch the button re-enables.
- **No per-file progress bars (D-06):** there is no individual upload progress UI — the disabled button + the gallery filling in tiles is the progress signal.
- **Tile click → lightbox open:** clicking any tile opens the lightbox at that photo's index. The grid scroll position is preserved underneath. The lightbox uses `<dialog>` semantics via React portal.
- **Lightbox keyboard navigation:**
  - `←` Arrow Left → previous photo (no-op at index 0)
  - `→` Arrow Right → next photo (no-op at last index)
  - `Esc` → close lightbox
  - `Tab` → cycles between control-bar buttons (close, prev, next, SET AS PRIMARY, DELETE PHOTO) — focus trapped inside the lightbox
  - `Enter` / `Space` on focused button → invokes button
- **Lightbox edge tap (mobile):** left third of image area = prev; right third = next. Visible chevron buttons remain in the control bar.
- **Lightbox prev/next at boundaries:** PREV button disabled on first photo (charcoal/40); NEXT button disabled on last photo. No wrap-around.
- **Set-primary interaction (D-07):** clicking `★ SET AS PRIMARY` calls `itemPhotosApi.setPrimary(wsId, photoId)` (D-09). Optimistic update: locally toggle `is_primary` flags in the cached photo list immediately so the badge moves; on success, invalidate `itemPhotoKeys.list(itemId)` and (per Claude's discretion in CONTEXT) ALSO invalidate `itemKeys.detail(itemId)` and `itemKeys.lists()` so the items list / detail header thumbnail updates. On failure, roll back the optimistic flag and surface the error toast `Could not update primary photo. Try again.`
- **Delete photo from lightbox:** triggers the `RetroConfirmDialog` (single-step destructive — NOT archive-first; photos have no archived state). On confirm, `itemPhotosApi.remove(wsId, photoId)` runs. On success, the dialog closes, success toast appears, the lightbox advances to the next photo (or previous if last; closes if empty). React Query invalidations: `itemPhotoKeys.list(itemId)`. If the deleted photo was the primary, additionally invalidate `itemKeys.detail(itemId)` and `itemKeys.lists()` (the items-list thumb cell will fall back to placeholder until a new primary is auto-promoted by backend or selected by user — backend behaviour is planner's call).
- **Pending-thumbnail handling (D-11):** the gallery polls **nothing**; React Query refetch on focus + invalidation after upload is sufficient. Tiles whose `thumbnail_url` is empty render the `PROCESSING…` placeholder. The lightbox always uses the full-size `url` field, which is always available immediately after upload — so users can view full-size images even before thumbnails are ready. SSE-driven thumbnail readiness is deferred (CONTEXT).
- **ObjectURL cleanup (D-12, success criterion #5):** if the implementation uses `URL.createObjectURL(file)` for client-side preview before upload (e.g., to optimistically render an uploaded tile while the server response is in flight), every created URL MUST be tracked in a `useRef<Set<string>>` and revoked via `URL.revokeObjectURL` either (a) when its corresponding tile unmounts, or (b) when the gallery component unmounts (whichever comes first). Hard requirement per success criterion #5.
- **List-page thumbnail data fetch:** see "Open Data-Shape Decision" at the top. Whatever the planner picks, the rendered cell MUST appear in the same render pass as the rest of the row — no per-row spinners cascading.
- **Archived item gallery:** if the item is archived, photos are still viewable (read-only). The ADD PHOTOS button hides on the archived detail page, and the lightbox hides the SET AS PRIMARY and DELETE PHOTO buttons (read-only). PREV/NEXT/CLOSE remain functional. (Backend permits photo CRUD on archived items, but the UI prevents it to match the archived-item read-only convention from Phase 60.)
- **Focus ring:** every focusable control gets `outline: 2px solid var(--color-retro-amber); outline-offset: 2px` on `:focus-visible`. Lightbox controls retain focus rings against the dark backdrop (the amber is sufficiently visible).
- **ARIA:** the lightbox uses `role="dialog"`, `aria-modal="true"`, `aria-labelledby="lightbox-title"` where `lightbox-title` is an `sr-only` heading reading `Photo {N} of {M} for {item.name}`. Image carries `alt="Photo of {item.name}, {N} of {M}"`. Icon-only buttons (close, prev, next on mobile) carry explicit `aria-label`.
- **Loading:**
  - Initial gallery load: `RetroPanel` skeleton — three placeholder tiles (8 mobile / 12 desktop) with the `PROCESSING…` mini-swatch — no shimmer (matches Phase 58/59/60 plain-loading approach).
  - List-page thumbnail cells during initial load: charcoal-bordered placeholder with the `<ImageOff>` glyph (no shimmer).
  - Upload-in-flight: ADD PHOTOS button disabled, label swapped to `WORKING…` (mono).
  - Lightbox set-primary in flight: SET AS PRIMARY button disabled, label `WORKING…`.
  - Lightbox delete in flight: DELETE PHOTO button disabled, label `WORKING…`; the close `×` and prev/next buttons remain enabled (user can navigate away mid-delete).
- **Cache invalidation summary:**
  - Upload success → invalidate `itemPhotoKeys.list(itemId)`
  - Set-primary success → invalidate `itemPhotoKeys.list(itemId)` + `itemKeys.detail(itemId)` + `itemKeys.lists()` (Claude's discretion per CONTEXT)
  - Delete success → invalidate `itemPhotoKeys.list(itemId)`; if deleted was primary, also `itemKeys.detail(itemId)` + `itemKeys.lists()`
  - No SSE listeners in this phase
- **Motion:** honor `prefers-reduced-motion` — lightbox fade-in/out replaced by instant swap; tile hover lift replaced by border-color change only; pending-thumbnail placeholder is static (no shimmer/pulse regardless of motion preference).
