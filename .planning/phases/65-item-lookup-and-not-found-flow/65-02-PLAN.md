<objective>
Add `itemsApi.lookupByBarcode(wsId, code)` to the items API domain with D-06 (wraps list), D-07 (case-sensitive exact match), and D-08 (workspace_id defense-in-depth + structured log) inlined in the helper so every future caller gets both guards automatically. Simultaneously apply the two user-locked schema scope additions from 65-CONTEXT.md:
- **D-23:** `itemCreateSchema` gains `brand: z.string().max(120).optional()` so UpcSuggestionBanner can write to a first-class `brand` field.
- **D-24:** `itemCreateSchema.barcode` regex loosens to `/^[A-Za-z0-9\-_]+$/` so real Code128 scans (e.g. `"TEST-CODE-123"`) can be saved via `/items/new`.

Both D-23 and D-24 are user-approved in CONTEXT.md (refreshed 2026-04-19) and the Plan 65-01 Wave 0 schemas.test.ts scaffold already has `it.todo` entries for them — Task 1 RED→GREEN converts those todos to real assertions.

Purpose: LOOK-01 says the user sees the matched item via `GET /api/workspaces/{wsId}/items?search={code}&limit=1` with an exact-match guard AND the frontend asserts workspace_id matches before rendering (Pitfall #5 — globally unique UPCs guarantee cross-tenant collisions). Wrapping both guards inside `lookupByBarcode` means the `useScanLookup` hook in Plan 65-04 CAN'T forget them. Meanwhile Phase 64 test fixtures (`HIST-01`, `TEST-CODE-123`, etc.) already use hyphens — the existing `/^[A-Za-z0-9]+$/` regex would cause successful scans to fail zod validation on `/items/new`, which violates LOOK-02's "barcode pre-filled" invariant. And D-23 closes the scope-reduction gap where UpcSuggestionBanner's BRAND [USE] chip was routing to `description` with a `Brand: ` prefix as a workaround — with a first-class `brand` form field, the banner can setValue directly, matching the copy deck contract.

Output: `itemsApi.lookupByBarcode` method + 5 real `it()` assertions replacing scaffolded `it.todo` in the API test + loosened barcode schema + new optional brand field + 5 green schema tests (D-23 × 2 + D-24 × 3) replacing the Plan 65-01 scaffolded `it.todo` entries.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend list endpoint → frontend lookup helper | Backend-returned item may belong to a DIFFERENT workspace on a backend regression; frontend must defend |
| scanned barcode → zod schema → DB | User-typed OR machine-scanned barcode string crosses into DB via `useCreateItem` |
| enrichment brand string → zod schema → DB | Third-party (OpenFoodFacts) brand string crosses into DB via user-initiated [USE] click — zod is the first-line validator |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-65-02-01 | Information disclosure | itemsApi.lookupByBarcode workspace guard | mitigate | D-08 defense-in-depth: assert `candidate.workspace_id === wsId` after list fetch; on mismatch return null AND emit structured `console.error({ kind: "scan-workspace-mismatch", code, returnedWs, sessionWs })`. Prevents Pitfall #5 cross-tenant UPC leak even if backend regresses. Verified by Task 2 test case |
| T-65-02-02 | Tampering | itemsApi.lookupByBarcode exact-match guard | mitigate | D-07 case-sensitive `candidate.barcode !== code` return null. Prevents a substring-match false positive (e.g. server returns barcode `"5449000000996"` when user scanned partial `"5449"`) |
| T-65-02-03 | Injection (V5 Input Validation) | schemas.ts barcode regex (D-24) | mitigate | Keep regex character-class restrictive: `/^[A-Za-z0-9\-_]+$/` still rejects spaces, control chars, quotes, URL-reserved chars — preserves zod's role as first-line input validator. Max-64 length cap unchanged |
| T-65-02-04 | Injection (V5 Input Validation) | schemas.ts brand field (D-23) | mitigate | `z.string().max(120).optional()` — 120-char cap mirrors Item.brand field size and protects the form from oversized third-party strings. React JSX auto-escapes the rendered value at the banner/detail site. No regex needed — brand is free text (trademarks contain special chars) but length is capped |
| T-65-02-05 | Repudiation | observability of cross-tenant attempts | mitigate | Structured log uses a specific `kind: "scan-workspace-mismatch"` so future telemetry ingest can alert on repeated occurrences (signals backend regression) |
</threat_model>
