### Accent (10%) reserved for ‚Äî explicit list

`retro-amber` appears on **exactly these surfaces** in Phase 65:

1. `SCAN AGAIN` button (MATCH / NOT-FOUND / LOADING / ERROR banner states ‚Äî `variant="primary"`)
2. `VIEW ITEM` button (MATCH state banner ‚Äî `variant="primary"`)
3. `CREATE ITEM WITH THIS BARCODE` button (NOT-FOUND and ERROR state banners ‚Äî `variant="primary"`)
4. `RETRY` button (ERROR state banner ‚Äî `variant="primary"`)
5. `CREATE ITEM` submit button on `/items/new` page footer (`variant="primary"`)
6. Per-field `[USE]` chips inside `UpcSuggestionBanner` (`variant="primary"` ‚Äî each suggestion row's accept affordance)
7. `USE ALL` button at bottom of `UpcSuggestionBanner` (`variant="primary"`)
8. The format pill inside MATCH / NOT-FOUND / LOADING states (existing Phase 64 `bg-retro-amber` pill pattern)
9. Focus-ring outline on every interactive element (inherited `focus-visible:outline-retro-amber`)

**Never used for:** neutral "cancel / dismiss / back / scan-again-when-secondary" buttons (those use `variant="neutral"` on cream), nor for the `DISCARD` destructive action (uses `variant="danger"` red), nor for body copy, nor for section backgrounds.

### Semantic: Destructive

- **Destructive stripe (`retro-red`):** ERROR state hazard bar on the widened `ScanResultBanner` (D-21) ‚Äî transient infra failure, visually louder than a soft yellow stripe.
- **Destructive fill (`retro-red`):** `DISCARD` button inside the dirty-guard `RetroConfirmDialog` when the user abandons `/items/new` with unsaved edits. Matches existing `SlideOverPanel` pattern (`variant="destructive"` + `"DISCARD CHANGES?"` title).

### Semantic: Warning (hazard-stripe)

Used for **non-destructive attention** states:
- `NOT-FOUND` banner (D-19) ‚Äî user action needed but nothing was harmed
- `UpcSuggestionBanner` (D-13) ‚Äî opt-in information, never overwrites user input
- LOOKING UP state ‚Äî **no stripe** (avoid alarming the user during a normal <1s query)

### Error / empty / disabled states summary

| Scenario | What the user sees | Copy |
|----------|-------------------|------|

## Interaction Contract

Beyond visual tokens, these interaction rules are part of the design contract for this phase:

| Surface | Rule |
|---------|------|
| `/scan` post-decode | Scanner stays mounted and paused; banner renders immediately in LOADING state; swaps to MATCH / NOT-FOUND / ERROR when `useScanLookup` resolves (D-20) |
| `SCAN AGAIN` (every banner state) | Always interactive ‚Äî even during LOADING (D-20). Unpauses scanner + resets banner state. Never waits for the in-flight query. |
| `VIEW ITEM` click (MATCH) | `navigate("/items/" + match.id)` ‚Äî no confirmation, no dirty-guard (banner has no form state) |
| `CREATE ITEM WITH THIS BARCODE` click (NOT-FOUND + ERROR) | `navigate("/items/new?barcode=" + encodeURIComponent(code))` ‚Äî scanner stays mounted on `/scan` per Phase 64 invariant; the user arrives at `/items/new` with barcode pre-filled |
| `/items/new` mount | `generateSku()` fires exactly once per mount; barcode pulled from `useSearchParams()`; `useBarcodeEnrichment(barcode)` fires iff `/^\d{8,14}$/.test(barcode)` (D-02, D-12) |
| `UpcSuggestionBanner` `[USE]` chip | `setValue(field, suggestion, { shouldDirty: true })` ‚Äî dirty flag propagates through `onDirtyChange` to the dirty-guard (D-14) |
| `UpcSuggestionBanner` dismissal | Local component state only ‚Äî never writes to the query cache or form; user can regenerate by navigating-away-and-back (D-14) |
| Dirty-guard trigger | CANCEL button click, browser back, React Router navigate, **and** tab/window close attempt (beforeunload) ‚Äî standard `ItemPanel` parity |
| Focus order on `/items/new` load | 1. Name field (react-hook-form autofocus), 2. SKU, 3. Barcode, 4. Description, 5. Category, 6. CREATE, 7. CANCEL. No focus ring stolen by `UpcSuggestionBanner` on mount ‚Äî suggestions announce via `aria-live="polite"` but don't steal focus. |
| Keyboard | `Esc` on dialog closes it (inherited from `RetroDialog`); `Enter` on `[USE]` chip applies suggestion; `Tab` walks banner rows in DOM order name ‚Üí brand ‚Üí USE ALL ‚Üí DISMISS |

## Accessibility Contract

| Item | Requirement |
|------|-------------|
| Color contrast | `retro-ink` on `retro-cream` = 14.8:1 (AAA); `retro-amber` on `retro-ink` button = 7.4:1 (AAA); `retro-red` on white = 4.9:1 (AA). All meet WCAG AA at 14px and AAA at 20px+. |
| Touch target | 44√ó44px floor (inherited from `RetroButton`) on every interactive banner action + `[USE]` chip |
| Focus visible | 2px `retro-amber` outline with 2px offset on every focusable element (inherited) |
| Screen reader state change | LOADING ‚Üí MATCH / NOT-FOUND / ERROR transition announced via `aria-live="polite"` region on the banner root. `UpcSuggestionBanner` renders inside an `aria-live="polite"` region so screen-reader users hear suggestions arrive. |
| Heading hierarchy | `/items/new` page uses single `<h1>NEW ITEM</h1>` ‚Üí banner heading drops to `<h2>` in context ‚Üí no skipped levels |
| Form labels | Inherited from `RetroFormField` ‚Äî every input has a visible label (no placeholder-only labels) |
| Reduced motion | The blinking cursor in LOADING state must respect `prefers-reduced-motion: reduce` ‚Äî the cursor glyph renders static (no animation) when the user opts out. CSS guard on the `@keyframes retro-cursor-blink` rule. |
