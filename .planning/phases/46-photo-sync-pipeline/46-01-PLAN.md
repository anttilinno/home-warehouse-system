<objective>
Fix two specific gaps in the photo sync pipeline left after Phase 44/45:

1. **SYNC-03** — `handleItemSynced` currently deletes all photos for a temp ID unconditionally after the upload loop, silently discarding any that failed. Fix: delete each photo individually only after a confirmed upload, mark failures as `status: "failed"` with `resolvedItemId` stored, and add a `retryFailedPhotoUploads` function called from `OfflineContext` on `SYNC_COMPLETE` and the `online` event.

2. **SYNC-04** — The items page only knows about optimistic pending items created via its own `useOfflineMutation`. Quick-capture items created on the quick-capture route are invisible until sync completes. Fix: load pending creates from IndexedDB on mount using the existing `getPendingCreates` helper and merge them into `optimisticItems`.

Purpose: Complete the v1.9 offline guarantee — no photo data is lost on partial failure, and users see all pending items regardless of which route created them.

Output: Modified `types.ts`, `capture-photo-uploader.ts`, `offline-context.tsx`, `items/page.tsx`, and updated test file with retry coverage.
</objective>

```typescript
// Load pending creates from any page (e.g., quick capture) into optimistic items
useEffect(() => {
  async function loadQuickCapturePending() {
    const pending = await getPendingCreates<Item>("items");
    if (pending.length === 0) return;
    setOptimisticItems(prev => {
      const existingIds = new Set(prev.map(i => i.id));
      const newPending = pending.filter(p => !existingIds.has(p.id));
      return newPending.length > 0 ? [...prev, ...newPending] : prev;
    });
  }
  loadQuickCapturePending();
}, []); // Mount-only: MUTATION_SYNCED subscription handles removal
```
