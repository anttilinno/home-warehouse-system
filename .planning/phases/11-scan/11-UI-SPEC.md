# Phase 11 — Scan — UI Design Contract

> Visual and interaction contract for the `/scan` camera surface, `/claim/:code`
> flow, and the SCAN-09/10 prefill affordances. **PARITY phase, existing design
> system.** Do NOT invent visual language: every surface reuses the Retro OS
> Pastel atoms in `frontend2/src/components/retro/` and the tokens in
> `frontend2/src/styles/tokens.css`. The ONE net-new layout territory is the
> live camera frame + the overlay chrome on top of it — designed below from the
> existing Retro OS tokens (no new color/type/spacing introduced).
>
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

> **Hard rule reminders (break = regression, enforced by `tokens.test.ts`):**
> 1. Silkscreen never below 16px, never mixed case, never body copy.
> 2. Pastel fills carry **ink text only**; colored text on white/cream uses
>    `--accent-*-deep` / `--warn-deep`.
> 3. `--fg-faint` is decorative/disabled only — never running text.
> 4. Radii 0 everywhere except badges (2px).
> 5. Every text pair ≥ 4.5:1 (AA).

## Spacing Scale

The locked 4px grid (`tokens.css`). All Phase 11 layout uses these utilities
(`p-sp-3`, `gap-sp-2`, …). No exceptions are introduced.

| Token | Value | Usage in Phase 11 |
|-------|-------|-------------------|
| sp-1 | 4px | Chip/icon gaps, badge insets, banner cursor offset |
| sp-2 | 8px | Button rows, overlay control gaps, history-row inner gap |
| sp-3 | 12px | Tab padding, overlay padding, banner padding, dialog body |
| sp-4 | 16px | Window body padding, form field stack, section gaps |
| sp-5 | 24px | Empty-state vertical padding, claim-form section breaks |
| sp-6 | 32px | (unused this phase — page-level only) |

**Exceptions:** Camera overlay controls use **min 44×44px touch targets** for the
torch toggle and the tab strip on touch devices (camera UX is one-handed mobile).
This is a touch-affordance floor layered ON the 4px grid (`min-h-[44px]`), not a
new spacing token. BevelButton default padding (`px-[14px] py-[6px]`) already
yields a comfortable target; the torch icon button must be explicitly padded to 44px.

## Color

Inherited from `tokens.css`. The camera frame introduces NO new color — it is a
black video surface (the camera's own pixels) framed by the existing 2px ink
bevel; overlay chrome sits on translucent ink scrims built from existing tokens.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--bg-desktop #fdf6ec` / `--bg-panel #ffffff` | Page desktop, Window body, tab panels |
| Secondary (30%) | `--bg-panel-2 #f7f0e2` | Inactive tab fill, overlay control strips, recessed banner row |
| Accent (10%) | pastel titlebars (`--titlebar-blue/mint/pink/butter`) | Window titlebar, active tab fill, banner state tint (semantic, see below) |
| Destructive | `--danger #b73348` on `--danger-bg #fbe3e8` | ERROR banner, CLEAR HISTORY confirm, claim-failure |

**Accent reserved for (explicit — never "all interactive"):**
- Active RetroTab fill (`bg-titlebar-blue bevel-raised-ink`) — atom-owned.
- Window titlebar — `blue` default for `/scan`, `mint` for `/claim/:code` (positive action).
- Banner state TINT (semantic, mapped 1:1 to the 4 states — see §Banner).
- The MATCH quick-action primary button (`variant="mint"` — positive/confirm).
- The decode-flash overlay (mint wash, see SCAN-03).

**Camera-frame scrim tokens (net-new layout, existing color only):**
- Overlay scrims over live video use `bg-fg-ink/55` … `bg-fg-ink/70` (the same
  `--fg-ink` already used for the RetroDialog scrim at `/40`). Text on the scrim
  is `--bg-panel #ffffff` (white-on-ink ≥ 4.5:1 — AA holds).
- The reticle/aim frame is a `2px solid` white (`--bevel-light`) corner-bracket
  set, NOT a full box (System-7 corner-mark idiom), so it never obscures the code.
- Torch-ON indicator uses `--titlebar-butter` chip (warning/active warmth) with ink text.

> **3-non-color-cue discipline (carried from LoansListPage §Overdue-row):**
> Every state that uses color ALSO carries (a) a distinct word in Silkscreen and
> (b) a distinct unicode glyph. Banner states, torch state, and the decode flash
> all satisfy three cues so the UI reads without color perception.

## Surface 1 — `/scan` page (single-mount camera shell)

**Route:** `/scan` (auth-gated, under AppShell). Sidebar `Scan` NavItem enabled
(`Sidebar.tsx:149`). **Titlebar:** `blue` (default/informational).

**Architecture contract (HARD parity fact — SCAN-01/02, CONTEXT OQ4):** the
`<BarcodeScanner>` (`@yudiel/react-qr-scanner`) mounts **exactly once** as a
sibling that fills the Window body and **stays mounted for the page lifetime**.
The Scan / Manual / History tabs and the result banner and the quick-action
overlay all render **ON TOP** of it as absolutely-positioned overlay layers —
they must NOT be RetroTabs children that would unmount the video element. Pause
is **prop-driven** (`paused={…}` / `scanning` prop), never unmount.

> **DEVIATION from the LoansListPage RetroTabs pattern (must be explicit for the
> planner):** LoansListPage swaps tab CONTENT inside `RetroTabs` (one panel
> mounts at a time). `/scan` CANNOT do that for the camera — the camera lives
> OUTSIDE the tab panel, persistent. RetroTabs here drives ONLY the overlay
> layer that is shown (the tab strip + the active overlay), while the video
> stays mounted underneath. Use RetroTabs for the **strip + ARIA/roving-tabindex
> + `?tab=` URL sync** (reuse the `setTab`/`useSearchParams` recipe verbatim from
> LoansListPage lines 80-89), but render its panels as positioned overlays, or
> use a custom tab strip that reuses RetroTabs' exact class tokens
> (`TAB_ACTIVE = "bg-titlebar-blue bevel-raised-ink"`, `TAB_BASE` caps). Planner
> picks; either way the camera node is never a descendant that re-mounts.

```
┌─ SCAN ─────────────────────────────────────[□][▣]┐  ← Window, titlebar blue
│ ┌───────────────────────────────────────────────┐ │
│ │ [ SCAN ][ MANUAL ][ HISTORY ]                  │ │  ← tab strip overlay (top),
│ │  ▔▔▔▔▔                                          │ │     RetroTabs caps; active=blue
│ │                                                 │ │
│ │        ┌─┐                       ┌─┐            │ │  ← reticle corner brackets
│ │                                                 │ │     (2px white), centered
│ │            · · · LIVE VIDEO · · ·               │ │     ~70% of frame width
│ │              (rear camera fills)                │ │
│ │        └─┘                       └─┘            │ │
│ │                                                 │ │
│ │                              ┌──────────┐       │ │  ← torch toggle (bottom-right),
│ │                              │ ⚡ TORCH  │       │ │     44×44 min; Android only,
│ │                              └──────────┘       │ │     auto-hidden on iOS (SCAN-04)
│ │  Point the camera at a barcode or QR code.      │ │  ← aim hint, white-on-ink scrim
│ └───────────────────────────────────────────────┘ │
└────────────────────────────────────────────────────┘
```

**Tab strip:** sits as a top overlay over the video on a `bg-fg-ink/55` strip so
the caps stay legible against any camera image. Active tab = `bg-titlebar-blue
bevel-raised-ink` (atom value); inactive caps use white text on the ink scrip
(NOT `bg-bg-panel-2`, which would vanish over video — this is the one place the
inactive-tab fill is overridden to a translucent ink strip for contrast).

**Reticle / aim frame:** four 2px white corner brackets framing a centered region
~70% of the frame width, 1:1-ish. No full box (would mask the code). Brackets are
`aria-hidden` decoration. Under `prefers-reduced-motion` the reticle is static
(it is static anyway — no pulse). Optional subtle non-motion treatment only.

**Torch toggle (SCAN-04):** a BevelButton-styled control, bottom-right, min
44×44. Rendered ONLY when `MediaStreamTrack.getCapabilities().torch === true`
(Android Chrome). On iOS the capability is absent → the control is **not
rendered at all** (no disabled ghost). Three-cue state:
- OFF: `⚡` glyph + cap `TORCH` + neutral bevel.
- ON: `⚡` glyph + cap `TORCH ON` + `--titlebar-butter` fill (ink text) + pressed bevel.

**Camera-permission-denied / no-camera state:** when the scanner reports a
permission/`NotAllowedError` or no device, the video area is replaced by a
`RetroEmptyState` over `bg-bg-panel-2` (NOT over the dead video):
- glyph `◎`, heading `CAMERA BLOCKED`, body (see Copywriting), action
  `SWITCH TO MANUAL` → activates the Manual tab. The Manual + History tabs
  remain fully usable when the camera is unavailable (graceful degrade).

## Surface 2 — Decode feedback (SCAN-03, visual only)

Audio (AudioContext beep) and haptic (`ios-haptics` / `navigator.vibrate`) are
non-visual and out of this contract. The VISUAL cue on a successful decode:

- A **full-frame mint wash flash** (`bg-titlebar-mint/70`) over the video for
  ~180ms, then a **center checkmark** `✓` in a 1px-ink white square (the
  RetroEmptyState thumb idiom) that holds ~400ms while the lookup fires.
- Three cues: color (mint) + glyph (`✓`) + the checkmark square shape.

**Motion contract (reuse the existing `steps()` motion language):** define the
flash as a `steps()` keyframe in `globals.css` alongside `status-blink` /
`retro-progress` (NOT an eased fade — System 7 has no material motion). Under
`prefers-reduced-motion: reduce`:
- **NO flash animation.** Instead, the checkmark square appears **instantly and
  holds** (static), and the banner (Surface 5) carries the state. This mirrors
  the locked `.status-dot--live` and `.retro-progress` reduced-motion fallbacks
  (`animation: none`, hold solid). The decode is still fully communicated by the
  banner + the static checkmark + the audio/haptic channels.

```
┌ [ SCAN ][ MANUAL ][ HISTORY ] ───────────┐
│  ▔▔▔▔▔▔                                    │
│  ENTER CODE                                │  ← RetroInput label (12px cap)

│  │ 0123456789012                        ▏│ │
│  └──────────────────────────────────────┘ │
```

- Manual tab does NOT pause the camera underneath unmounting; it stays mounted,
  paused via prop while Manual is the active overlay (so returning to Scan is
  instant on iOS).

## Surface 4 — History tab (SCAN-06/07)

`localStorage` key `hws-scan-history`, **last 10** (pick 10 per CONTEXT; roadmap
×10 wins over the ×20 parity note). Each entry: `{ code, ts }`. Row stores code +
relative time. Row tap re-fires the lookup through the shared state machine.

```
┌ [ SCAN ][ MANUAL ][ HISTORY ] ───────────────────┐
│              ▔▔▔▔▔▔▔                                │
│  RECENT SCANS                  [ ✕ CLEAR HISTORY ] │  ← 11px cap + danger BevelButton
│  ┌───────────────────────────────────────────────┐│
│  │ 0123456789012                      2 min ago ▸ ││  ← mono code | rel-time | ▸
│  │ 5901234123457                     14 min ago ▸ ││     row = full-width tap target
│  │ ABC-128-XYZ                          1 hr ago ▸ ││     hover = bg-info-bg (table idiom)
│  └───────────────────────────────────────────────┘│
└────────────────────────────────────────────────────┘
```

- Rows reuse the **RetroTable hover/stripe idiom** (`bg-info-bg` on hover,
  `--table-stripe` even rows) — but as a simple list of `<button>` rows (each row
  is a real button for a11y; the whole row is the re-fire affordance). Code in
  mono `tabular-nums`; relative time in 11–12px muted Plex Sans, right-aligned;
  trailing `▸` glyph (`aria-hidden`) signals "re-runs lookup".
- `CLEAR HISTORY` = `BevelButton variant="danger"` (glyph `✕`), disabled when
  history empty. Opens `RetroConfirmDialog` (pink, danger) — see Copywriting.
- **Empty state:** `RetroEmptyState` glyph `◇`, heading `NO SCANS YET`, body +
  no action (the empty CTA is "go scan", which is the adjacent Scan tab).

## Surface 5 — 4-state result banner (SCAN-08)

A single persistent banner region that sits as an overlay strip across the BOTTOM
of the camera frame (and is the primary result surface for Manual/History too).
Maps directly from `itemsApi.lookupByBarcode` (CONTEXT): `pending → LOADING`,
`item → MATCH`, `null → NOT-FOUND`, `throw → ERROR`. Query keys
`["item-by-barcode", wsId, code]` (CONTEXT constraint 4).

Each state carries **three cues**: a Silkscreen STATE word + a glyph + a semantic
pastel tint (ink text on the tint — AA holds). The scanned code is echoed in mono.

```
LOADING ───────────────────────────────────────────────
┌──────────────────────────────────────────────────────┐
│ [LOADING] ◌  0123456789012  Looking up code▏           │  ← blinking cursor ▏ (mono)
└──────────────────────────────────────────────────────┘   tint: bg-bg-panel-2 (neutral)

MATCH ─────────────────────────────────────────────────
┌──────────────────────────────────────────────────────┐
│ [MATCH] ✓  Cordless Drill                              │  ← item.name (body)
│ 0123456789012                          [ ▸ ACTIONS ]   │  ← code mono; opens overlay (S6)
└──────────────────────────────────────────────────────┘   tint: bg-titlebar-mint (positive)

NOT-FOUND ─────────────────────────────────────────────
┌──────────────────────────────────────────────────────┐
│ [NOT FOUND] ✕  0123456789012                           │
│ No item has this code.        [ ⊕ CREATE WITH CODE ]   │  → /items/new?barcode=<code>
└──────────────────────────────────────────────────────┘   tint: bg-titlebar-butter (attention)

ERROR ─────────────────────────────────────────────────
┌──────────────────────────────────────────────────────┐
│ [ERROR] ✕  Couldn't look up that code.                 │
│ 0123456789012                          [ ↻ TRY AGAIN ] │  → re-fire same code
└──────────────────────────────────────────────────────┘   tint: bg-danger-bg, text-danger word
```

| State | Glyph | Tint | State word color | CTA |
|-------|-------|------|------------------|-----|
| LOADING | `◌` | `bg-bg-panel-2` | ink | none (cursor animates) |
| MATCH | `✓` | `bg-titlebar-mint` | ink | `▸ ACTIONS` → quick-action overlay (S6) |
| NOT-FOUND | `✕` | `bg-titlebar-butter` | ink | `⊕ CREATE WITH CODE` → `/items/new?barcode=<code>` (SCAN-09; `encodeURIComponent`) |
| ERROR | `✕` | `bg-danger-bg` | `text-danger` (deep companion) | `↻ TRY AGAIN` (re-fire) |

The STATE word uses `StatusPill` where the variant maps:
`LOADING→info`, `MATCH→ok`, `NOT-FOUND→warn`, `ERROR→danger` (all ink-on-pastel
except danger, which the StatusPill atom already handles). The Silkscreen
one-word label sits beside the pill or the pill text IS the word — planner
picks; both satisfy the cue rule.

**Blinking-cursor (terminal) variant — `prefers-reduced-motion`-aware (SCAN-08):**
The LOADING line ends in a mono block cursor `▏` that blinks. **Reuse the exact
`status-blink` keyframe convention** (`@keyframes` with `steps(1, end)`,
`@media (prefers-reduced-motion: reduce){ animation: none }` → cursor holds
SOLID). Add a `.scan-cursor--blink` class in `globals.css` next to
`.status-dot--live` so there is one motion language. Under reduced motion the
cursor is a solid block and the word `LOADING` + the `◌` glyph carry the state.

### UPC opt-in suggestion banner (SCAN-10)

Triggered separately on the **item-create form** (NOT on `/scan`): when the
prefilled `?barcode=` matches `/^\d{8,14}$/`, fetch `GET /barcode/{code}`
(GLOBAL route, key `["barcode", wsId, code]`). On a found product, render a
suggestion banner ABOVE the form fields, inside the existing ItemFormPage Window:

```
┌──────────────────────────────────────────────────────┐
│ [INFO] PRODUCT FOUND  ·  via OpenFoodFacts            │  ← RetroBadge variant="info"
│ Bosch Cordless Drill — Bosch                          │  ← product name — brand (body)
│         [ USE NAME ]  [ USE ALL ]  [ DISMISS ]        │
└──────────────────────────────────────────────────────┘   tint: bg-info-bg
```

- `bg-info-bg`, `RetroBadge variant="info"` `PRODUCT FOUND`, plus a small muted
  source chip (`via OpenFoodFacts` / `OpenProductsDB`).
- **USE** = fill the Name field only. **USE ALL** = fill name + brand (+ image if
  the form ever supports it). **DISMISS** = hide the banner (no fetch retry).
  All three are `BevelButton`; USE ALL = `variant="primary"`, USE/DISMISS =
  `neutral`.
- Wiring: this banner is NEW in ItemFormPage; it does not exist today. The form
  already consumes `?barcode=` (ItemFormPage.tsx:80) — the banner adds the
  product fetch + setValue plumbing. Flag for planner: ItemFormPage is a
  single-writer concern for the plan that adds SCAN-10.

## Surface 6 — Quick-action overlay (SCAN-11, post-MATCH)

After a MATCH, tapping `▸ ACTIONS` (or auto, planner's call — recommend explicit
tap to avoid yanking focus mid-scan) opens a **state-adaptive** action overlay.
Render it as a `RetroDialog` (centered, blue titlebar `MATCHED ITEM`) so it
reuses the locked focus-trap + ESC-via-modal-stack + scrim recipe — and crucially
does NOT unmount the camera underneath.

```
┌─ MATCHED ITEM ───────────────────────────────[✕]┐
│  Cordless Drill                                   │  ← item.name (body, 16px-ish)
│  0123456789012                                    │  ← code, mono muted
│  ┌─────────────────────────────────────────────┐ │
│  │ [ ◎ VIEW ITEM ]        (always)              │ │  → /items/:id   primary
│  │ [ ⊕ LOAN ]             (hidden if active loan)│ │  → /loans/new?itemId=:id   mint
│  │ [ ↺ UNARCHIVE ]        (only if archived)    │ │  → itemsApi.restore   neutral
│  │ [ ✓ MARK REVIEWED ]    (only if needs_review)│ │  → review mutation   neutral
│  │ [ ◂ BACK TO SCAN ]     (always)              │ │  → close overlay, resume camera
│  └─────────────────────────────────────────────┘ │
└───────────────────────────────────────────────────┘
```

**State-adaptive rules:**

| Action | Visibility condition | Target | Variant |
|--------|---------------------|--------|---------|
| VIEW ITEM | always | `/items/:id` | `primary` |
| LOAN | **hidden if the item has an active loan** | `/loans/new?itemId=:id` | `mint` |
| UNARCHIVE | only if `item.is_archived` | `itemsApi.restore(wsId,id)` + toast | `neutral` |
| MARK REVIEWED | only if `needs_review` | review mutation + toast | `neutral` |
| BACK TO SCAN | always | close overlay, `paused=false` | `neutral` |

> **UNRESOLVED DATA-SHAPE FLAGS (planner/researcher must close before build):**
> The detail `Item` type (`lib/types.ts:113`) carries `is_archived` (→ UNARCHIVE
> works today) but **does NOT carry `active_loans` or `needs_review`.**
> - **Active-loan state:** derive via `loansApi.byItem(wsId, item.id)` →
>   `active.length > 0` (this partition already exists, `lib/api/loans.ts:30`).
>   The overlay must fire that query (key `["loans", wsId, "by-item", id]`) when
>   it opens; LOAN hides while it loads OR is non-empty (fail-safe: hide LOAN
>   until the active count is known to avoid offering a duplicate loan).
> - **`needs_review`:** there is NO `needs_review` field anywhere in frontend2
>   today (grep clean). MARK REVIEWED is a roadmap-prescribed action with no
>   backend/type support yet. **Recommendation:** scope MARK REVIEWED OUT of
>   Phase 11 unless 11-RESEARCH confirms a backend review flag + endpoint; if it
>   lands, gate the button on that field. Do not ship a dead button. (Flagged for
>   the gsd-planner to resolve as a research item, not a design item.)

## Surface 7 — `/claim/:code` (SCAN-12, claim-as-loan)

**Route:** `/claim/:code` — **login required** (gate behind `RequireAuth`; an
unauthenticated visitor is bounced to `/login?next=/claim/:code`). Resolves the
`:code` (shortlink short_code OR barcode) → an item, then presents a
claim-as-loan form. **Titlebar:** `mint` (positive action).

```
┌─ CLAIM ITEM ────────────────────────────────[□][▣]┐  ← Window, titlebar mint
│  ┌───────────────────────────────────────────────┐ │
│  │  Cordless Drill                                 │ │  ← resolved item name (body 16px)
│  │  CODE  0123456789012                            │ │  ← code, mono muted
│  │  ───────────────────────────────────────────── │ │
│  │  BORROWER         (you, prefilled)              │ │  ← RetroInput / read-only
│  │  DUE DATE         [ 2026-06-20            ]     │ │  ← date RetroInput
│  │  NOTE             [ ............................]│ │  ← RetroTextarea (optional)
│  │                                                 │ │
│  │              [ Cancel ]  [ ⊕ CLAIM AS LOAN ]    │ │  ← neutral + mint primary
│  └───────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────┘
```

- Reuse the LoanFormPage form chrome (RHF + zod + RetroInput/RetroTextarea) and
  POST through `loansApi.create` against the resolved item's inventory entry —
  same contract as `/loans/new?itemId=` (LoanFormPage already auto-selects when
  one inventory entry matches an item, `LoanFormPage.tsx:87`).
- **Three load states:** RESOLVING (mono "Resolving code…" line, reuse
  `.retro-progress`), RESOLVED (the form above), UNRESOLVABLE (`RetroEmptyState`
  glyph `◎`, heading `CODE NOT FOUND`, body + action `BACK TO SCAN` → `/scan`).
- **Success:** `retroToast.success` + navigate to the created loan / item detail.
- **Failure:** persistent `retroToast.error` (the atom's Infinity-duration
  default) + in-form `role="alert"` danger banner (the ItemFormPage `root` error
  idiom, lines 319-327).

> **UNRESOLVED ENDPOINT FLAG (research, not design):** the exact shortlink-resolve
> endpoint for `/claim/:code` (short_code → item/inventory) is not mapped in
> frontend2 yet (CONTEXT OQ6). The VISUAL contract above is endpoint-agnostic;

### Genuinely UNRESOLVED (flagged — needs a decision, mostly research not design)

1. **Camera-frame chrome treatment under Retro OS (DESIGN — proposed).** Retro OS
   is cream/white/pastel; a live camera is a black rectangle, alien to the
   palette. **Proposal (recommend adopting):** treat the video as the Window
   BODY content with the standard 2px ink inner border, set the video container
   `bg-fg-ink` (so letterboxing reads as intentional ink, not a gap), and place
   ALL overlay chrome (tabs, banner, torch, hint) on translucent
   `bg-fg-ink/55–70` strips with white text — i.e. the camera is "a photo in a
   System-7 picture frame." No phosphor/scanline reskin (that was the scrapped
   Premium Terminal direction — do NOT revive it). This keeps the camera honest
   while the chrome stays Retro OS. **Decision owner: gsd-ui-checker / user sign-off.**

2. **`MARK REVIEWED` has no backend/type support** (`needs_review` grep-clean).
   Recommend scoping OUT of Phase 11 unless 11-RESEARCH confirms a review flag +
   endpoint. RESEARCH item, not design.

3. **Shortlink-resolve endpoint for `/claim/:code`** not yet mapped in frontend2
