<objective>
The notifications feature (NOTIF-01/02/03): a USER-scoped api + query/mutation hooks, a
bell button with an unread-count badge, and a dropdown listing notifications with per-item
mark-read + mark-all-read. This plan OWNS the TopBar.tsx single-writer edit that swaps the
reserved disabled bell-slot placeholder for the real bell.

Notifications are USER-scoped, NOT workspace-scoped: every endpoint is `/api/notifications/...`
with NO `{ws}` segment, and the query key is `["notifications"]` (no wsId). The unread badge
polls `GET /api/notifications/unread/count` on a `refetchInterval` — there is NO `notification`
entry in the SSE invalidation map (VALIDATION confirmed), so SSE invalidation is an optional
follow-up, not this plan's job.

Verified backend DTO (read this session from `auth/notification/handler.go`):
- `GET /api/notifications?page=&limit=` → `{ items: NotificationResponse[], total, page, total_pages }`
- `GET /api/notifications/unread` → same envelope
- `GET /api/notifications/unread/count` → `{ count: number }`  (JSON tag `count`, int64)
- `POST /api/notifications/{id}/read`  → 204 (no body)
- `POST /api/notifications/read-all`   → 204 (no body)   ← NOTE: `read-all`, NOT `/read-all` under an id
- `NotificationResponse` JSON tags: `id`, `user_id`, `workspace_id?`, `notification_type`
  (enum LOAN_DUE_SOON|LOAN_OVERDUE|LOAN_RETURNED|LOW_STOCK|WORKSPACE_INVITE|MEMBER_JOINED|SYSTEM),
  `title`, `message`, `is_read`, `read_at?`, `metadata?`, `created_at`.

Purpose: NOTIF-01/02/03 — the TopBar gains a working bell, dropdown, mark-read, and unread badge.
