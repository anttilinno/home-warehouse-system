## Backend contract (VERIFIED from source 2026-06-13)

Recorded here because the design depends on these exact shapes. Source:
`backend/internal/domain/warehouse/inventory/{entity,handler,repository}.go` +
`movement/{entity,handler}.go`.

**Entry wire shape (`InventoryResponse`):** `id, workspace_id, item_id,
location_id, container_id?, quantity, condition, status, date_acquired?,
purchase_price? (cents), currency_code?, warranty_expires?, expiration_date?,
notes?, is_archived, created_at, updated_at`. **It carries item/location/
container *IDs only* вЂ” no display names.** Name resolution comes from the Phase 6
read endpoints (items / locations / containers lists), client-joined.

**Condition enum (7, fixed):** `NEW В· EXCELLENT В· GOOD В· FAIR В· POOR В· DAMAGED В·
FOR_REPAIR`.
**Status enum (7, fixed):** `AVAILABLE В· IN_USE В· RESERVED В· ON_LOAN В· IN_TRANSIT
В· DISPOSED В· MISSING`.

**Endpoints used:** `GET /inventory` (page/limit only вЂ” **no server-side
search/facet filters**, limit default 50 max 100), `GET /inventory/{id}`,
`GET /inventory/by-item/{item_id}` (INV-08 panel), `GET /inventory/expiring?days=`
(default 30, max 365), `POST /inventory`, `PATCH /inventory/{id}` (full update),
`PATCH /inventory/{id}/status`, `PATCH /inventory/{id}/quantity`,
`POST /inventory/{id}/move`, `POST /inventory/{id}/archive|restore`. Movements
(read-only): `GET /movements`, `GET /inventory/{inventory_id}/movements`,
`GET /locations/{location_id}/movements`.

**Three resolutions forced by the backend (autonomous):**

1. **`GET /inventory` has NO filter params** (only `page`/`limit`). Unlike the
   items list (which has `search`/`category_id`/`archived`), **inventory filters
   are client-side over the loaded page set** (status/condition/location facets
   filter what's already fetched). The list is **pagination-first** (see
   virtualization decision). Search facets read as live client filters, not URL
   round-trips to the server.
2. **Move is whole-entry only.** `MoveInventoryInput.Body` = `{location_id,
   container_id?}` вЂ” **there is no `quantity` field**. The backend Move method
   relocates the entire entry; the movement RECORD logs the full entry quantity.
   **в†’ Split/partial move is NOT supported this phase.** The move dialog ships
   the single whole-entry variant; the partial-move UX variant is flagged
   below as deferred (needs a backend split endpoint).
3. **Movements are read-only with ID-only fields.** `MovementResponse` =
   `{id, inventory_id, from_location_id?, from_container_id?, to_location_id?,
   to_container_id?, quantity, moved_by? (uuid), reason?, created_at}`. **No
   name/user-name on the read model** вЂ” `moved_by` is a bare UUID (the SSE event
   carries `user_name`, the REST read model does not). The "who" column resolves
   against the workspace member list if available, else shows a muted fallback.
   Movements have **no SSE emit** вЂ” they are a side effect of a move; invalidate
   `["movements", wsId]` manually on a successful move mutation.

pagination affordance (RetroPagination) is the scroll strategy.** Revisit

## 2 В· Create-Entry Form (`/inventory/new`)

A **blue-titled `Window`** (`ADD ENTRY`), centered page panel
(`max-w-[560px] mx-auto`) under AppShell вЂ” RHF + zod, optimistic invalidation of
`["inventory", wsId, ...]` (CONTEXT). **Resolved (autonomous): single-window
form, not a wizard** (matches the Phase 7 item form). **Edit reuses the same
form** at `/inventory/{id}/edit` (`EDIT ENTRY`, title only differs) backed by
`PATCH /inventory/{id}` вЂ” note that `PATCH /inventory/{id}` does NOT accept
`status` (status changes only via the dedicated status endpoint / inline edit), so
**the edit form omits the Status field**; the create form includes it (the create
body requires it).

### Field grouping (top в†’ bottom, `RetroFormField`-wrapped)

1. **What & where**: Item (`RetroSelect`, required `*`, populated from `["items", wsId]`) В· Location (`RetroSelect`, required `*`, from `["locations", wsId]`) В· Container (`RetroSelect`, optional, from `["containers", wsId]` вЂ” empty option `вЂ” No container`).
2. **Stock**: Quantity (`RetroInput type="number"`, mono, required, `min=1`) В· Condition (`RetroSelect`, required, 7 Title-Case options) В· Status (`RetroSelect`, required, 7 Title-Case options вЂ” **create only**).
3. **Dates** (optional): Acquired (`RetroInput type="date"`, mono) В· Warranty expires (`RetroInput type="date"`, mono) В· Expiry date (`RetroInput type="date"`, mono). Each carries a 12px `text-fg-muted` hint `Optional`.
4. **Notes**: Notes (`RetroTextarea`, `min-h-[88px]`, optional).

**Resolved (autonomous): purchase price + currency are NOT in the form this
phase.** The backend accepts `purchase_price`/`currency_code`, but value-tracking
is a discrete concern (no Phase 7b requirement, no sketch). Defer to a later
value/valuation phase; the fields stay absent (not zero-injected вЂ” Pitfall 4/7).

**Pickers are native `RetroSelect`** (CONTEXT locked): no type-ahead, no
free-typing. If a select's source list is empty (e.g. no locations yet), the
field is disabled with a hint `No locations yet вЂ” add one first.` linking to the
relevant management surface.

### Validation error treatment (sketch 007 in-window style вЂ” identical to Phase 7)

- Per-field: `RetroFormField` error line вЂ” `вњ•`-prefixed `text-[12px] font-semibold text-danger`, `aria-invalid` + `aria-describedby`; field flips to `border-danger bg-danger-bg`. Required marker is **ink `*`**.
- Cross-field: if Expiry date is before Acquired date в†’ field error on Expiry `Expiry can't be before the acquired date.` (client-side zod refinement).
- Form-level (submit/server failure): a `bg-danger-bg border-2 border-border-ink` banner at the top of the body вЂ” `вњ• Couldn't save this entry. {reason}` (14px `text-danger`) вЂ” plus a persistent `retroToast.error`.

### Submit / cancel placement

- Footer row pinned bottom: `border-t-2 border-border-ink pt-sp-3`, right-aligned, `gap-sp-2`: **`CANCEL`** (`BevelButton` neutral в†’ back) + **`SAVE ENTRY`** (create) / **`SAVE CHANGES`** (edit) (`BevelButton variant="primary"`).

- On success: `retroToast.success` `DONE` `Entry saved.` + invalidate + navigate to the owning item detail (`/items/{item_id}`) so the new entry is visible in the INV-08 panel.

editable cell в†’ it swaps in place to an input/select в†’ commit on blur/Enter,
cancel on ESC. Optimistic with revert-on-error (CONTEXT locked вЂ” the Phase 7
photo-reorder pattern). Only **Qty / Status / Condition** are inline-editable
(they map to dedicated endpoints); all other fields edit via the form.

### Affordance + lifecycle

| State | Treatment |
|-------|-----------|
| Rest | The cell value renders normally (qty = mono; status/condition = StatusPill). On row hover, an editable cell shows a faint 1px `border-table-rule` inset + `cursor-text` (qty) / `cursor-pointer` (selects) hinting it's editable. |
| Editing | Click (or Enter/Space when the cell is keyboard-focused) swaps the display for an inline control: **Qty в†’** `RetroInput type="number"` (`min-w-[64px]`, mono, `min=0`, auto-selects contents); **Status в†’** `RetroSelect` (7 options); **Condition в†’** `RetroSelect` (7 options). The control carries the locked 3px `outline-titlebar-blue` focus ring (the sole edit-mode color). |
| Commit | **Blur OR Enter** commits. Qty в†’ `PATCH /inventory/{id}/quantity` `{quantity}`; Status в†’ `PATCH /inventory/{id}/status` `{status}`; Condition в†’ `PATCH /inventory/{id}` (full-entry patch with the changed condition + unchanged fields вЂ” there is no condition-only endpoint). Optimistic: the cell shows the new value immediately. |
| Cancel | **ESC** (routed through `useModalStack`? NO вЂ” an inline cell is not a modal surface; ESC here is a local field handler on the input, scoped to the editing cell, and does not pop the modal stack). Reverts to the committed value, no request. |
| Success | Cell returns to rest with the new value. No toast on inline success (it would be noisy at density) вЂ” the optimistic value IS the confirmation. |
| Error | Optimistic value **reverts** to the prior value; a `retroToast.error` fires `Couldn't update {field}. {reason}` (`.error` persists). The cell re-renders at rest with the old value (no stuck edit state). |
| Validation | Qty: empty or `<0` rejected inline (the cell shows `border-danger` + a `вњ•` micro-marker and does NOT commit; blur with an invalid value cancels back to the prior value rather than firing a doomed request). |

### Accessibility

- Editable cells are keyboard-reachable (`tabIndex`, `role` appropriate); Enter/Space enters edit; ESC cancels; Tab from an editing control commits-and-moves. `aria-label` announces the field + item (`Edit quantity for {item name}`). Focus returns to the cell on commit/cancel. Never `outline:none` вЂ” the edit control uses the locked input ring; the rest-state cell uses the 2px ink focus ring.
- Status/condition selects announce their options; the optimistic change is announced via the toast on error only (success is silent вЂ” the visible value is the SR-detectable change).
