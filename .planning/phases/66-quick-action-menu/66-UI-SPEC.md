# Phase 66 — UI Design Contract

> Visual and interaction contract for the Quick-Action Menu that replaces
> `ScanResultBanner` as the post-decode surface on `/scan`. Generated by
> gsd-ui-researcher, verified by gsd-ui-checker.

**Pre-populated from:** `66-CONTEXT.md` (D-01..D-23 LOCKED), Phase 65 prior art
(`ScanResultBanner.tsx`), `frontend2/src/index.css` `@theme` tokens, existing
retro primitives (`RetroDialog`, `RetroButton`, `HazardStripe`).

**What the user was NOT asked:** nothing. CONTEXT.md and Phase 65 banner cover
every field. The "Claude's Discretion" list at the bottom of CONTEXT.md is
resolved inline below.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (fully custom retro library — v2.0 decision: "shadcn/ui fights retro aesthetic") |
| Preset | not applicable |
| Component library | custom retro primitives at `frontend2/src/components/retro/`, imported via the `@/components/retro` barrel (Phase 54 rule) |
| Dialog primitive | `RetroDialog` (native HTML `<dialog>` + `showModal()`) — provides `::backdrop`, focus-trap, ESC dismissal, built-in `[X]` close button |
| Icon library | none (monospace ASCII / unicode glyphs: `▍` blinking cursor, `X` close button) |
| Font (mono) | `IBM Plex Mono`, `JetBrains Mono`, `ui-monospace` fallback — used for headings, codes, all action-button labels |
| Font (sans) | `system-ui, -apple-system, sans-serif` — used only for prose body paragraphs (not-found hint, error body) |

**Dialog lifecycle (D-01, D-02, D-20):** `{banner && <QuickActionMenu … />}`
inside `ScanPage.tsx`. `QuickActionMenu` holds a `useRef<RetroDialogHandle>`
and calls `dialogRef.current?.open()` in a mount effect + `.close()` in the
cleanup. Mount = open; unmount = close. No `isOpen` state.

**Scanner invariant (D-04, D-21):** scanner subtree stays mounted underneath;
`paused = banner !== null` preserved verbatim from Phase 64 D-02. Native
`::backdrop` dims the viewfinder ~50% (baked into `RetroDialog` via
`backdrop:bg-black/50`). No "hide viewfinder" logic — keeps spatial context
and avoids iOS PWA permission reset (Pitfall #1).

**Dismiss parity (D-03):** ESC keypress, backdrop click, the built-in `[X]`
close button inside `RetroDialog`, and the in-dialog `BACK TO SCAN` neutral
button all have identical semantics — each calls `onScanAgain()` which runs
`setBanner(null)`. Clearing the banner unpauses the scanner AND unmounts the
dialog through the conditional render. Single dismissal code path.

**Animation (CONTEXT.md Discretion #4):** `RetroDialog` ships without an
entrance animation — leave as-is. The LOOKING UP heading in the LOADING state
uses the Phase 65 `.retro-cursor-blink` class (`index.css` lines 61–75) which
already carries a `@media (prefers-reduced-motion: reduce) { animation: none;
opacity: 1; }` guard — reuse verbatim.

## Spacing Scale

Declared values (Tailwind v4 `@theme` tokens in `frontend2/src/index.css`; all
multiples of 4):

| Token | Value | Usage in Quick-Action Menu |
|-------|-------|----------------------------|
| xs    | 4px   | Icon/cursor inline gap (`▍` blink glyph `ml-xs`) |
| sm    | 8px   | Dialog-header inner padding (`top-sm right-sm` of close `[X]`); hazard-stripe → heading breathing room (`mb-sm`); action-row vertical gap (`gap-sm` between the code metadata rows inside MATCH) |

## Typography

All four dialog states share the same typographic contract. Sizes (3 roles,
2 weights) pulled from Phase 65 `ScanResultBanner` to preserve muscle memory:

| Role | Size | Weight | Line Height | Font | Usage |
|------|------|--------|-------------|------|-------|
| Heading | 20px | bold (700) | 1.2 | mono | State label: `MATCHED`, `NOT FOUND`, `LOOKUP FAILED`, `LOOKING UP…` — inside `<h2 className="text-[20px] font-bold uppercase text-retro-ink mb-sm">` |
| Display | 24px | bold (700) | 1.2 | mono | Matched-item `name` and `short_code`; the echoed code in LOADING / NOT-FOUND / ERROR states — `font-mono font-bold text-[24px] text-retro-ink break-all` |
| Label | 14px | bold (700) | 1.2 | mono | Metadata row labels (`NAME`, `CODE`, `FORMAT`); button text (`RetroButton` base is `text-[14px] font-bold uppercase`); meta paragraph sub-labels |
| Body | 14px | regular (400) | 1.5 | sans | NOT-FOUND hint (`No item in this workspace matches this barcode.`) and ERROR remediation body (`Could not reach the server. …`) — `font-sans text-[14px] text-retro-ink`. Body prose is the ONLY sans-serif text in the dialog; everything else is monospace for retro terminal consistency. |

Weights — exactly **2**: regular (400) for body prose; bold (700) for all
mono text (headings, display, labels, buttons). Phase 65 banner uses the
same two-weight contract; Phase 66 preserves it.

## Color

The 60/30/10 allocation below is the contract for the dialog surface only.
Scanner viewfinder colors (underneath the backdrop) are Phase 64's concern.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--color-retro-cream` (`#F5F0E1`) | Dialog body surface (`bg-retro-cream` inherited from `RetroDialog`); neutral `RetroButton` fill (`BACK TO SCAN`, `LOAN`, `MARK REVIEWED`, `UNARCHIVE` when present) |
| Secondary (30%) | `--color-retro-ink` (`#1A1A1A`) | All text and borders (`text-retro-ink`, `border-retro-ink`); dialog border via `RetroDialog`'s `border-retro-extra-thick border-retro-ink`; button outlines via `RetroButton`'s `border-retro-thick border-retro-ink`; `::backdrop` scrim at 50% opacity (`backdrop:bg-black/50`) dimming the paused viewfinder |
| Accent (10%) | `--color-retro-amber` (`#D4A017`) | Primary-variant `RetroButton` fill — reserved ONLY for the one **lead action per state** (see D-13): `VIEW ITEM` in MATCH, `CREATE ITEM WITH THIS BARCODE` in NOT-FOUND, `RETRY` in ERROR, and (for the NOT-FOUND/ERROR bg stripe overlap) hazard yellow `#FFD700` via `bg-hazard-stripe` on the `HazardStripe variant="yellow"` header |
| Destructive | `--color-retro-red` (`#CC3333`) | ERROR state's top `HazardStripe variant="red"` header (solid red strip); the built-in `[X]` close button's fill (inherited from `RetroDialog`); NOT used for any action button in Phase 66 (Mark Reviewed and Unarchive are benign/reversible per D-14, so neither gets the danger variant) |

**Accent reserved for (EXPLICIT LIST — no drift):**
1. The one primary-variant `RetroButton` per state (the lead CTA)
2. The `bg-hazard-stripe` diagonal yellow+black pattern in the NOT-FOUND state header (via `<HazardStripe variant="yellow" />`)
3. The format pill (MATCH state, inherited from Phase 65) — `bg-retro-amber text-retro-ink px-sm py-xs`

Nothing else may take the amber fill. Neutral variant is the default for
every non-primary action (LOAN, MARK REVIEWED, UNARCHIVE, BACK TO SCAN —
these all render with `variant="neutral"` which uses `bg-retro-cream
text-retro-ink hover:bg-retro-amber` — hover crosses into amber but the idle
color stays cream).

**Second semantic (red)** reserved for:
1. `HazardStripe variant="red"` at the top of the ERROR state
2. The `[X]` close button fill (inherited from `RetroDialog`; consistent across every retro dialog in the app)

There are no destructive confirmation dialogs in Phase 66 (D-14 explicitly
rejects confirm-first UX for Mark Reviewed / Unarchive — both are
reversible).

### Empty State

Not applicable as a standalone screen — the dialog IS the post-scan surface.
The closest equivalent is the LOADING state (empty of data while awaiting
resolution); its copy is already specified above.

### Error State

body paragraph + primary `RETRY` + fallback `CREATE ITEM WITH THIS BARCODE`

**None.** Per D-14, Mark Reviewed and Unarchive are fire-and-forget —
reversible via the item detail page if the user changes their mind. Adding
a confirm layer contradicts the fast-rescan intent of `/scan`.

## Action Matrix (from CONTEXT.md D-16, reproduced for downstream consumers)

Action set in MATCH state is derived from three booleans on the matched
`Item` + the `useLoansForItem(item.id)` probe. Buttons render as a
**vertical stack of full-width `RetroButton`s** (D-12) in the order shown.

| `is_archived` | `activeLoan` | `needs_review` | Rendered buttons (top-to-bottom) |
|:---:|:---:|:---:|---|
| false | null | false | VIEW ITEM (primary) · LOAN (neutral) · BACK TO SCAN (neutral) |
| false | null | true  | VIEW ITEM (primary) · LOAN (neutral) · MARK REVIEWED (neutral) · BACK TO SCAN (neutral) |
| false | non-null | false | VIEW ITEM (primary) · BACK TO SCAN (neutral) |
| false | non-null | true  | VIEW ITEM (primary) · MARK REVIEWED (neutral) · BACK TO SCAN (neutral) |
| true  | null | — | VIEW ITEM (primary) · UNARCHIVE (neutral) · BACK TO SCAN (neutral) |
| true  | non-null | — | VIEW ITEM (primary) · UNARCHIVE (neutral) · BACK TO SCAN (neutral) |

**Loading-state rendering of the LOAN slot (D-10):** while
`useLoansForItem(item.id)` is in flight (not yet resolved), the slot
renders as a `RetroButton variant="neutral"` with `disabled` set,
`aria-busy="true"`, and the muted label `LOAN` (same text, grayed-out
fill from `disabled:bg-retro-gray disabled:text-retro-cream`). The button
is **not a spinner** — it's a disabled skeleton that morphs into the real
LOAN / UNARCHIVE / (hidden) state on resolve, without flicker.

**Discretion resolved (CONTEXT.md #5):** use the standard disabled
`RetroButton` + `aria-busy="true"` attribute. No bespoke skeleton styling —
`RetroButton`'s existing `disabled` classes already produce the correct
muted appearance, and screen-readers understand `aria-busy` for transient
loading affordances.

### State-Matrix Notes

- The matrix is evaluated ONLY in MATCH state. LOADING / NOT-FOUND / ERROR
  use their own layouts (see Copywriting Contract — Primary CTA table).
- `useLoansForItem` runs ONLY after `lookup.status === "success" &&
  lookup.match` (D-17 gate). Don't burn a request on NOT-FOUND / ERROR /
  LOADING.
- Archived items hide MARK REVIEWED regardless of `needs_review`
  (rationale in D-16: archived items aren't in active rotation; the
  review flag is stale).
- `needs_review` is **additive** (D-15) — order VIEW ITEM → LOAN → MARK
  REVIEWED → BACK TO SCAN when all conditions apply. Not a pre-loan gate.

## Interaction Contract

### Dialog Open / Close

| Trigger | Effect |
|---------|--------|
| Decode resolves / Manual submit / History tap | `handleDecode` → `setBanner({code, format, timestamp})` → `{banner && <QuickActionMenu …/>}` mounts → mount `useEffect` calls `dialogRef.current?.open()` (`showModal()`) |
| `ESC` keypress inside dialog | Native `<dialog>` `cancel` event → the `onClose` callback → `onScanAgain()` → `setBanner(null)` → dialog unmounts |
| Backdrop click | Native `<dialog>` emits `close` event → `onClose` callback → `onScanAgain()` → `setBanner(null)` |
| `[X]` close button (built into `RetroDialog`) | `RetroDialog` internally calls `dialogRef.current?.close()` → fires `close` event → same path as backdrop |
| `BACK TO SCAN` in-dialog neutral button | `onScanAgain()` directly (same callback) |
| `VIEW ITEM`, `LOAN`, `CREATE ITEM WITH THIS BARCODE` | `setBanner(null)` FIRST (D-11) → then `navigate(...)` — prevents a stale dialog waiting to re-render on browser-back |
| `MARK REVIEWED` / `UNARCHIVE` | Fire `useMarkReviewedItem.mutate(match.id)` / `useRestoreItem.mutate(match.id)` — dialog STAYS OPEN (D-11). On mutation success, `itemKeys.all` invalidation refetches `match`; the affected button simply isn't rendered on the next render pass. User may continue tapping other actions or dismiss. |

### Focus Management

`RetroDialog` uses native `<dialog>`+`showModal()` which hands the browser
the focus-trap responsibility. First focusable element on open defaults to
the `[X]` close button (it's first in the DOM). Downstream planner can
choose to `autoFocus` the primary button instead — **recommended default**:
let browser handle it; do not add `autoFocus` on the primary button
because focus trapping the user on the primary action can feel aggressive
for a terminal-retro aesthetic. Escape still wins.

### Keyboard

| Key | Effect |
|-----|--------|
| `ESC` | Dismiss (same as backdrop click / BACK TO SCAN) |
| `Tab` / `Shift+Tab` | Browser-native focus cycling inside the `<dialog>` focus trap |
| `Enter` / `Space` on a focused button | Fires the button's click handler |

No custom keybinds in Phase 66.

## Delta from Phase 65 Banner (for auditor)

Anything not listed here is byte-identical to Phase 65 `ScanResultBanner`'s
visual contract.

- **Container:** `RetroPanel` (inline, below tabs) → `RetroDialog` (modal
  overlay). `::backdrop` dim is new.
- **Layout of actions:** `flex flex-wrap gap-md justify-end` horizontal
  → `flex flex-col gap-md` vertical full-width (D-12). No more "buttons
  jump rows at narrow widths" — they're always stacked.
- **Primary-button count per state:** Phase 65 showed up to THREE primary
  amber buttons in the ERROR state (`RETRY` + `CREATE ITEM WITH THIS
  BARCODE` + `SCAN AGAIN`). Phase 66 demotes `CREATE ITEM WITH THIS
  BARCODE` and `BACK TO SCAN` to neutral — **one primary per state** is
  the Phase 66 invariant (D-13). 10% accent stays 10%.
- **Dismiss button label:** `SCAN AGAIN` → `BACK TO SCAN`. Reflects the
  new modal-above-scanner lifecycle vs the old banner-below-scanner
  lifecycle. New msgid; do not reuse `SCAN AGAIN`.
- **New actions:** `LOAN`, `MARK REVIEWED`, `UNARCHIVE` in MATCH state
  (state-adaptive per D-16).
- **Deleted surface:** `SCAN AGAIN` as a click target is no longer needed
  inside the dialog (replaced by `BACK TO SCAN`) — however, the SCAN
  AGAIN msgid may still be referenced elsewhere. Do NOT remove the msgid
  from the catalog; only stop referencing it in the QuickActionMenu
  component.
