# Phase 5 — UI Design Contract

> Visual and interaction contract for the **Auth surface**: login completion
> (email/password + Google/GitHub OAuth + env-gated Authelia SSO), register
> flow, `/auth/callback` exchange page, the TopBar **workspace switcher**, and
> the **Settings skeleton** (Security page — sessions, password change, account
> deletion — plus the Connected Accounts page). Generated by gsd-ui-researcher
> (autonomous run), verified by gsd-ui-checker.

> **This spec composes — it does not invent.** Every surface is built from the
> **Phase 4 atom library** (`@/components/retro/*`: RetroDialog, RetroConfirmDialog,
> Popover, RetroFormField, RetroInput, RetroTextarea, RetroSelect, RetroCheckbox,
> StatusPill, RetroEmptyState, RetroTable, RetroTabs, BevelButton, RetroBadge,
> Window, retroToast/RetroToaster) over the **Phase 3 chrome** (AppShell, TopBar
> reserved slots, useModalStack ESC arbiter, PageHeader) and **Phase 2 tokens**
> (`tokens.css`). **No NEW atom is introduced.** Where Phase 5 needs a behavior
> not yet shipped (a `/login` error taxonomy, a callback loading idiom), it is
> specified as a *composition* of existing atoms + a single justified chrome
> dimension, recorded as **Resolved (autonomous)**.

> **Direction:** Retro OS Pastel (sketches 006–008, BINDING per sketch-findings
> skill + MANIFEST). The login chrome is **validated sketch 007** (centered
> Window on the cream desktop, sunken inputs, in-window error band, primary →
> divider → OAuth hierarchy). This spec extends the **shipped `LoginPage.tsx`**
> (`frontend2/src/features/auth/LoginPage.tsx`) — it does not fork it.

**Backend contract (verified against source — trust code over docs):**

| Surface | Endpoint(s) | Notes |
|---------|-------------|-------|
| Register | `POST /auth/register` `{ email, full_name, password (min 8) }` → `{ token, refresh_token }` + cookies | min-8 password enforced server-side; mirror in zod. |
| Login | `POST /auth/login` `{ email, password }` | shipped. |
| OAuth initiate | `GET /auth/oauth/{provider}` (browser nav, `google` / `github`) | sets `oauth_state` cookie, 302 to provider. |
| OAuth callback | provider → backend `/auth/oauth/{provider}/callback` → 302 to `{AppURL}/auth/callback?code=…` **or** `?error=…` | frontend route consumes the query. |
| OAuth exchange | `POST /auth/oauth/exchange` `{ code }` → `{ access_token, refresh_token }` + cookies | one-time code (Redis); single use. |
| OAuth error codes (query `?error=`) | `provider_unavailable`, `invalid_state`, `authorization_cancelled`, `email_not_verified`, `server_error` | **exact** taxonomy — see Error Copy table. |
| Authelia SSO | `GET /auth/authelia/login` at the **BARE ingress path** (NOT through `/api` proxy — commit 8e13faf) | env-gated button. |
| Sessions | `GET /users/me/sessions` · `DELETE /users/me/sessions/{id}` · `DELETE /users/me/sessions` (all others) | session fields: device info (parsed UA), IP, `last_active_at`, `created_at`. |
| Password | `PATCH /users/me/password` `{ current_password?, new_password (min 8) }` | `current_password` optional → the OAuth-only "set password" path; surface via `has_password`. |
| Account delete | `GET /users/me/can-delete` → `{ can_delete, blocking_workspaces[] }` · `DELETE /users/me` `{ confirmation: "DELETE" }` (case-insensitive; 409 if sole owner) | type-DELETE gate. |
| Connected accounts | `GET /auth/oauth/accounts` → `{ accounts[] }` · `DELETE /auth/oauth/accounts/{provider}` | unlink 409 = `ErrCannotUnlinkLastAuth` ("cannot unlink sole authentication method when no password is set") → lockout guard. |
| Workspaces | `GET /users/me/workspaces` | D-12: `WorkspaceProvider` context + localStorage; first-workspace fallback. |
| Identity | `GET /users/me` → includes `has_password` (drives password-vs-set-password + unlink lockout messaging) | |

validated buttons read "Log in" / "Sign in with Google" (the visual uppercase in
the v3.0 login comes from CSS `text-transform`, per CLAUDE.md auth-contract note).
DOM text is sentence-case body copy (Plex Sans), so it is NOT the Silkscreen
display face and does not violate the uppercase rule — display uppercase is
reserved for titlebars/brand. OAuth/SSO/submit button DOM text is sentence-case.

## Surface Specs

### 1. Login page additions (extends shipped `LoginPage.tsx`, sketch 007)

The shipped login Window stays the spine: centered `Window` (blue titlebar
`LOG IN`) on `bg-bg-desktop` with the faint `WAREHOUSE.SYS` watermark, brandline,
in-window error band, RetroInput email/password, primary `BevelButton`. Phase 5
**adds** below the primary submit, in the sketch-007 `.actions` stack order:

- **Divider:** an `OR` row — `flex items-center gap-sp-2`, 11px UPPERCASE
  `tracking-[0.14em] text-fg-muted`, dotted `border-fg-faint` rules either side
  (verbatim sketch 007 `.divider`).
- **OAuth button group** (hierarchy = secondary, below the divider — validated
  sketch 007 "primary vs OAuth hierarchy clear"): a `grid gap-sp-2` of
  **full-width neutral `BevelButton`s** (NOT primary — primary is reserved for
  the email/password submit; OAuth is the alternative path):
  - `Sign in with Google` — neutral BevelButton, leading 18px Google brand tile, `→ window.location = "/api/auth/oauth/google"` (browser nav, not fetch — OAuth needs a full redirect).
  - `Sign in with GitHub` — neutral BevelButton, leading GitHub brand tile, `→ /api/auth/oauth/github`.
- **Authelia SSO button** (env-gated): rendered **only** when enabled; a neutral
  BevelButton `Sign in with SSO`, leading Authelia `A` tile, navigating to the
  **bare ingress path** `/auth/authelia/login` (NOT `/api/...` — commit 8e13faf
  precedent). Placed **last** in the OAuth group (least-common path for most users).
- **Register link:** below the action stack, a centered 13px row — `New here?`
  (`text-fg-muted`) + a text-button `Create an account` (`text-accent-blue-deep`,
  underline-on-hover) routing to `/register`. (A link, not a tab toggle — separate
  route per CONTEXT discretion; keeps the login Window single-purpose.)
- **Login statusbar** (sketch 007 footer, optional but specced): a `bg-bg-panel-2`
  `border-t-2 border-border-ink` strip, `padding: 4px var(--sp-3)`, 11px mono —
  left `caps lock: on/off` (live, accessibility nicety), right `hh:mm` clock.

**Resolved (autonomous): `VITE_AUTHELIA_ENABLED` build-time flag.** CONTEXT
offers env-flag vs backend-probe. Decision: **build-time `VITE_AUTHELIA_ENABLED`**
(no public config-probe endpoint exists in the auth handlers — confirmed by
source scan; adding one is out of scope). The button is tree-shaken/hidden when
the flag is falsy. The button routes to the bare ingress path regardless (the
ingress, not the SPA, owns Authelia trust).

**Error band (error taxonomy — sketch 007 `.error` treatment).** A single
in-window band above the form: `flex items-center gap-sp-2 border-2 border-danger
bg-danger-bg text-danger px-sp-3 py-sp-2 text-[13px] font-semibold`, `✕`-prefixed,
`role="alert"`. Copy is taxonomy-driven (Copywriting Contract). The shipped
generic "Wrong email or password" band is the `denied` case; Phase 5 widens it to
read an error reason (from login failure or `?error=` on a same-page OAuth bounce).

### 2. Register form (`/register` route)

A second centered `Window` (blue titlebar `CREATE ACCOUNT`), same width/chrome as
login, same brandline + watermark desktop. Built from **RetroFormField + RetroInput**:

| Field | Control | Validation (zod, mirrors backend) | Error copy |
|-------|---------|-----------------------------------|------------|
| Full name | RetroInput (text) | `min(1)` required | `Enter your name.` |
| Email | RetroInput (`type=email`, `mono`) | `z.email()` | `Enter a valid email address.` |
| Password | RetroInput (`type=password`, `mono`) | `min(8)` | `Use at least 8 characters.` |
| Confirm password | RetroInput (`type=password`, `mono`) | `.refine(===password)` | `Passwords don't match.` |

- Each field is a **RetroFormField** — label (12px UPPERCASE), control, hint/error
  swap, `aria-invalid` + `aria-describedby` per the Phase 4 shared field states.
  The password field carries a persistent **hint** `At least 8 characters.`
  (becomes the error on violation).
- Submit: full-width primary `BevelButton` `Create account`; `disabled` while
  `isSubmitting`.
- **Server-error band** (above the form, sketch-007 treatment): email-taken /
  other 4xx → `That email is already registered. Log in instead?` with an inline
  `Log in` link to `/login`.
- **Success → authenticated transition:** on 2xx the cookie is set (backend
  returns cookies + refresh token). `setRefreshToken(data.refresh_token)`, fire a
  **success toast** `Account created — welcome.` (mint), then `navigate("/")`
  (the dashboard) — identical to the login success path (no email-verify wall).
- OAuth group + divider repeat below the form (a user can sign up via provider
  from here too) — same buttons as login; the register route reuses the OAuth/SSO
  block component.
- Below: `Already have an account?` + `Log in` link → `/login`.

### 3. `/auth/callback` page (OAuth code exchange)

Route `frontend2/src/features/auth/CallbackPage.tsx` at `/auth/callback`.
Renders the **same centered-Window-on-desktop chrome** (no AppShell — the user is
not authenticated until exchange succeeds). On mount: read `?code` / `?error`
from the URL.

- **Loading state (the retro "progress" idiom — DEFINED here):** a Window (blue
  titlebar `SIGNING IN`) whose body centers a **stepped progress block**:
  - A 32×32px `bevel-sunken` square cell (`bg-bg-panel-2 border-2 border-border-ink`)
    housing a hard **stepped block march** — an ink square that advances through
    cells via a CSS `steps()` keyframe (NO smooth spin — System 7 has no eased
    motion; this matches the StatusDot `steps()` blink precedent):
    ```css
    @keyframes retro-progress { 0%{opacity:1} 33%{opacity:.4} 66%{opacity:.7} 100%{opacity:1} }
    .retro-progress { animation: retro-progress 0.9s steps(3,end) infinite; }
    ```
  - Below: 14px body `Completing sign-in…` (`text-fg-muted`), and a mono caption
    `exchanging credentials` (12px `tabular-nums` feel). **Respects
    `prefers-reduced-motion`** → the block holds solid (no animation).
  - **Resolved (autonomous): retro-os spinner = stepped block, not a spinner.** A
    rotating spinner is a material idiom alien to System 7. The stepped-opacity
    block reuses the established `steps()` motion convention (StatusDot blink) and
    the bevel-sunken cell — one motion language across the app.
- **On success:** `POST /auth/oauth/exchange { code }` → cookies set +
  `setRefreshToken` → a brief mint **success flash** (titlebar swaps mint,
  `Signed in ✓`, ~400ms) → `navigate("/")`. (One-time code is single-use — guard
  against React-StrictMode double-fire with a ref latch so the second invoke
  doesn't hit a consumed code and show a false error.)
- **Error states** (driven by `?error=` taxonomy OR exchange failure): the Window
  titlebar swaps **pink**, the body shows the in-window **error band** + copy +
  **two BevelButtons**: primary `Back to login` (→ `/login`) and neutral
  `Try again` (re-initiates the same provider, where the provider is known from
  `?provider`/state; otherwise omitted). No infinite auto-retry.

### 4. Workspace switcher (TopBar pill → Popover)

Replaces the Phase 3 **disabled placeholder** workspace pill (TopBar slot 3) with
a live trigger. Composes the Phase 4 **Popover** (chromeless-utility) anchored
under the pill.

- **Trigger:** the existing bevel pill (`bg-bg-panel-2 border-2 border-border-ink`,
  13px Plex Sans 600) — current workspace name + `▾`. Now interactive:
  `aria-haspopup="listbox" aria-expanded`. Becomes a `<button>`; loses the Phase 3
  `aria-disabled`.
- **Popover (plain titlebar):** `min-w-[220px]`, `bevel-raised`, `z-40`. Optional
  10px UPPERCASE `tracking-[0.14em] text-fg-muted` header `WORKSPACES`. Rows =
  Popover items (`role="option"`):
  - Each row: workspace name (14px) + a trailing `✓` glyph on the **current**
    workspace; the current row carries the accent `bg-titlebar-blue` fill +
    `aria-selected="true"`.
  - Hover/keyboard-active row = `bg-titlebar-blue` (the Phase 4 active-descendant rule).
  - Click a non-current row → `setCurrentWorkspaceId(id)` (D-12 context, persisted
    to localStorage), close popover, **invalidate all workspace-scoped query keys**
    (entity caches re-fetch under the new wsId), fire a mint toast
    `Switched to {name}.`
- **Loading state:** while `GET /users/me/workspaces` is in flight, the pill shows
  a skeleton label (`text-fg-faint` em-dash placeholder, `aria-busy`); the popover,
  if opened early, shows a single muted row `Loading workspaces…`.
- **Empty / single state:** with exactly one workspace the pill renders the name
  but is **non-expanding** (no `▾`, `aria-disabled`) — nothing to switch to. With
  **zero** workspaces (edge: deactivated/removed) the popover shows a
  RetroEmptyState-style row `No workspaces. Contact an owner.` (no create flow this
  phase — members/workspace creation is Phase 12).
- ESC closes via `useModalStack` (LOCKED — never logs out). Focus returns to the pill.

### 5. Settings skeleton (`/settings/*`)

A **minimal but stable** hub shell (full hub is Phase 12). Rendered **inside
AppShell** (chrome + Bottombar/FAB live). PageHeader breadcrumb
`SYSTEM › SETTINGS`.

- **Hub shell:** `/settings` redirects to `/settings/security`. The two built
  pages are presented as **RetroTabs** (Phase 4 folder tabs) across the top of the
  settings `main` pane — `Security` and `Connected Accounts` (active tab =
  `bg-titlebar-blue` accent fill). Tabs map to routes `/settings/security` and
  `/settings/accounts` (deep-linkable; the active route drives the selected tab).
  Future hub tabs (Profile, Members, Preferences — Phase 12) are **not rendered**
  this phase (no disabled stubs — keep the tab row truthful).
- Content column: `max-w-[720px]`, stacked **cards** (each a `Window` with a blue
  titlebar) `gap-sp-5`.

#### 5a. Security page (`/settings/security`)

Three cards, top to bottom:

**Card A — Active Sessions** (titlebar `SESSIONS`, blue):
- A **RetroTable** of sessions from `GET /users/me/sessions`. Columns: **Device**
  (parsed UA, 14px body + a mono sub-line for browser/OS), **IP** (mono, 12px),
  **Last active** (`last_active_at`, mono relative + absolute on hover), **action**
  (right-aligned).
- The **current session row** carries a leading `RetroBadge variant="info"`
  `THIS DEVICE` and **no revoke button** (you can't revoke the session you're in
  from the per-row action — use Log out for that). Other rows get a neutral inline
  `BevelButton` `Revoke` (the sketch-008 inline-row-action density).
- **Revoke-all-others:** a card-footer (`bg-bg-panel-2` recessed strip) holding a
  **danger BevelButton** `Revoke all other sessions` → **RetroConfirmDialog** (pink
  titlebar) before firing `DELETE /users/me/sessions`. Disabled when only the
  current session exists.
- Empty (shouldn't happen — you're always in a session): RetroEmptyState `No other
  sessions.`
- On revoke success → mint toast `Session revoked.` / `All other sessions revoked.`
  and re-fetch the list.

**Card B — Password** (titlebar `PASSWORD`, blue):
- **Two modes, driven by `has_password` (from `GET /users/me`):**
  - **Change-password** (`has_password === true`): RetroFormField stack — `Current
    password` (RetroInput password), `New password` (min-8, hint `At least 8
    characters.`), `Confirm new password`. Submit primary BevelButton `Change
    password` → `PATCH /users/me/password { current_password, new_password }`.
  - **Set-password** (`has_password === false`, OAuth-only accounts): a **butter
    explainer band** (warning, non-destructive) `You signed in with a provider and
    haven't set a password yet. Add one to enable email + password login.` then the
    `New password` + `Confirm` fields only (no current-password) → `PATCH
    /users/me/password { new_password }`. Submit `Set password`.
- Error (wrong current password → 4xx): inline error band `Current password is
  incorrect.` Success → mint toast `Password updated.`

**Card C — Danger Zone** (titlebar **pink** `DANGER ZONE`):
- Eyebrow 11px UPPERCASE `tracking-[0.14em] text-warn-deep` + a one-line consequence
  body `Deleting your account is permanent and cannot be undone.`
- On mount the card calls `GET /users/me/can-delete`:
  - `can_delete === true`: a **danger BevelButton** `Delete my account` → opens the
    **type-DELETE RetroConfirmDialog** (pink titlebar): body `Type DELETE to
    confirm. This permanently removes your account and cannot be undone.` + a
    RetroInput (the confirmation field); the confirm button (`Delete account`,
    `variant="danger"`) is **disabled until the input === "DELETE"** (case-insensitive
    per backend). Focus defaults to **Cancel** (Phase 4 safe-default rule). Confirm →
    `DELETE /users/me { confirmation: "DELETE" }` → clears query cache + workspace
    context → `navigate("/login")`.
  - `can_delete === false`: the button is **disabled**; a **danger band** lists the
    blocking workspaces — `You're the sole owner of: {names}. Transfer ownership or
    delete these workspaces before deleting your account.` (Member transfer UI is
    Phase 12 — this phase only *surfaces* the block.)

#### 5b. Connected Accounts page (`/settings/accounts`)

One card (titlebar `CONNECTED ACCOUNTS`, blue). A **provider row** per supported
provider (Google, GitHub), reconciling `GET /auth/oauth/accounts` against the
supported set:

- **Row layout:** `flex items-center gap-sp-3 py-sp-3 border-b border-table-rule`
  — leading 28px ink-bordered provider tile (Silkscreen initial), provider name
  (14px) + linked email/`display_name` sub-line (12px `text-fg-muted` mono),
  spacer, a **status pill** (`LINKED` ok / `NOT LINKED` neutral), then the action.
- **Not linked → `Link`** neutral BevelButton → browser nav to `/api/auth/oauth/{provider}`
  (the same initiate path; on return the account is linked and the row flips).
- **Linked → `Unlink`** neutral BevelButton → **RetroConfirmDialog** (pink)
  `Unlink {provider}? You'll no longer be able to sign in with it.` → `DELETE
  /auth/oauth/accounts/{provider}`.
- **Lockout guard:** if unlinking would remove the user's **only** sign-in method
  (no password set AND this is the last linked provider), the **Unlink button is
  disabled** with a butter inline note `This is your only way to sign in. Set a
  password first to unlink.` (computed client-side from `has_password` + linked
  count). If the backend still 409s (`ErrCannotUnlinkLastAuth`), surface a **danger
  toast** `Can't unlink — it's your only sign-in method. Set a password first.`
- Success → mint toast `{Provider} linked.` / `{Provider} unlinked.` and re-fetch.

| Concern | Treatment |
|---------|-----------|

| OAuth buttons | real `<a>`/`<button>` performing a full navigation (OAuth requires a top-level redirect, not fetch); provider brand tile is `aria-hidden` (decorative) — the button text carries the accessible name. |
| Callback live region | `SIGNING IN` Window body `aria-busy="true"` while exchanging; the loading caption is in a `role="status"` (polite) region; error state swaps to `role="alert"`. |
| Switcher | trigger `aria-haspopup="listbox" aria-expanded`; popover `role="listbox"`, rows `role="option" aria-selected`; current row `aria-current="true"`. Keyboard: ↑/↓ move active descendant, Enter selects, ESC closes via `useModalStack`. Focus returns to the pill on close. |
