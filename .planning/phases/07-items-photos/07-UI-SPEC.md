# Phase 7 — UI Design Contract

> Visual and interaction contract for **Items + Photos**: the items list (the
> make-or-break density surface — sketch 008 IS this page), item detail with a
> photo gallery + lightbox, create/edit forms, and the photo upload pipeline.
> Generated by gsd-ui-researcher (autonomous run), verified by gsd-ui-checker.

> **Direction anchor.** Retro OS Pastel (sketches 006-008, MANIFEST + the
> sketch-findings skill — BINDING). This phase **consumes** the Phase 4 atom
> library wholesale and **invents no new atoms** (CONTEXT locked). Every
> treatment below is a *composition* of shipped atoms (`RetroTable`,
> `FilterBar`, `FilterPopover`, `BulkActionBar`, `SavedFilters`,
> `RetroPagination`, `useTableSelection`, `RetroTabs`, `RetroDialog`,
> `RetroConfirmDialog`, `RetroFormField`/`RetroInput`/`RetroSelect`/
> `RetroCombobox`/`RetroTextarea`/`RetroFileInput`, `RetroEmptyState`,
> `StatusPill`/`RetroBadge`, `retroToast`, `Window`, `BevelButton`) on the Phase
> 2 token system + Phase 3 chrome (AppShell / Sidebar / Bottombar / modal
> stack). Legacy `frontend/components/items/**` is consulted for **layout +
> field composition only**; its stack (shadcn Dialog, `next/image`, `@dnd-kit`,
> `lucide`, sonner-direct) does NOT carry over — each maps to a shipped retro
> atom below. Every re-interpretation is recorded as **Resolved (autonomous)**.

**No NEW contrast pairing is introduced this phase** — every text/bg pair already

danger-on-danger-bg + danger-on-cream, deep companions on white, muted-on-panel).
**Flag for AA test: none.**

## 1 · Items List Page (`/items`) — sketch 008, hardened

One **mint-titled `Window`** (`ITEMS — {workspace name}`), `bodyClassName=""`
(flush, table-density), tiled in AppShell. Sidebar **INVENTORY** group "Items"
entry active. `useShortcuts("items", [N|/|F])` registers route chips on the
Bottombar.

### Window-body composition (top → bottom)

1. **`FilterBar`** (recessed `bg-bg-panel-2` strip):
   - Search input (`type="search"`, sunken, `w-[260px]`, placeholder `Filter items…`) → drives `?q=`.
   - Facet triggers (`FilterPopover`): `CATEGORY ▾`, `LOCATION ▾` (multi-select checklists) → `?category=`, `?location=`. **`ARCHIVED ▾`** facet exposes the archived toggle (default hidden) → `?archived=`.
   - `flex-1` spacer, then a mono `{n} items` count.
   - **Primary CTA slot**: a `BevelButton variant="mint"` `⊕ ADD ITEM` → `/items/new`.
   - **Active-filter chips row** (when filters applied): blue chips with ink `✕` per facet + a `CLEAR ALL` reset (FilterBar built-in).
2. **`SavedFilters`** row, placed **immediately above the FilterBar inside the same panel-2 strip region** (so presets read as part of the filter toolbar): preset chips (active = blue, `aria-pressed`) + `▾ PRESETS` menu (`SAVE CURRENT…`, per-preset delete → confirm). Presets serialize the full `?q=&category=&location=&archived=&sort=&page=` shape (localStorage, per-table key) — **Resolved (autonomous): SavedFilters sits above FilterBar** so the applied-filter chips and the preset chips don't visually collide (presets = saved state, chips = live state).
3. **`RetroTable`** (the density surface — column set below).
4. **`RetroPagination`** strip (server-driven, 25/page — CONTEXT locked; meta reads `page {n} of {m} · 25 / page`). **Resolved (autonomous): per-page = 25** (sketch 008's "50 / page" was sketch filler; CONTEXT locks 25).

### Column set (Resolved — autonomous, per sketch 008 density)

| # | Column | Cell content | Class | Notes |
|---|--------|--------------|-------|-------|
| 1 | (select) | per-row `RetroCheckbox` | — | Appears when bulk-select active; header cell = select-all checkbox (indeterminate dash on partial). |
| 2 | (thumb) | 26×26px ink-framed primary-photo thumbnail (or `◇` placeholder glyph when no photo) | — | Verbatim sketch 008 `.thumb`. |
| 3 | **Name** | item name | `td.name` (600 weight) | Row click → detail. |
| 4 | **Category** | category label | — | **Added vs sketch 008** (sketch showed Barcode here; category is higher-signal for scanning a home inventory). |
| 5 | **Location** | `Garage / A2` style path | — | Sketch 008 verbatim. |
| 6 | **Qty** | quantity | `td.mono` (`tabular-nums`, right-align) | Sketch 008. |
| 7 | **Status** | `StatusPill` (ok/warn/info/danger) | — | Sketch 008 badge column. |
| 8 | (actions) | `EDIT` + `↧` (overflow menu) inline buttons | `td.actions` (right) | Borderless-on-rest, bevel-on-hover fallback (sketch-findings anti-pattern note) acceptable; default ship = the sketch-008 small beveled buttons. |

**Resolved (autonomous): Barcode is NOT a list column.** Sketch 008 showed it,
but barcode is a mono lookup key, not a scan-the-list signal; it lives on the
detail page + is searchable via `?q=`. Category replaces it in the column set
for home-inventory legibility. Labels are NOT a column (they live on detail —
read-only until Phase 10); `updated` is NOT a column (low signal at density —
shown on detail). This keeps the table at 6 visible data columns, inside sketch
008's validated density envelope.

### Row interaction

- **Row click → `/items/{id}`** (whole row is the navigation target; `cursor-pointer`). The leading checkbox cell and the trailing action buttons `stopPropagation` so selecting/editing never navigates.
- **Sort**: clickable `<th>` on Name / Category / Location / Qty / Status toggles `?sort=` (asc/desc); active sort header shows a `↑`/`↓` mono glyph. URL-driven (deep-linkable).
- **Hover**: `bg-info-bg` (existing `.rtable` rule). **Selected**: full `bg-titlebar-blue` + ink rule.

### Bulk selection visual + BulkActionBar / Bottombar interplay (LOCKED Phase 4 contract)

- `useTableSelection(rows)` drives selection (click selects single; Shift+Click range; Ctrl/Cmd+Click toggle). Selected rows = blue fill; range = contiguous blue block with a 2px ink left-border on the anchor/focus endpoints.
- **Desktop (≥768px): bulk actions surface as Bottombar key-cap chips** — the selection context registers a bulk-action set into the Phase 3 shortcuts SSOT when `selected.size > 0`. **There is no second desktop action bar** (Phase 4 Success Criterion 5).
- **Mobile (<768px, no Bottombar): the `BulkActionBar` atom renders inline** above the FAB: `{n} SELECTED` badge, `ARCHIVE` (neutral) + `DELETE` (danger → RetroConfirmDialog) + `✕ CLEAR`.
- Bulk actions this phase: **ARCHIVE / UNARCHIVE** (non-destructive) and **DELETE** (destructive, only on already-archived selections — routes through RetroConfirmDialog, pink). Delete of a non-archived selection is disabled with a hint toast.

### Archived-row treatment (Resolved — autonomous)

Archived items are hidden by default (filtered out) and surfaced only when the
`ARCHIVED` facet is on. When shown:
- Row text → `text-fg-muted` (the row reads "dimmed but present"), thumb frame keeps its ink border but the image is `opacity-60`.
- An `ARCHIVED` **neutral `RetroBadge`** sits in the Status column **before** the status pill.
- Action buttons swap: `EDIT` → `RESTORE` (mint) + the `↧` menu exposes `DELETE…` (danger, type-to-confirm). `text-fg-muted` is sanctioned here (whole-row dimming is a state, not running body copy — the badge + restore button carry the signal non-color-only).

### Empty + filtered-empty states (`RetroEmptyState`)

- **No items at all** (zero in workspace): eyebrow `INVENTORY`, glyph `◇`, heading `NO ITEMS YET`, body `Your warehouse is empty. Add your first item to start tracking it.`, action `⊕ ADD ITEM` → `/items/new`.
- **Filtered, no matches**: heading `NO MATCHES`, body `No items match these filters. Clear a filter or adjust your search.`, action `CLEAR ALL` (clears `?q/category/location/archived`).
- **Only-archived hidden** (all matches are archived, archived facet off): body adds the hint `Some items are archived — turn on the Archived filter to see them.`
- States render **inside the table window body** (replacing `<tbody>`), pager hidden when empty.

## 2 · Item Detail (`/items/{id}`)

One **mint-titled `Window`**, titlebar = the item name (UPPERCASED, truncates;
Silkscreen ≥16px rule satisfied by the titlebar `uppercase` + `truncate`). The
titlebar `actions` slot carries `EDIT` (`BevelButton` neutral → `/items/{id}/edit`)
+ an `↧` overflow menu (`ARCHIVE` / `RESTORE`, `DELETE…`) before the decorative
corner box. Two-column body at `lg` (`grid-cols-[minmax(0,1fr)_320px]`), single
column below.

**Resolved (autonomous): detail uses a tabbed main pane, NOT a long scroll.**
Left column = `RetroTabs` with three folder tabs: **DETAILS**, **PHOTOS**,
**HISTORY**. Right column = a persistent side rail (loan status + inventory
stub) visible regardless of active tab. Folder tabs are canonical System-7 and
keep the make-or-break density discipline off the detail page too.

### Left column — tabs

**DETAILS tab** — field layout: a definition-style two-column grid
(`grid-cols-[minmax(0,140px)_1fr] gap-x-sp-4 gap-y-sp-3`) of 12px UPPERCASE
`text-fg-muted` labels + 14px ink values:
- Name · Category · Location · Quantity (mono) · **Barcode** (mono, with a small `COPY` neutral button) · Description (full-width, wraps) · Created / Updated timestamps (mono `tabular-nums`, muted).
- **Labels row** (full-width, below fields): a wrap of label chips (`RetroBadge` neutral). **Read-only attach/detach this phase** — each chip carries an ink `✕` to detach (attach/detach endpoints exist); an `⊕ ADD LABEL` neutral button opens a `FilterPopover`-style checklist of existing workspace labels to attach. Label *creation/management* is Phase 10 (the popover lists existing labels only; if empty → `No labels yet — manage labels in Phase 10.` muted row).

**PHOTOS tab** — the gallery (see §4 gallery grid) + the upload entry point
(`⊕ ADD PHOTOS` → upload dialog).

**HISTORY tab** — loan history list (read-only, `["loans", wsId, ...]` keys):
each entry a `bg-bg-panel-2` row — borrower · checkout date · return date (or
`— still out` mono) · status pill. Empty → `RetroEmptyState` glyph `◇`, heading
`NO LOAN HISTORY`, body `This item has never been loaned out.`

### Right column — side rail (persistent)

- **Active-loan panel** (`Window` titlebar **pink** when on loan = attention, **mint** when available):
  - On loan: `● On loan to {borrower}` (`text-accent-pink-deep`, no fill) + due date (mono) + a read-only `RETURN` affordance disabled with hint `Loan actions arrive in Phase 8.` (CRUD is Phase 8; panel is read-only here).
  - Available: `● Available` (`text-accent-mint-deep`).
- **Inventory STUB panel** (`Window`, plain/`bg-bg-panel-2` titlebar `INVENTORY`):
  **Resolved (autonomous): explicit "coming with 7b" styled placeholder.** A
  recessed `bg-bg-panel-2 bevel-sunken` panel with a 10px eyebrow `INVENTORY`,
  a centered `◇` glyph, and muted body `Stock entries arrive in 7b.` The panel
  is a **real slot** (a named region in the detail layout) so 7b swaps its
  contents in without relayout — it is NOT a hidden/absent element. No CTA.

### Detail empty/error

- Item not found / deleted: full-window `RetroEmptyState`, heading `ITEM NOT FOUND`, body `This item may have been deleted. Return to the items list.`, action `← BACK TO ITEMS` → `/items`.
- Load error: error toast (`retroToast.error`, persists) + inline `RetroEmptyState` heading `COULDN'T LOAD ITEM`, body `Something went wrong. Try again.`, action `RETRY`.

## 3 · Create / Edit Form (`/items/new`, `/items/{id}/edit`)

A **blue-titled `Window`** (`ADD ITEM` / `EDIT ITEM`), rendered as a centered
page panel (`max-w-[560px] mx-auto`) under AppShell — RHF + zod, optimistic
invalidation of `["items", wsId, ...]` + detail keys (CONTEXT). **Resolved
(autonomous): single-window form, not a multi-step wizard.** Legacy used a
create/edit *wizard*; the retro-os create surface is one window with grouped
fields (System-7 dialogs don't wizard a CRUD create). Drafts/sessionStorage
autosave is SKIPPED (CONTEXT deferred — not trivially cheap).

### Field grouping (top → bottom, `RetroFormField`-wrapped)

1. **Identity**: Name (`RetroInput`, required, `*` ink marker) · Category (`RetroCombobox` — existing categories, free-typeable) · Location (`RetroCombobox`).
2. **Quantity & code**: Quantity (`RetroInput type="number"`, mono) · **Barcode** (`RetroInput`, mono).
3. **Notes**: Description (`RetroTextarea`, `min-h-[88px]`).

### Barcode field w/ prefill state (Resolved — autonomous)

- `/items/new?barcode=XXXX` prefills the Barcode field. When prefilled, the field shows a hint line: `Prefilled from scan — edit if needed.` (`text-fg-muted`) and the field gets a subtle **filled** affordance: an inline `RetroBadge` `info` reading `FROM SCAN` sits in the field's `RetroFormField` label row (right-aligned). No special validation — the value is editable like any other.
- The `itemsApi.lookupByBarcode` helper (404→null, `encodeURIComponent`, case-sensitivity guard — the Phase 65 locked pattern) is re-added if the frontend2 wipe removed it; it powers the scan→prefill path (full scan page is Phase 11) and must keep the G-65-01 integration test green.

### Validation error treatment (sketch 007 in-window style)

- Per-field: `RetroFormField` error line — `✕`-prefixed `text-[12px] font-semibold text-danger`, `aria-invalid` + `aria-describedby`; the field flips to `border-danger bg-danger-bg`. Required marker is **ink `*`** (color reserved for the actual error).
- Form-level (submit failure / server error): a `bg-danger-bg border-2 border-border-ink` banner at the top of the window body — `✕ Couldn't save this item. {reason}` (14px `text-danger`) — plus a persistent `retroToast.error`.

### Submit / cancel placement

- Footer action row pinned to the window bottom: `border-t-2 border-border-ink pt-sp-3`, right-aligned, `gap-sp-2`: **`CANCEL`** (`BevelButton` neutral → back / discard) + **`SAVE ITEM`** (create) / **`SAVE CHANGES`** (edit) (`BevelButton variant="primary"`).
- **Unsaved-changes guard**: navigating away with a dirty form opens a **butter** RetroConfirmDialog `DISCARD CHANGES?` / body `Your edits will be lost.` / confirm `DISCARD` (neutral) / cancel `KEEP EDITING`.
- On success: `retroToast.success` `DONE` `Item saved.` + invalidate + navigate to detail (create → new detail; edit → back to detail).

## 4 · Photo Pipeline UI

Client pipeline: canvas resize/compression + EXIF orientation fix (ported
STRUCTURE from legacy `image.ts`, **no new deps** — hand-rolled canvas path),
10MB cap, JPEG/PNG/HEIC accept-list, native FormData multipart via `api.ts`
(locked — no upload library, no `@dnd-kit`, no `next/image`, no sonner-direct).

### Upload zone (`RetroFileInput` atom)

- A **blue-titled `RetroDialog`** `ADD PHOTOS` opened from detail PHOTOS tab / `⊕ ADD PHOTOS`.
- Body top: `RetroFileInput` (sunken dashed drop zone + `BROWSE…`), `accept="image/jpeg,image/png,image/heic"`, `maxSize={10 * 1024 * 1024}`, `multiple`. Dragover → `border-titlebar-blue bg-info-bg`, prompt `Release to add`. Oversized/wrong-type → the atom's `✕`-prefixed danger error.
- **HEIC**: accept-list includes HEIC; if backend rejects unconverted HEIC (research verifies), the client adds it to `accept` only and surfaces a per-file warn badge `HEIC — converted on upload`. **Resolved (autonomous): accept HEIC at the picker; defer conversion to the backend** (no client HEIC decode dep).

### Per-file progress (Resolved — autonomous: per-file, not batch)

Below the file list, each queued file is a `bg-bg-panel-2 border-2 border-border-ink`
row: 26×26px thumb preview · mono filename · mono `tabular-nums` size · a
**per-file progress affordance**: a sunken-bevel track with a `bg-titlebar-blue`
fill (mono `tabular-nums` `{pct}%`), → `✓ DONE` (mint badge) on success / `✕ FAILED`
(danger badge + `RETRY`) on error. Per-file (not one batch bar) so a single
failure is isolated and retryable. Overall footer: `{done}/{total} uploaded`.

### Duplicate-warning dialog (Resolved — autonomous: butter titlebar)

On upload, call `/photos/check-duplicate` **before committing**. On a hit, a
**butter-titled `RetroDialog`** `POSSIBLE DUPLICATE` (warning semantic, NOT a
`RetroConfirmDialog` — the legacy `AlertTriangle` maps to the butter
warning titlebar, not pink danger):
- Body: `Found {n} similar photo(s) already on this item.` + a `flex flex-wrap gap-sp-2` of up to 4 existing-photo thumbnails (each 80×80px ink-framed, with a mono `{similarity}%` badge overlay in an ink-bordered corner chip); `+{n-4}` overflow tile.
- Footer: `CANCEL` (neutral — skip this file) + `UPLOAD ANYWAY` (`primary`). Per-file decision; uploading state disables both.

### Caption edit affordance

- Inline on the gallery thumbnail (hover/focus reveals an `✎ CAPTION` neutral button) → a small **blue `RetroDialog`** `EDIT CAPTION` with a `RetroTextarea` (200-char limit, mono `tabular-nums` counter `{n}/200`), `SAVE` / `CANCEL` footer. Enter-to-save, ESC-to-cancel (via modal stack).

### Reorder mechanism (Resolved — autonomous: buttons, NOT drag)

**Drag-and-drop reorder is declined** (legacy `@dnd-kit` is a new dep + a
keyboard/AT-accessibility burden the retro-os ban on new deps forbids). Each
gallery thumbnail carries **`◂` / `▸` move buttons** (overlaid on hover/focus,
borderless-on-rest → bevel-on-hover) that swap order with the neighbor; the
move is keyboard-operable and announced (`aria-label="Move {caption} earlier/later"`).
Simplest accessible mechanism; persists via the reorder endpoint with optimistic
update + revert-on-error (`retroToast.error`).

### Gallery grid + primary badge + bulk-photo selection

- Grid: `grid` `repeat(auto-fill, minmax(96px, 1fr))` `gap-sp-1`; each cell = a 96×96px ink-framed square-crop thumbnail (`object-cover`). Click → lightbox at that index.
- **Primary badge**: the primary photo's cell carries a `★ PRIMARY` `RetroBadge` (`info`/blue, top-left, 2px radius). Non-primary cells expose a `★ SET PRIMARY` neutral button on hover/focus.
- Per-photo `✕ DELETE` (danger button on hover/focus) → pink `RetroConfirmDialog` `DELETE PHOTO?` / `This photo will be permanently removed.` / confirm `DELETE`.
- **Bulk photo select**: a `SELECT` toggle on the gallery toolbar turns cells into selectable tiles (click toggles a 2px ink outline + blue inner ring + `RetroCheckbox` corner). A gallery `BulkActionBar` appears: `{n} SELECTED` · `EDIT CAPTION` (bulk-apply one caption) · `DELETE` (danger → confirm) · `✕ CLEAR`.
- **Zip download**: a gallery toolbar `⤓ DOWNLOAD ALL` neutral `BevelButton` → triggers the backend zip endpoint (browser download). When photos are bulk-selected, it reads `⤓ DOWNLOAD {n}` and zips the selection.

### Lightbox (Resolved — autonomous: dark chromeless overlay, modal-stack ESC)

**Not** a pinstriped Window (a full-bleed photo viewer wants a neutral dark
field, per legacy). A `fixed inset-0 z-40 bg-fg-ink/85` overlay (darker than the
dialog scrim so the photo reads), rendered through the Phase 3 portal, ESC via
`useModalStack` (TUI-02 — no own document listener; CONTEXT locked).

- **Top chrome** (`bg-fg-ink/0` → no gradient; a thin `bg-bg-panel` toolbar strip pinned top so chrome stays on-brand): left = mono `tabular-nums` `{i} / {n}` index; right = `⊕` zoom-in / `⊖` zoom-out (1×–3×, mono `{pct}%`) · `⤓` download-original · `✕ CLOSE` — all `BevelButton` neutral (on the panel strip, so ink-on-panel, AA-safe — NOT white-on-photo).
- **Arrows**: 44×44px `◂` / `▸` `BevelButton`s vertically centered left/right (disabled at bounds); ←/→ keys navigate; touch swipe navigates (carried from legacy).
- **Caption display**: when present, a bottom `bg-bg-panel border-t-2 border-border-ink p-sp-3` strip with centered 14px body caption (ink-on-panel — readable, not white-on-gradient).
- **Resolved (autonomous): chrome is on opaque panel strips, not translucent-on-photo.** Legacy used white text on black gradients (fails the retro-os AA bar + the "colored/decorative text" discipline). Retro-os keeps lightbox controls on `bg-bg-panel` ink strips → ink-on-panel AA-safe, on-brand, and the photo still fills the dark field between them.
- A11y: `role="dialog" aria-modal aria-label="Photo {i} of {n}"`; focus trapped, restored on close; reduced-motion → no zoom/slide transition.

## Accessibility

| Concern | Treatment |
|---------|-----------|
| Focus-visible | Every interactive element reuses the Phase 4 ring: 2px ink `focus-visible:outline-2 outline-border-ink outline-offset-2` (controls, row checkboxes, sort headers, tabs, gallery thumb buttons, lightbox chrome, pager, chips). Form fields keep RetroInput's 3px `outline-titlebar-blue`. Never bare `outline:none`. |
| Row navigation | The row is a navigation target — implemented so keyboard users reach detail (the Name cell is a real link/button; whole-row click is a mouse convenience layered over it). |
| ESC discipline | Lightbox, upload dialog, duplicate dialog, caption editor, all confirms, filter/preset popovers route ESC EXCLUSIVELY through the Phase 3 `useModalStack` capture-phase arbiter. No surface owns a document ESC listener (TUI-02 LOCKED). Stack pops topmost-first; logout never fires on bare ESC. |
| Lightbox keys | ←/→ navigate, `+`/`-`/`0` zoom — registered while the lightbox owns the top of the modal stack, removed on close (not a permanent document listener). |
| Focus trap + restore | All dialogs/lightbox trap focus, restore to invoker on close; popovers move focus in, restore to trigger. |
| Error signalling | `aria-invalid` + `aria-describedby` + `✕` glyph + message (never color-only) — color-blind safe. |
| Selection state | `aria-selected` rows, `aria-multiselectable`, header select-all `aria-label`, keyboard row focus outline distinct from the blue selection fill. Gallery bulk-select tiles `aria-pressed`. |
| Status color | Status pills carry ink-on-pastel (AA); inline status text uses deep companions on white; archived dimming pairs with a non-color badge + restore action. |
| Touch targets | Lightbox arrows + per-photo action buttons ≥44px; row checkbox hit area ≥24px; mobile BulkActionBar buttons ≥44px. |
| Reduced motion | Upload progress, lightbox zoom/slide, toast motion respect `prefers-reduced-motion` → instant. |
| Contrast — NO NEW pairings | All pairs reuse Phase 2 verified tokens (ink-on-panel/panel-2/blue/ok/warn/info/danger-bg, danger-on-danger-bg + danger-on-cream, deep companions on white, muted-on-panel). Lightbox chrome is ink-on-panel (NOT white-on-photo). **Flag for AA test: none.** |
