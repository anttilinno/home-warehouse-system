## Three-Cue Discipline (color never the sole signal)

Where color conveys state, two non-color cues accompany it. This phase's stateful
surfaces and their three cues:

| State | Cue 1 (color) | Cue 2 (glyph/icon) | Cue 3 (text) |
|-------|---------------|--------------------|--------------|
| Active-loan count badge (>0) | `info` pastel fill | `↧` glyph prefix | `{n} OUT` uppercase label |
| Active-loan count = 0 | neutral (no fill) | — | em-dash `—` (no badge rendered) |
| Delete blocked | `danger` red badge + `danger-bg` banner | `⚠` glyph | `Active loans` text + "View active loans" link |
| Nothing out (panel) | `accent-mint-deep` text | `●` glyph | `Nothing out` text |
| Overdue chip (in panels) | `danger` pill | `⚠` glyph | `−{n}d` text — owned by `BorrowerLoanPanels`, not us |

## Surface 1 — `/borrowers` list (BORR-01, BORR-04)

Mirror `InventoryListPage` density exactly. Mint Window.

```
┌─ BORROWERS — {workspace} ──────────────────────────[□][▣]┐  ← Window, titlebarVariant="mint"
│ ┌── FilterBar (bg-panel-2 recessed) ────────────────────┐ │
│ │ [search: "Filter borrowers…"]   {n} borrowers   [⊕ NEW BORROWER] (mint) │
│ └────────────────────────────────────────────────────────┘ │
│ ┌── RetroTable ──────────────────────────────────────────┐ │
│ │ NAME            CONTACT              LOANS        ⌄ACTIONS│ │
│ │ Alice Smith     alice@ex.com         [↧ 2 OUT](info)  EDIT │ ← row click → /borrowers/:id
│ │ Bob Jones       +372 5555 123        —            EDIT │ │
│ │ Carol …         (mono, muted if —)                      │ │
│ └────────────────────────────────────────────────────────┘ │
│ ┌── RetroPagination (bg-panel-2) ◂PREV  [1] 2 3  NEXT▸ … ┐ │
└──────────────────────────────────────────────────────────┘
```

**Window:** `title={t`BORROWERS — ${workspaceName}`}`, `titlebarVariant="mint"`.

**FilterBar:**
- `searchPlaceholder={t`Filter borrowers…`}`
- `itemCount={visible.length}` (mono `{n} borrowers`)
- `facets={[]}` — borrowers have no enum facets (no status/condition). Empty facets
  array; the FilterBar still renders search + count + CTA.
- `filterChips` / `onRemoveFilter` / `onClearAll` — only the search chip applies; an
  active non-empty search shows a `CLEAR ALL` reset (mirror loans list, which has no
  facet chips either).
- `primaryAction`: `<BevelButton variant="mint" onClick={goNew}>⊕ NEW BORROWER</BevelButton>`

**RetroTable columns (4):**

| `<th>` | Cell | Notes |
|--------|------|-------|
| `Name` | `font-semibold` borrower name | sortable optional — match loans (loans list is unsorted; borrowers may stay unsorted for parity simplicity). Default: unsorted. |
| `Contact` | email if present, else phone, else muted `—` | `font-mono text-[13px]`; `text-fg-muted` when `—` |
| `Loans` | active-loan count badge (see below) or `—` | `info` `RetroBadge`: `↧ {n} OUT` when count>0; muted `—` when 0 |
| (actions) | `aria-hidden` `<th />`, right-aligned | `EDIT` BevelButton (neutral). No inline DELETE in the row — delete lives on the detail page behind the guard. |

**Row interaction:** whole `<tr>` is `cursor-pointer`, `onClick → navigate(/borrowers/:id)`.
The actions cell calls `e.stopPropagation()` (verbatim InventoryListPage pattern).

**Active-loan count source (OQ3 resolution — see Open Questions):** the list does
NOT call `loansApi.byBorrower` per row (N+1). For parity v3.0, the `Loans` column
renders the count proactively ONLY on the detail page; **on the list it shows the
badge from `total_borrowers`-style data if available, otherwise renders muted `—`**.
Planner decision allowed: if a cheap per-borrower count is not on `BorrowerResponse`
(it is not — see CONTEXT backend surface), the list omits live counts and shows `—`;
the authoritative count + delete guard live on the detail page. **Default: list
`Loans` column = `—` placeholder; detail page = live.** (Keeps the list to a single
`/borrowers` read, no per-row fan-out.)

**Empty state** (`RetroEmptyState`, glyph `☺` to match the nav glyph, or default `◇`):
- No borrowers, no search: heading `NO BORROWERS`, body + `⊕ ADD BORROWER` action → `goNew`.
- Search active, no match: heading `NO MATCHES`, body, `CLEAR ALL` action.

**Loading / error:** verbatim InventoryListPage — `Loading…` mono muted; error
`Couldn't load borrowers. Try again.` in `text-danger`.

**Shortcuts:** `N → /borrowers/new`, `/ → focus search`. Read `t` through a ref;
destructure stable `.mutate` (render-loop landmine, hit 4× — CONTEXT constraint 5).

**Sidebar nav:** flip the existing `Borrowers` entry in
`components/layout/Sidebar.tsx` line 145 from disabled to `to="/borrowers"`
(add the `to` prop; it already carries `glyph="☺"` and `count={stats?.total_borrowers}`).
This is the single nav change. The count badge on the nav uses the shipped
`stats.total_borrowers` (already wired) — no new dashboard work.

**Routes:** register in `routes/index.tsx` — **literal before param** (AP-1):
```
<Route path="borrowers/new" element={<BorrowerFormPage />} />   ← BEFORE the literal list
<Route path="borrowers" element={<BorrowerListPage />} />
<Route path="borrowers/:id/edit" element={<BorrowerFormPage />} />
<Route path="borrowers/:id" element={<BorrowerDetailPage />} />
```
`routes/index.tsx` is a shared single-writer file (Phase-8 lesson — CONTEXT plan split).

- `EDIT` — neutral BevelButton → `navigate(/borrowers/:id/edit)`.
- `DELETE…` — `variant="danger"`. **Disabled (`disabled` + `aria-disabled`) when the

**Profile grid:** verbatim ItemDetailPage `DetailsTab` `<dl>` /

**Delete guard (BORR-05 — the load-bearing decision):**

Active-loan count is read from `useBorrowerLoans(wsId, borrowerId)` (the SAME hook
`BorrowerLoanPanels` already uses — query is shared/cached, no extra fetch). When
`active.length > 0`:

1. **Proactive disable:** the titlebar `DELETE…` button is `disabled` + `aria-disabled`.
2. **Inline banner** between the profile grid and the loan panels (mirrors the form
   error banner chrome):
   ```
   <div role="status" className="border-2 border-border-ink bg-danger-bg p-sp-3 text-[14px] text-danger flex items-center gap-sp-2">
     <span aria-hidden="true">⚠</span>
     <span>Return the active loans before deleting this borrower.</span>
     <a href / Link to="/loans?tab=active"> View active loans </a>   (underlined, text-danger)
   </div>
   ```
   The "View active loans" link → `/loans?tab=active` (the shipped loans active tab).
3. **Reactive backstop:** if the user reaches DELETE anyway (race) and the API returns
   **400 "cannot delete borrower with active loans"**, catch that exact 400, show the
   same banner, and `retroToast.error(t`Couldn't delete — this borrower has active loans.`)`.
   (CONTEXT constraint 3: 400 string, NOT 409.)

**Delete confirm** (when allowed): `RetroConfirmDialog`, `titlebarVariant="pink"`
(danger family — NOT the type-to-confirm gate the item uses; borrowers are lighter):
- `title=DELETE BORROWER?`, `confirmLabel=DELETE` (`confirmVariant` defaults to
  danger), `cancelLabel=Cancel`, body
  `Delete {name}? This can't be undone.` On confirm → delete mutation →
  on success `navigate("/borrowers")` + success toast.
  (A plain confirm, not type-to-confirm — the active-loan guard already prevents the
  dangerous case; borrowers carry no irreplaceable child data like item photos.)

**Mounted panels:** `<BorrowerLoanPanels wsId={wsId} borrowerId={id} />` — DONE,

`BORROWER NOT FOUND` empty state with `← BACK TO BORROWERS` action; other error →

## Data Strategy (binds the UI to the backend)

Resolutions for CONTEXT's open questions, so the UI contract is unambiguous.

### Pagination (OQ1 / constraint 1) — RESOLVED: fetch ≤100 + client-paginate
The list endpoint returns a **bare `{ items: [] }` with NO `total`/`total_pages`**.
RetroPagination needs a `pageCount`. Resolution (matches the roadmap's explicit
"want RetroPagination" + the ≤100 cap): fetch `GET /borrowers?limit=100&page=1`
(clamped — 422 over 100), hold the single page client-side, and **client-paginate**
into RetroPagination pages of e.g. `BORROWERS_PER_PAGE = 25`:
- `pageCount = Math.max(1, Math.ceil(rows.length / PER_PAGE))`
- `perPage = PER_PAGE`
- page state is **client-local** (not URL-required; URL `?page` optional, mirror
  inventory's `?page` round-trip only if trivially free).
- Workspaces with >100 borrowers are out of v3.0 parity scope (seeded data is small);
  note the cap in a code comment as inventory does for its 100-item join.

### Search (OQ2 / constraint 2) — RESOLVED: client-filter the loaded ≤100 page
The list endpoint has **no `q` param**; a separate `/borrowers/search` endpoint exists
but is for the Phase-8 pickers. To stay consistent with **InventoryListPage and
LoansListPage (both client-filter, no URL round-trip)**, the `/borrowers` list search
box **client-filters the loaded ≤100 page** (name + contact substring, debounce not
required at ≤100 rows — neither inventory nor loans debounce). Do NOT wire the list
search box to `/borrowers/search`. The separate endpoint stays the picker's tool.
*(This overrides the CONTEXT prompt's "wired to /search endpoint, debounced" framing:
the shipped convention is client-filter; matching it is the parity mandate. Flagged in
Open Questions below.)*

### Delete guard active-loan count (OQ3) — RESOLVED: detail-only, via shared hook
Detail page reads active count from `useBorrowerLoans` (already mounted for the panels
— shared React Query cache, no extra fetch). Proactive disable + banner is primary;
the 400 catch is the backstop. The **list does NOT compute per-row counts** (avoids
N+1) — list `Loans` column shows `—`. (See Surface 1.)

### Archive (OQ4) — RESOLVED: out of scope
`POST /borrowers/{id}/archive` exists but BORR-05 is a **hard delete with the
active-loan guard**. No archive affordance in v3.0 parity (unlike items, which expose
ARCHIVE). The detail titlebar carries only EDIT + DELETE….

### Query keys (constraint 6)
`["borrowers", wsId, ...]` prefix for SSE invalidation. Loan reads stay under
`["loans", wsId, ...]` (panels' convention). Borrower mutations invalidate
