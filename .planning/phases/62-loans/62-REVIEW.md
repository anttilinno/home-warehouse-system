# Phase 62: Code Review Report

**Reviewed:** 2026-04-17
**Depth:** standard
**Files Reviewed:** 51
**Status:** issues_found

## Summary

Phase 62 implements the full loan lifecycle — create, return, extend due date, unified PATCH edit, and the item+borrower decoration layer for all loan-list endpoints. The backend domain model is cleanly layered: invariants in the entity, workspace scoping in the service, SQL generated by sqlc. The frontend uses TanStack Query with a consistent invalidation strategy and well-structured per-panel hooks.

Three critical issues are found. The `ReturnLoan` SQL query is not workspace-scoped, violating the defence-in-depth principle the codebase explicitly enforces. The `LoanRepository.Delete` implementation silently no-ops, which is a correctness hazard for any future caller. Two frontend links in the borrower panel navigate to `/items/{inventory_id}` instead of `/items/{item_id}`, causing guaranteed 404s when a user clicks through from a borrower's active or history loan row to the item detail page. Five warnings cover a non-atomic create/return path, a dead `nil` guard masking a data-loss risk in Return, stale-check gaps in `Save`, a misleading parameter name that will cause future misuse, and a cache invalidation that targets the wrong key type. Four info items cover dead SQL queries, a type mismatch on the return API, an unused interface method, and pagination UX debt.

## Critical Issues

### CR-01: `ReturnLoan` SQL is not workspace-scoped — violates defence-in-depth principle

**File:** `backend/db/queries/loans.sql:10-14` / `backend/internal/infra/queries/loans.sql.go:700-724`

**Issue:** The `ReturnLoan` query filters only on `id`:

```sql
UPDATE warehouse.loans
SET returned_at = now(), updated_at = now()
WHERE id = $1
RETURNING *;
```

All other mutating queries (`UpdateLoan`, `ExtendLoanDueDate`) include `AND workspace_id = $N`. The codebase comments explicitly cite T-62-02 as the reason for workspace scoping in SQL. The current call path does verify workspace ownership via `GetLoan` before calling `ReturnLoan`, but if the call chain is ever refactored the scope guard is gone. The `Save` method in `loan_repository.go` reaches this branch at line 43 using only the loan's own ID.

**Fix:** Update `loans.sql` and regenerate with sqlc:

```sql
-- name: ReturnLoan :one
UPDATE warehouse.loans
SET returned_at = now(), updated_at = now()
WHERE id = $1 AND workspace_id = $2
RETURNING *;
```

Update `LoanRepository.Save` to pass `l.WorkspaceID()` to the query.

### CR-02: `LoanRepository.Delete` is a silent no-op — any caller will believe deletion succeeded when nothing happened

**File:** `backend/internal/infra/postgres/loan_repository.go:218-222`

**Issue:**

```go
func (r *LoanRepository) Delete(ctx context.Context, id uuid.UUID) error {
    // Note: This would typically be a hard delete or archive
    // For now, we'll just return nil as loans are typically not deleted
    return nil
}
```

`Delete` satisfies the `Repository` interface and returns nil without executing any SQL. Any caller depending on deletion being persisted will silently succeed with the record still in the database. This is not a theoretical risk — the `pendingchange` service (wired in router.go line 253) depends on `loanRepo` and may invoke Delete via the approval workflow.

**Fix:** Either remove `Delete` from the `Repository` interface, or implement it defensively:

```go
func (r *LoanRepository) Delete(ctx context.Context, id uuid.UUID) error {
    return errors.New("loan deletion is not implemented")
}
```

A panicking or error-returning stub is always safer than a silent no-op.

### CR-03: Borrower panel links navigate to `/items/{inventory_id}` — wrong ID type causes guaranteed 404

**File:** `frontend2/src/features/loans/panels/BorrowerActiveLoansPanel.tsx:86` / `frontend2/src/features/loans/panels/BorrowerLoanHistoryPanel.tsx:78`

**Issue:**

```tsx
<Link to={`/items/${loan.inventory_id}`}>
  {loan.item.name}
</Link>
```

The route `items/:id` in `routes/index.tsx` maps to `ItemDetailPage`, which calls `useItem(id)` fetching by the item definition ID. Passing `loan.inventory_id` (an inventory row UUID) to this route will produce either a "ITEM NOT FOUND" error state or a 404 on every click. The same pattern appears in `LoanRow.tsx` line 44.

The loan response includes `loan.item.id` (the item definition ID, set by the decoration layer) which is the correct value.

**Fix:**

```tsx
<Link to={`/items/${loan.item.id}`}>
```

Apply to:
- `BorrowerActiveLoansPanel.tsx` line 86
- `BorrowerLoanHistoryPanel.tsx` line 78
- `LoanRow.tsx` line 44 (the item cell Link)

## Warnings

### WR-01: `Service.Create` and `Service.Return` are non-atomic — a crash between the two saves leaves inventory status permanently inconsistent

**File:** `backend/internal/domain/warehouse/loan/service.go:96-107` (Create) and `131-148` (Return)

**Issue:** In `Create`, `inventoryRepo.Save` (line 99) persists the inventory as `ON_LOAN` before `repo.Save` (line 104) persists the loan row. If the process terminates or `repo.Save` returns an error, the inventory is stuck `ON_LOAN` with no loan record. The test `TestService_Create_SaveErrors / loan save error` confirms this path. The same race exists in `Return` in reverse.

**Fix:** Wrap both saves in a database transaction. The `TxManager` is already wired in `router.go` (line 129). At minimum, document this gap as a tracked issue and add a repair-mode scan that cross-checks `ON_LOAN` inventory against active loans.

### WR-02: `Service.Return` checks `if inv != nil` after an API that never returns `(nil, nil)` — dead guard masks a data-loss scenario

**File:** `backend/internal/domain/warehouse/loan/service.go:135`

**Issue:**

```go
inv, err := s.inventoryRepo.FindByID(ctx, loan.InventoryID(), workspaceID)
if err != nil {
    return nil, err
}
if inv != nil {     // dead: FindByID never returns (nil, nil)
    ...
}
```

`FindByID` either returns a value or an error (`shared.ErrNotFound`). The `if inv != nil` guard is dead code. If any future implementation of `FindByID` were to return `(nil, nil)` (e.g., a mock or a new repository), the inventory status update would be silently skipped, persisting the loan as returned while inventory stays `ON_LOAN`.

Additionally, when `FindByID` returns `ErrNotFound` (inventory deleted), the loan cannot be returned at all — it becomes permanently stuck in active state. This is confirmed in `TestService_Return_InventoryDeleted`.

**Fix:** Remove the nil guard; treat deleted inventory as a recoverable case:

```go
inv, err := s.inventoryRepo.FindByID(ctx, loan.InventoryID(), workspaceID)
if err != nil {
    if !errors.Is(err, shared.ErrNotFound) {
        return nil, err
    }
    // Inventory deleted; still persist the return — loan is authoritative.
} else {
    if err := inv.UpdateStatus(inventory.StatusAvailable); err != nil {
        return nil, err
    }
    if err := s.inventoryRepo.Save(ctx, inv); err != nil {
        return nil, err
    }
}
```

### WR-03: `LoanRepository.Save` silently drops notes changes on existing loans

**File:** `backend/internal/infra/postgres/loan_repository.go:41-62`

**Issue:** The existing-record branch in `Save` only covers two update paths: `ReturnLoan` (when `returnedAt` flips) and `ExtendLoanDueDate` (when `dueDate` changes). Notes changes fall through to "No changes needed" and are silently discarded. The current `Service.Update` correctly bypasses `Save` and calls `repo.Update` directly, so this is not an active data-loss bug. However, the `Save` method's implied upsert contract makes this a trap for future callers.

**Fix:** Document explicitly at the top of `Save` that notes-only updates on existing loans must use `repo.Update`:

```go
// Save handles three cases for existing loans: Create (new), Return (returnedAt flip),
// and ExtendDueDate (dueDate change). Notes-only or combined edits MUST use repo.Update.
// Callers that need to persist notes changes on existing loans will get a silent no-op here.
```

### WR-04: `useReturnLoan` invalidates `itemKeys.detail(inventoryId)` with an inventory ID — the cache key is keyed by item definition ID

**File:** `frontend2/src/features/loans/hooks/useLoanMutations.ts:112-113`

**Issue:**

```typescript
qc.invalidateQueries({ queryKey: itemKeys.detail(inventoryId) });
```

`itemKeys.detail` is keyed by the item definition ID (from `/items/:id`). `inventoryId` is a loan's inventory row UUID. These two IDs are different in practice. Passing an inventory UUID to `itemKeys.detail` produces a cache key that will never match a cached item detail entry, so the item detail page does not refresh after a loan is returned.

**Fix:** Pass the item definition ID, which is available from the `Loan` object's `item.id` field (set by the backend decoration layer). Update mutation variables to include it:

```typescript
mutationFn: ({ id }) => loansApi.return(workspaceId!, id),
onSuccess: (_void, { id, itemId, borrowerId }) => {
  qc.invalidateQueries({ queryKey: itemKeys.detail(itemId) });
  ...
```

Or broaden to `qc.invalidateQueries({ queryKey: itemKeys.lists() })` as a simpler fix.

### WR-05: `useLoansForItem` parameter is named `inventoryId` but callers pass an item definition ID — misleading name will cause future misuse

**File:** `frontend2/src/features/loans/hooks/useLoansForItem.ts:23`

**Issue:** The hook signature is `useLoansForItem(inventoryId: string | undefined)` and its cache key is `loanKeys.forItem(inventoryId)`. However, both `ItemActiveLoanPanel` and `ItemLoanHistoryPanel` call it with `itemId` (from the URL param `:id`, which is an item definition ID). The underlying API call `loansApi.listForItem(wsId, inventoryId!)` hits `GET /items/{item_id}/loans`, which expects an item definition ID — so the data is correct end-to-end, but the parameter name says "inventory". Any future caller reading the name and passing an actual inventory row UUID will get incorrect results silently.

**Fix:** Rename throughout:

```typescript
export function useLoansForItem(itemId: string | undefined) {
  const query = useQuery({
    queryKey: itemId ? loanKeys.forItem(itemId) : ["loans", "forItem", "pending"],
    queryFn: () => loansApi.listForItem(workspaceId!, itemId!),
    enabled: !!workspaceId && !!itemId,
  });
```

Also rename `loanKeys.forItem`'s parameter from `inventoryId` to `itemId` in `loans.ts`.

### IN-02: `loansApi.return` typed as `post<void>` but the backend returns a full `LoanResponse`

**File:** `frontend2/src/lib/api/loans.ts:95`

**Issue:**

```typescript
return: (wsId: string, id: string) => post<void>(`${base(wsId)}/${id}/return`),
```

The backend `POST /loans/{id}/return` handler returns a decorated `LoanResponse`. Typing it as `void` causes the returned loan to be silently discarded. Current callers rely on query invalidation so this is not a functional bug, but the type is incorrect and will mislead future callers attempting to read the return value from `mutateAsync`.

**Fix:**

```typescript
return: (wsId: string, id: string) => post<Loan>(`${base(wsId)}/${id}/return`),
```

### IN-04: History pagination next-page heuristic navigates to an empty last page when the server returns exactly `pageSize` items

**File:** `frontend2/src/features/loans/LoansListPage.tsx:91-93`

**Issue:** The total-count hint adds 1 to keep the "next page" button enabled when the page is full:

```typescript
const historyTotalHint = (ui.page - 1) * HISTORY_PAGE_SIZE + currentLoans.length + 1;
```

When the current page happens to be the exact last page (server returns exactly 25 items), the user can navigate to page N+1 only to find an empty list. The component JSDoc acknowledges this as a known tradeoff ("a dedicated total could be wired later").

**Fix:** No immediate fix required; track as UX debt. The cleanest resolution is to expose a `total_count` field on the paginated backend response and use it directly.
