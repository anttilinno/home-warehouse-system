# Phase 62: Code Review Report

**Reviewed:** 2026-04-17
**Depth:** standard
**Files Reviewed:** 28
**Status:** issues_found

## Summary

Phase 62 introduces a full loan lifecycle: create, return, extend, and a new unified PATCH edit endpoint. The backend domain model is well-structured with proper workspace scoping, the SQL is generated by sqlc, and the frontend uses TanStack Query with consistent invalidation. Overall the code is solid. The critical findings are both in the backend: a missing workspace-scope guard in `ReturnLoan` SQL and a no-op `Delete` implementation that silently succeeds without persisting anything. Five warnings cover a logic error in the service layer's `Return` method, a non-atomic create path, a stale-check inconsistency in `Save`, an incorrect `inventoryId` passed to cache invalidation on return, and a `useLoansForItem` hook that passes `itemId` where the API and server expect `inventoryId`. Four info items cover dead queries, a `Delete` stub, and minor patterns.

## Critical Issues

### CR-01: `ReturnLoan` SQL missing workspace_id scope

**File:** `backend/internal/infra/queries/loans.sql.go:700-724`
**Issue:** The generated `ReturnLoan` query filters only on `id` with no `workspace_id` predicate:
```sql
UPDATE warehouse.loans
SET returned_at = now(), updated_at = now()
WHERE id = $1
```
The caller in `LoanRepository.Save` (line 43) reaches this path after it has already loaded the loan via `GetLoan` (which IS workspace-scoped), so an attacker cannot exploit this in practice today. However, the defence-in-depth principle the codebase explicitly calls out (comments in `loans.sql` and `loan_repository.go` repeatedly reference T-62-02) is violated at the SQL layer. If the call path is ever refactored the scope guard is gone. All other mutating queries (`UpdateLoan`, `ExtendLoanDueDate`) already include `workspace_id` in their WHERE clauses.

**Fix:** Update `loans.sql` and regenerate with sqlc:
```sql
-- name: ReturnLoan :one
UPDATE warehouse.loans
SET returned_at = now(), updated_at = now()
WHERE id = $1 AND workspace_id = $2
RETURNING *;
```
Callers must pass `workspace_id`; update `LoanRepository.Save` to supply it.

### CR-02: `LoanRepository.Delete` is a silent no-op

**File:** `backend/internal/infra/postgres/loan_repository.go:218-222`
**Issue:** The `Delete` method always returns `nil` without executing any SQL:
```go
func (r *LoanRepository) Delete(ctx context.Context, id uuid.UUID) error {
    // Note: This would typically be a hard delete or archive
    // For now, we'll just return nil as loans are typically not deleted
    return nil
}
```
`Delete` satisfies the `Repository` interface. Any future caller that depends on deletion being persisted will silently fail with no error, no indication, and the record still in the database. This is a correctness hazard: it cannot be detected from the call site.

**Fix:** Either remove `Delete` from the `Repository` interface until it is needed, or implement it defensively:
```go
func (r *LoanRepository) Delete(ctx context.Context, id uuid.UUID) error {
    return errors.New("loan deletion is not implemented")
}
```
A panicking or error-returning stub is always safer than a silent no-op.

## Warnings

### WR-01: `Service.Return` checks `inv != nil` after an API that never returns nil

**File:** `backend/internal/domain/warehouse/loan/service.go:131-143`
**Issue:** After calling `s.inventoryRepo.FindByID(ctx, loan.InventoryID(), workspaceID)`, the code then branches on `if inv != nil`. The `FindByID` contract on `inventory.Repository` returns `shared.ErrNotFound` (non-nil error) when the row does not exist — it never returns `(nil, nil)`. The nil guard is therefore dead code that could mask a real "inventory row missing" scenario: if `FindByID` ever does return `(nil, nil)` in a future implementation the inventory status update is silently skipped and the loan is persisted as returned while the inventory item remains `ON_LOAN`.

**Fix:** Remove the nil guard and treat the error as fatal:
```go
inv, err := s.inventoryRepo.FindByID(ctx, loan.InventoryID(), workspaceID)
if err != nil {
    return nil, err
}
if err := inv.UpdateStatus(inventory.StatusAvailable); err != nil {
    return nil, err
}
```

### WR-02: Loan create is non-atomic — inventory status and loan record saved in separate statements

**File:** `backend/internal/domain/warehouse/loan/service.go:96-107`
**Issue:** `Service.Create` updates inventory status (`inventoryRepo.Save`) and then saves the loan (`repo.Save`) in two independent database calls with no transaction wrapper. If the process crashes, or the second call fails after the first succeeds, the inventory item will be permanently stuck in `ON_LOAN` status with no matching loan row. The same risk exists in `Service.Return` (lines 136-148).

**Fix:** Wrap both operations in a database transaction. This requires either a `WithTx` pattern in the repository layer or an explicit `Transactor` interface. At minimum, document the known atomicity gap as a tracked issue so it is not lost.

### WR-03: `LoanRepository.Save` stale-check does not detect notes changes

**File:** `backend/internal/infra/postgres/loan_repository.go:41-62`
**Issue:** The `Save` method's existing-record branch only updates via `ReturnLoan` (when `returnedAt` flips from nil to set) or `ExtendLoanDueDate` (when `dueDate` changes). Changes to `notes` alone fall through to "No changes needed" and are silently dropped. This means that if `Service.Update` is ever invoked via the `Save` path (rather than the direct `repo.Update` path), note changes would be lost without error.

The current code in `Service.Update` bypasses `Save` and calls `repo.Update` directly, so this is not an active data-loss bug today. But the `Save` interface contract implies upsert semantics, and `notes`-only updates silently no-op.

**Fix:** Either document explicitly that `Save` does not persist note updates (making the `repo.Update` path the only correct path for edits), or add a notes-change branch to `Save`:
```go
// If notes changed, update via UpdateLoan with setNotes=true
```

### WR-04: `useReturnLoan` invalidates `itemKeys.detail(inventoryId)` but passes `loan.inventory_id`

**File:** `frontend2/src/features/loans/hooks/useLoanMutations.ts:110-114`
**Issue:** On return success the hook invalidates `itemKeys.detail(inventoryId)` where `inventoryId` is taken from the mutation variables (`loan.inventory_id`). `itemKeys.detail` is keyed by the item definition ID (from `/items/:id`), not the inventory ID. Passing an inventory UUID as the key will never match a cached item detail entry, so the item detail page will not reflect the returned loan (status change from ON_LOAN → AVAILABLE) until the user manually refreshes or the cache expires.

**Fix:** The mutation variables should carry the item definition ID (available from `loan.item.id`), or the invalidation should broaden to `itemKeys.lists()` rather than `detail(inventoryId)`:
```ts
// In useReturnLoan onSuccess:
qc.invalidateQueries({ queryKey: itemKeys.lists() });
// OR pass itemId (not inventoryId) in the mutation variables
```

### WR-05: `useLoansForItem` and `ItemActiveLoanPanel` pass `itemId` to an endpoint that expects `inventoryId`

**File:** `frontend2/src/features/loans/hooks/useLoansForItem.ts:24-30`
**Issue:** The hook is called `useLoansForItem` and its parameter is named `inventoryId`, but both `ItemActiveLoanPanel` (line 29) and `ItemLoanHistoryPanel` (line 22) call it with `itemId` — the item definition ID from the URL param `:id`. The underlying API call is `loansApi.listForItem(wsId, inventoryId!)` which hits `GET /items/{item_id}/loans`. The backend endpoint `ListItemLoansInput` expects an `item_id` path parameter and uses `ListLoansByItem` which joins through inventory. So the data is actually correct end-to-end.

However, the hook's parameter name (`inventoryId`) and the `loanKeys.forItem(inventoryId)` cache key are misleading: they say "inventory" but they mean "item definition". This creates a risk that future callers pass an actual inventory UUID and get incorrect results, and it makes the `forItem` vs `forBorrower` key split ambiguous.

**Fix:** Rename the parameter from `inventoryId` to `itemId` throughout `useLoansForItem` and update `loanKeys.forItem` accordingly:
```ts
export function useLoansForItem(itemId: string | undefined) {
  const query = useQuery({
    queryKey: itemId ? loanKeys.forItem(itemId) : ["loans", "forItem", "pending"],
    queryFn: () => loansApi.listForItem(workspaceId!, itemId!),
    enabled: !!workspaceId && !!itemId,
  });
  ...
}
```

### IN-01: Dead SQL queries — `GetLoanWithDetails` and `ListActiveLoansWithDetails` are generated but never called

**File:** `backend/internal/infra/queries/loans.sql.go:144-335`
**Issue:** Two queries — `GetLoanWithDetails` and `ListActiveLoansWithDetails` — are defined in `loans.sql`, generated by sqlc, and compiled into the binary, but are never referenced anywhere in the codebase. Phase 62 replaced this pattern with the decoration-lookup approach (batch separate queries). These dead queries add noise and maintenance surface.

**Fix:** Remove `GetLoanWithDetails` and `ListActiveLoansWithDetails` from `loans.sql` and regenerate. If they are intentionally kept for future use, add a comment explaining why.

### IN-02: `loansApi.return` typed as `post<void>` but `ReturnLoan` handler returns a `LoanResponse`

**File:** `frontend2/src/lib/api/loans.ts:95`
**Issue:** The return endpoint is typed as returning `void`:
```ts
return: (wsId: string, id: string) => post<void>(`${base(wsId)}/${id}/return`),
```
The backend handler for `POST /loans/{id}/return` returns a fully-decorated `LoanResponse`. The `void` typing causes the returned loan body to be silently discarded. While the current callers only use the mutation for side-effect invalidation (they re-fetch from cache), the type is incorrect and any future caller attempting to read the returned loan from `mutateAsync` will get `undefined`.

**Fix:**
```ts
return: (wsId: string, id: string) => post<Loan>(`${base(wsId)}/${id}/return`),
```

### IN-03: `LoanRepository.Delete` is in the `Repository` interface but is a dead stub

**File:** `backend/internal/domain/warehouse/loan/repository.go:23`
**Issue:** `Delete(ctx context.Context, id uuid.UUID) error` is part of the `Repository` interface. No caller invokes it. This is covered in CR-02 for safety, but from a code-quality standpoint the interface should not promise capabilities that do not exist.

**Fix:** Remove `Delete` from the `Repository` interface until a deletion use case is implemented.

### IN-04: Pagination `totalCount` in `LoansListPage` is a heuristic that can show a stale "next page" button

**File:** `frontend2/src/features/loans/LoansListPage.tsx:92-93`
**Issue:** The history tab pagination total is approximated as `(page-1)*pageSize + items.length + 1` when the page is full. This means the "next page" button is always shown on a full page even if the current page is the last one, causing the user to navigate to an empty page before pagination disappears.

**Fix:** This is a known design tradeoff documented in the component JSDoc ("good enough for the tab chip; a dedicated total could be wired later"). Acceptable for now, but track it as a UX debt item.
