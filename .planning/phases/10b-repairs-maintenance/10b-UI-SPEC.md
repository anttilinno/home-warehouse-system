> Visual + interaction contract for **Repairs + Maintenance** (RPR-01..04, MNT-01..03 +
> parity): the per-inventory-row **Repairs drawer** (repair log list + cost rollup +
> start/complete/edit/delete + nested photos + nested non-photo attachments), the **repair

> + complete-advances-next-due), the standalone **`/maintenance/due` page**, and a sidebar nav

> **Direction anchor.** Retro OS Pastel (sketches 006-008, MANIFEST + the
> `sketch-findings-home-warehouse-system` skill ‚Äî BINDING). This is a **PARITY phase with an
> existing design system: it invents NO new visual language and NO new atoms.** Every treatment
> below is a *composition* of shipped atoms via the `@/components/retro` barrel (`Window`,
> `BevelButton`, `RetroDialog`, `RetroConfirmDialog`, `RetroFormField`/`RetroInput`/

## Backend wire shapes (verified 2026-06-13 against backend source ‚Äî the design binds to these)

Recorded because the design depends on the exact fields + lifecycle the read models carry.

### Repair log ‚Äî `repairlog/entity.go` + `handler.go`
- **Status lifecycle (LOCKED):** `PENDING ‚Üí IN_PROGRESS ‚Üí COMPLETED` (`RepairStatus` const).
  - `StartRepair()` is valid ONLY from `PENDING` (else `ErrInvalidStatusTransition`).
  - `Complete(newCondition?)` is valid ONLY from `IN_PROGRESS`.
  - `UpdateDetails(...)` / warranty / reminder edits are **rejected on a COMPLETED repair**
    (`ErrRepairAlreadyCompleted`).
- **Create body** (`POST /repairs`): `inventory_id`, `description` (req, `minLength:1`),
  `repair_date?`, `cost?` (**cents, int**), `currency_code?`, `service_provider?`, `notes?`,
  `is_warranty_claim?` (bool), `reminder_date?`.
- **PATCH body** (`PATCH /repairs/{id}`): `description?`, `repair_date?`, `cost?`,
  `currency_code?`, `service_provider?`, `notes?` (NO status ‚Äî status moves via start/complete).
- **Complete body** (`POST /repairs/{id}/complete`): `new_condition?` (`NEW|EXCELLENT|GOOD|FAIR|
  POOR|DAMAGED|FOR_REPAIR` ‚Äî the inventory `Condition` union). Sets `completed_at` +
  optionally sets the inventory entry's condition.
- **Response** carries `id, workspace_id, inventory_id, status, description, repair_date?,
  cost?, currency_code?, service_provider?, completed_at?, new_condition?, notes?` (and
  `is_warranty_claim`, `reminder_date?`). Dates are `*string` (RFC3339 / YYYY-MM-DD).
- **Per-entry list** `GET /inventory/{inventory_id}/repairs` ‚Üí `{ items: RepairLog[], total }`
  (paginated envelope; `limit` caps 100).
- **Cost rollup** `GET /inventory/{inventory_id}/repair-cost` ‚Üí
  `{ items: [{ currency_code?, total_cost_cents, repair_count }] }`.

### Repair photos ‚Äî `repairphoto/handler.go` (MIRRORS itemphoto exactly)
- `GET /repairs/{repair_log_id}/photos/list`, `GET .../photos/{id}`, `PUT .../photos/{id}/caption`,
  `DELETE .../photos/{id}` (huma JSON) **+ a Chi multipart upload** `POST /repairs/{repair_log_id}/photos`
  with form field name **`"photo"`** (verified: `r.FormFile("photo")`), and blob serve routes
  `.../photos/{id}/file` + `.../photos/{id}/thumbnail`. **Identical shape to `photosApi`** ‚Äî
  the only difference is `/items/{itemId}` ‚Üí `/repairs/{repairLogId}`.

### Repair attachments (non-photo) ‚Äî `repairattachment/handler.go`
- `GET /repairs/{repairLogId}/attachments` ‚Üí list of `RepairAttachmentResponse`
  (`id, repair_log_id, file_id, attachment_type, title?, file_name?, file_mime_type?, file_size?`).
- `POST /repairs/{repairLogId}/attachments` body `{ file_id, attachment_type, title? }` ‚Äî links an
  **already-uploaded file** (it does NOT upload bytes). `attachment_type` ‚àà
  `PHOTO | MANUAL | RECEIPT | WARRANTY | OTHER` (invalid ‚Üí 400). Unknown file ‚Üí 404; cross-workspace
  file ‚Üí 400.
- `DELETE /repairs/{repairLogId}/attachments/{attachmentId}`.
- **The `file_id` is produced by the item-attachment upload domain** (`attachment/handler.go`:
  `POST /items/{item_id}/attachments/upload`, multipart ‚Üí a `File` row with an id). See OQ3.

### Maintenance ‚Äî `maintenance/handler.go` + `service.go`
- `GET /maintenance` (list), `GET /inventory/{inventory_id}/maintenance` (per-entry schedules).
- `GET /maintenance/due` ‚Üí `{ items: [{ ...schedule, item_name, is_overdue }] }`. **`is_overdue` is
  SERVER-computed** (`Schedule.IsOverdue(now)`) ‚Äî the MNT-02 due list + the MNT-03 dashboard feed source.
- `POST /maintenance` body `{ title (req), notes?, interval_days (‚â•1), next_due (date) }`;
  `PATCH /maintenance/{id}` (same fields optional); `DELETE /maintenance/{id}`.
- `POST /maintenance/{id}/complete` body `{ notes? }` ‚Äî **server-side**: writes a repair_logs row,
  sets `last_completed_at`, and **advances `next_due` to `max(today, next_due + interval)`** (verified
  `service.go` comment + `Schedule.Complete`). The UI just calls it + invalidates (OQ4 resolved).
- **Schedule response:** `id, title, notes?, interval_days, next_due (YYYY-MM-DD), last_completed_at?`.

### Repair status ‚Üí StatusPill variant mapping (Resolved ‚Äî autonomous; mirrors `loanStatus.ts` discipline)

A pure `repairStatus(status)` helper (the **`loanStatus.ts` pattern** ‚Äî no React, reads only the
server `status` enum, NEVER derives state from dates). `StatusPill` is the thin `RetroBadge` preset
(pastel fill + **ink** text). **No new variants, no new pairings.**

| Repair status (server) | Variant | Display | Rationale |
|------------------------|---------|---------|-----------|
| `PENDING` | **info** (blue) | `Pending` | Scheduled, not started ‚Äî neutral-informational. |
| `IN_PROGRESS` | **warn** (butter) | `In progress` | Active work ‚Äî warn fill draws the eye (matches the 7b warn-condition idiom). |
| `COMPLETED` | **ok** (mint) | `Completed` | Done, positive. |

> **CONTEXT used "draft/in-progress/completed".** The backend enum is **`PENDING`/`IN_PROGRESS`/
> `COMPLETED`** (verified `entity.go`). The pill DISPLAY uses Title-Case `Pending`/`In progress`/
> `Completed`. **Resolved (autonomous): "draft" ‚Üí "Pending"** (the real first state).

### Maintenance overdue cue (Resolved ‚Äî autonomous; the 3-non-color-cue rule, NOT color-only)

Overdue is **SERVER-authoritative** (`due-list` row `is_overdue` flag ‚Äî NEVER client `next_due < now`
math; this is the explicit "loans lesson" carry-forward in CONTEXT ¬ß3). An overdue due-list row carries
**THREE non-redundant cues** so it is never color-only (sketch-findings + AA discipline):

1. **Row tint** ‚Äî overdue rows get a `bg-danger-bg` (#fbe3e8) row fill, distinct from the white /
   `#fcf8f0`-stripe normal rows.
2. **Status pill / badge** ‚Äî the row shows a **danger `Overdue`** `StatusPill` (the word + the pink fill).
3. **Next-due chip glyph** ‚Äî the Next-due cell renders a **danger** chip prefixed `‚öÝ` (e.g. `‚öÝ {date}`)
   instead of the neutral mono date.

The pill word + the `‚öÝ` glyph carry the meaning for color-blind users; the row tint is reinforcing,
not sole. Non-overdue due rows render the next-due date as a neutral mono `tabular-nums` value.

**No NEW contrast pairing this phase.** Every text/bg pair already passes `tokens.test.ts`
(ink-on-panel/panel-2/blue/ok/warn/info/danger-bg, danger-on-danger-bg, deep companions on white).
**Flag for AA test: none.**

### Cents ‚Üí currency display rule (Resolved ‚Äî autonomous, must-specify)

Costs are stored + sent as **integer cents** (`cost`, `total_cost_cents`). The UI:
- **Displays** via a shared `formatCents(cents, currencyCode?)` helper ‚Üí `Intl.NumberFormat(undefined,
  { style: "currency", currency: currencyCode ?? "USD" })` applied to `cents / 100`, rendered in
  IBM Plex Mono `tabular-nums` (e.g. `$42.50`). Absent cost ‚Üí muted `‚Äî`.
- **Inputs** via a currency `RetroInput` (`type="text"` / `inputMode="decimal"`, mono) whose zod
  transform parses the major-unit string ‚Üí `Math.round(value * 100)` **cents int** before the mutation.
  **NEVER sends a float `cost`.** Empty ‚Üí omit `cost`.
- `currencyCode` defaults to the workspace/inventory currency if the API supplies one on the rollup;
  the cost field offers no currency picker this phase (single-currency assumption ‚Äî Resolved R-cents).

### Lifecycle action behaviour (server-flag discipline)

- **`START`** ‚Üí `POST /repairs/{id}/start`. Optimistic: pill flips `Pending ‚Üí In progress` (the
  action set swaps to `COMPLETE/EDIT/DELETE`). On error revert + `retroToast.error`
  `Couldn't start this repair. {reason}`. Success toast `DONE ¬∑ Repair started.`
- **`COMPLETE`** ‚Üí the complete dialog (¬ß2) ‚Üí `POST /repairs/{id}/complete {new_condition?}`.
  Optimistic: pill flips `In progress ‚Üí Completed`, `completed_at` set, actions collapse to `DELETE`.
  **Because complete can set the inventory condition, invalidate BOTH `["repairs", wsId, ...]` AND
  `["inventory", wsId]`** (CONTEXT ¬ß7). Success toast `DONE ¬∑ Repair completed.`
- **`DELETE`** ‚Üí pink `RetroConfirmDialog` ‚Üí `DELETE /repairs/{id}`. Optimistic row removal + revert on
  error. Success toast `DONE ¬∑ Repair deleted.`
- All mutations live in a `useRepairMutations(wsId, invId)` hook (the `usePhotoMutations`/
  `useInventoryMutations` pattern): prefix-invalidate `["repairs", wsId]` (and `["inventory", wsId]`
  on complete), `retroToast.error` on failure.

## 2 ¬∑ Repair Create / Edit / Complete dialogs (RHF + zod, nested in the drawer)

> **Resolved (autonomous, OQ1): the repair form is a `RetroDialog` opened OVER the repairs drawer
> (modal-stack nested), NOT a route.** Rationale: a repair is scoped to one inventory entry already
> open in the drawer; a route round-trip would tear down the drawer context. This mirrors the photo
> upload + caption dialogs that already nest inside galleries. (A repair is created *for the inventory
> entry whose drawer is open* ‚Äî `inventory_id` is implicit, never a form field.)

### Create / Edit dialog (`RetroDialog`, **blue** titlebar)

- **Title**: `ADD REPAIR` (create) / `EDIT REPAIR` (edit). RHF + zod, `gap-sp-4` field stack
  (mirrors `InventoryFormPage` grouping).
- **Fields** (`RetroFormField`-wrapped):
  1. **Description** (`RetroTextarea`, **required `*`**, `min-h-[88px]`) ‚Äî `minLength:1`; empty ‚Üí
     field error `Description is required.`
  2. **Repair date** (`RetroInput type="date"`, mono, optional) ‚Äî serialized `YYYY-MM-DD`; absent ‚Üí omitted.
  3. **Cost** (currency `RetroInput`, mono, optional) ‚Äî hint `Amount in your workspace currency.`;
     zod transforms major-unit ‚Üí cents int (see Cents rule). Empty ‚Üí omit.
  4. **Service provider** (`RetroInput type="text"`, optional).
  5. **Warranty claim** (`RetroCheckbox`, optional) ‚Äî label `Covered under warranty`.
  6. **Reminder date** (`RetroInput type="date"`, mono, optional) ‚Äî hint `Optional ‚Äî a follow-up
     maintenance reminder.`
- **Footer** (pinned, `border-t-2 border-border-ink pt-sp-3`, right): `CANCEL` (neutral) +
  `SAVE REPAIR` (create) / `SAVE CHANGES` (edit) (`primary`, disabled while submitting).
- **Validation**: sketch-007 in-window ‚Äî per-field `‚úï`-prefixed `text-danger` line + `aria-invalid`;
  form-level submit failure ‚Üí a `bg-danger-bg border-2 border-border-ink` banner
  `‚úï Couldn't save this repair. {reason}` + persistent `retroToast.error`.
