Update the existing `SocialLogin` component in `frontend/features/auth/components/social-login.tsx` to be offline-aware:

1. **Add imports:**
   - `useState, useEffect` from "react"
   - `useNetworkStatus` from `@/lib/hooks/use-network-status`
   - `WifiOff` from `lucide-react`
   - Keep existing `useTranslations` and `Button` imports

2. **Add offline detection with SSR hydration guard:**
   - Call `useNetworkStatus()` to get `{ isOffline }`
   - Add a `mounted` state (`useState(false)`) initialized to false, set to true in `useEffect(() => { setMounted(true) }, [])`
   - Compute `showOffline = mounted && isOffline` (prevents SSR hydration mismatch -- server always renders buttons as enabled, client only disables after mount)
   - This follows the same pattern as `OfflineIndicator` component in `frontend/components/pwa/offline-indicator.tsx`

3. **Update button rendering:**
   - Add `disabled={showOffline}` to both Google and GitHub buttons
   - Keep existing SVG icons and `variant="outline"` styling
   - The buttons should still use `useTranslations("auth")` for labels: `t("oauth.continueWithGoogle")` and `t("oauth.continueWithGithub")` (from the translation keys added in 42-01)

4. **Add offline message below buttons:**
   - When `showOffline` is true, render a paragraph below the button grid:
     ```
     <p className="text-center text-xs text-muted-foreground flex items-center justify-center gap-1">
       <WifiOff className="h-3 w-3" />
       {t("oauth.offlineRequired")}
     </p>
     ```
   - When online, do not render this paragraph

5. **Do NOT use Tooltip or TooltipProvider:**
   - Research suggested tooltips, but the auth layout does NOT include a `TooltipProvider` and adding one is outside scope. The inline text message below buttons is sufficient and more accessible on mobile (tooltips are hover-only, bad for touch).

6. **Wrap in a container div:**
   - Wrap the entire return in `<div className="space-y-2">` to provide spacing between the button grid and the offline message

7. **Button click handlers NOT added in this plan:**

The buttons remain non-functional (no onClick) until Phase 41 wires them up. This is intentional -- Phase 42 depends on Phase 41 for the full OAuth flow, but the offline behavior and i18n can be done independently.
   - Actually, wait -- Phase 42 depends on Phase 41 per roadmap. But we are building Phase 42 plans now. The translation keys and offline behavior are additive -- they modify the existing placeholder component. When Phase 41 adds click handlers, it will work with the disabled state already in place. The component should be structured so that Phase 41 can easily add `onClick` handlers to the buttons.
