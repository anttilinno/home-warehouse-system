<objective>
Make the Members backend parity-true so the frontend Members page (Plan 12-06) can render
member identity and add members by email — WITHOUT building an email-invite/email-send system.

Two surgical Go changes over the existing member domain:
1. **Enrich identity on read.** `MemberResponse` currently exposes only `user_id`. The SQL
   `ListMembersByWorkspace` (workspace_members.sql.go:167) ALREADY joins `auth.users` and selects
   `u.email, u.full_name`, but the repository drops those columns in `Reconstruct`. Thread them
   through the `Member` entity (as optional read-only fields) → `MemberResponse.email` +
   `MemberResponse.full_name`.
2. **Add-by-email.** `POST /members` currently takes a `user_id` UUID. Extend its body to accept
   an `email`, resolve it to an EXISTING registered user's id (404
   "no registered user with that email" if absent), then add that user. No pending-invite row,
   no email send — there is no email infra and it is out of scope (per ORCHESTRATOR DECISION / OQ6).

Per VALIDATION binding override 7: server already guards own-role-change (400) and last-owner
removal (400) — do NOT touch those; only verify they remain green.

Purpose: parity — members must show as people (name/email), and an admin must add a teammate by
the email they registered with, not by hunting a UUID.
Output: enriched MemberResponse, email-accepting POST /members, Go unit tests, all backend gates green.

This plan is Go-only and DISJOINT from the frontend foundation (Plan 12-02) — it runs in parallel
in Wave 1. It uses `go` toolchain gates (NOT bun).
</objective>
