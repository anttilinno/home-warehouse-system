<objective>
The single-writer foundation for the Settings hub. This is the ONLY plan that edits
`SettingsLayout.tsx` and `routes/index.tsx`. It:
1. Rewrites `SettingsLayout` from a 2-tab RetroTabs sub-nav into a thin centered `<Outlet/>`
   wrapper (`mx-auto max-w-[720px]`) — the grouped landing becomes the index, not a redirect.
2. Builds `SettingsLandingPage` — the iOS/System-7 grouped-row landing (SETT-01): three `Window`
   groups (ACCOUNT / PREFERENCES / WORKSPACE) of `<Link>` rows with trailing `›` glyph, mirroring
   the AccountsPage `<ul>` row idiom. NO leading icons (UI-SPEC: icon style unresolved). Paperless
   row = disabled "Set up in DMS" / "COMING SOON" pointer (does NOT build DMS).
3. Registers ALL new subpage routes in `routes/index.tsx` so downstream plans only ADD component
   files (the route is pre-wired). Lazy-imports the new pages so the route table is the single
   source of truth and downstream plans never touch this file.
4. Creates `settingsApi` (`src/lib/api/settings.ts`) — the typed module every subpage imports.
5. Extends `lib/types.ts`: a `Preferences` type, a `Member` type, and the `User` interface gains
   the seven preference fields GetMe returns.
6. Extends the shared MSW base mocks (`handlers.ts`): the `ME` fixture gains preference fields,
   plus handlers for `PATCH /users/me`, `PATCH /users/me/preferences`, `POST|DELETE /users/me/avatar`,
   members CRUD, and `GET /export/workspace`.

Purpose: a stable contract surface so the four Wave-2 plans and the Wave-3 Members plan are fully
parallel (disjoint files, all only READ this plan's outputs).
Output: landing + layout + route table + api + types + MSW base, with a landing unit test.

Runs in Wave 1 in parallel with the Go backend members plan (12-01) — disjoint (TS vs Go).
</objective>
