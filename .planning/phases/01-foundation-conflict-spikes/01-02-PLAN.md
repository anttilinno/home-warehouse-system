<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub PR submission → workflow runner | Untrusted PR code triggers Actions runner with read access to repo |
| Workflow runner → npm registry | `bun install --frozen-lockfile` resolves deps from public registry |
| Workflow runner → repo merge gate | Job exit codes determine merge eligibility |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | PR introduces `import 'idb'` to bypass online-only constraint | mitigate | `forbidden-imports` job runs `node scripts/check-forbidden-imports.mjs`; exit 1 fails the merge gate (the threat the entire plan exists to address) |
| T-02-02 | Tampering | PR modifies `scripts/check-forbidden-imports.mjs` to weaken patterns | mitigate | `paths:` trigger includes `scripts/check-forbidden-imports.mjs` so script edits re-run the self-tests; `scripts/__tests__/check-forbidden-imports.test.mjs` covers fixture cases that would fail on a weakened regex |
| T-02-03 | Tampering | Malicious PR pins `setup-bun` to a typo-squatted action | accept | `oven-sh/setup-bun@v2` is the official action; pinning to `@v2` exposes us to v2 patch updates but the alternative (full SHA pin) is high-friction. Solo-developer threat model: the PR author IS the maintainer |
| T-02-04 | Denial of Service | Workflow runs forever on hung `bun install` | accept | GitHub Actions has a default 6h job timeout; sufficient for solo-developer cadence |
| T-02-05 | Repudiation | Failed CI run not visible to maintainer | mitigate | GitHub UI surfaces failed checks on PR view; no extra mitigation needed |
| T-02-06 | Information Disclosure | Workflow logs leak secrets via `bun install` env | mitigate | Job uses no secrets (`secrets:` block absent); `bun install --frozen-lockfile` uses public registry only |
</threat_model>
